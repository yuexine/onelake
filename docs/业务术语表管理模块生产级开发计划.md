# 业务术语表管理模块生产级开发计划

## 1. 目标与边界

业务术语表管理模块用于统一维护业务概念、指标口径、同义词、负责人、审定状态，以及业务术语到物理资产字段的映射关系。生产化后的核心价值是让“业务说法”能够稳定落到“表、字段、指标、API、质量、安全策略”上，避免同名异义、口径漂移和资产不可理解。

本计划基于当前代码现状制定：

- 前端已有 `/catalog/glossary` 页面，但仍使用 `glossaryTerms` mock 数据。
- 后端 `module-modeling` 已有 `modeling.data_standard` 表和 `DataStandard` 实体，但没有 Repository、Service、Controller。
- `module-catalog` 已有资产、字段 JSON、血缘、质量分、安全标签、DaaS 订阅画像等真实闭环。
- `module-common` 已有 canonical `DomainEvents`、Outbox、Redis Stream 消费和审计基础设施。
- 质量、安全、数据服务、编排模块已有可复用 API 和事件消费者模式。

## 2. 推荐架构选择

### 方案 A：术语表归属 Catalog

优点：距离资产和字段最近，做搜索和资产详情展示最快。

缺点：术语、指标口径、主题域、建模标准会和 Catalog 资产索引职责混在一起，长期会让 Catalog 同时承担“资产缓存”和“业务标准主数据”。

### 方案 B：术语表归属 Modeling，Catalog 做投影和检索

优点：贴合现有 `SubjectDomain`、`Metric`、`DataStandard`；术语、指标、口径、建模字段命名和 DWD/ADS 建模天然相关。Catalog 通过绑定表和事件获得术语投影，保持目录职责清晰。

缺点：Catalog 搜索需要跨模块查询或维护一份冗余投影。

### 方案 C：抽到 Common

优点：所有模块都能依赖。

缺点：业务术语不是基础设施，放入 Common 会扩大公共模块职责，并让业务演进受到底层模块耦合影响。

### 推荐

采用方案 B：`module-modeling` 作为业务术语主数据所有者，`module-catalog` 消费术语绑定变化并将术语投影到资产/字段详情，其他模块通过 API 或事件联动。

## 3. 生产级能力清单

### P0 必须交付

1. 术语 CRUD：创建、查询、编辑、删除或归档。
2. 审定状态：`DRAFT`、`REVIEWING`、`APPROVED`、`DEPRECATED`、`ARCHIVED`。
3. 术语字段：编码、名称、业务域、定义、口径、同义词、负责人、数据 Owner、状态、版本。
4. 术语到字段绑定：支持绑定资产 FQN、资产 ID、字段名、关系类型、来源、置信度和状态。
5. Catalog 联动：资产搜索支持术语/同义词，资产详情 Schema 展示字段绑定的业务术语。
6. 前端真实化：`Glossary.tsx` 去 mock，接真实 API，补 loading、empty、error、权限和保存失败状态。
7. 多租户与权限：所有查询按 `TenantContext` 隔离，写操作按角色控制。
8. 审计和事件：术语变更、审定、绑定变更写审计并发布 Outbox 事件。

### P1 应交付

1. 术语版本历史和 diff。
2. 审批流接入：高影响术语变更进入 `security.approval_request`。
3. 影响分析：术语口径变更时展示绑定字段、下游资产、质量规则、API。
4. 建模联动：TableWizard/DWD 模型字段映射支持选择术语，自动带出定义、字段注释、命名建议。
5. 质量联动：术语口径和约束生成质量规则建议。
6. 安全联动：敏感术语绑定字段后触发 PII 扫描或脱敏策略建议。
7. DaaS 联动：API 返回字段自动继承术语定义、口径和密级提示。

### P2 增强项

1. 自动推荐绑定：根据字段名、描述、标签、历史绑定和同义词推荐术语。
2. OpenMetadata glossary writeback。
3. 术语批量导入导出。
4. 语义检索或全文检索。
5. 术语健康度：无绑定、重复定义、口径冲突、长期未审定统计。

## 4. 数据模型设计

### 4.1 主表：`modeling.business_term`

建议新增独立表，不直接扩展 `modeling.data_standard`。原因是当前 `data_standard` 字段过少且命名偏“数据标准”，新表可以承载生产级版本、审定、负责人和绑定关系。后续可将 `data_standard` 作为兼容遗留表，或迁移为视图。

核心字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | UUID | 主键 |
| `tenant_id` | UUID | 租户隔离 |
| `code` | VARCHAR(64) | 术语编码，如 `GMV`、`AOV` |
| `name` | VARCHAR(128) | 中文或业务名称，如 `成交总额` |
| `domain_id` | UUID | 关联 `modeling.subject_domain` |
| `definition` | TEXT | 业务定义 |
| `caliber_sql` | TEXT | 指标口径或表达式 |
| `synonyms` | JSONB | 同义词数组 |
| `owner_id` | UUID | 业务负责人 |
| `owner_name` | VARCHAR(128) | 冗余展示名 |
| `steward_id` | UUID | 数据治理负责人 |
| `status` | VARCHAR(16) | 生命周期状态 |
| `version` | INT | 当前版本 |
| `sensitivity_level` | VARCHAR(8) | 建议密级，可为空 |
| `tags` | JSONB | 业务标签 |
| `created_by` | UUID | 创建人 |
| `updated_by` | UUID | 更新人 |
| `approved_by` | UUID | 审定人 |
| `created_at` | TIMESTAMPTZ | 创建时间 |
| `updated_at` | TIMESTAMPTZ | 更新时间 |
| `approved_at` | TIMESTAMPTZ | 审定时间 |

约束：

- `UNIQUE (tenant_id, code)`，同租户内术语编码唯一。
- `status` 限定枚举。
- `definition` 和 `caliber_sql` 在 `APPROVED` 时必须满足非空规则，具体由 Service 校验。

### 4.2 绑定表：`modeling.business_term_binding`

核心字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | UUID | 主键 |
| `tenant_id` | UUID | 租户隔离 |
| `term_id` | UUID | 术语 ID |
| `asset_id` | UUID | Catalog asset ID，可为空 |
| `asset_fqn` | VARCHAR(512) | 资产 FQN |
| `column_name` | VARCHAR(128) | 字段名，可为空，空表示表级绑定 |
| `relation_type` | VARCHAR(16) | `DEFINES`、`USES`、`DERIVES` |
| `source` | VARCHAR(16) | `MANUAL`、`CATALOG`、`MODELING`、`IMPORT`、`SUGGESTED` |
| `confidence` | NUMERIC(5,2) | 推荐置信度 |
| `status` | VARCHAR(16) | `ACTIVE`、`PENDING`、`REJECTED`、`STALE` |
| `created_by` | UUID | 创建人 |
| `created_at` | TIMESTAMPTZ | 创建时间 |
| `updated_at` | TIMESTAMPTZ | 更新时间 |

约束：

- `UNIQUE (tenant_id, term_id, asset_fqn, column_name)`。
- 字段级绑定要求 `column_name` 非空。
- 当资产字段被删除或更名时，绑定不物理删除，标记为 `STALE`。

### 4.3 版本表：`modeling.business_term_version`

用于审计和回溯。每次 `APPROVED` 或口径变更生成快照，保存 term 主表的 JSON snapshot、变更原因、变更人。

### 4.4 Catalog 投影

短期不新增 `catalog.asset` 列，避免把资产 JSON 越扩越大。通过查询 `business_term_binding` 返回资产字段绑定。若后续为了搜索性能需要冗余，可在 `catalog.asset.columns` 的字段 JSON 中增加：

```json
{
  "name": "gmv",
  "type": "DECIMAL(18,2)",
  "terms": [
    { "id": "...", "code": "GMV", "name": "成交总额", "status": "APPROVED" }
  ]
}
```

## 5. 后端接口设计

所有接口使用现有 `ApiResponse`，路径建议放在 `/api/v1/modeling/glossary`。

### 5.1 术语接口

| 方法 | 路径 | 用途 | 权限 |
| --- | --- | --- | --- |
| `GET` | `/terms` | 分页/筛选术语，支持 keyword、domainId、status、ownerId | 登录 |
| `POST` | `/terms` | 创建术语草稿 | `DE`、`ADMIN` |
| `GET` | `/terms/{id}` | 获取术语详情，含绑定摘要 | 登录 |
| `PUT` | `/terms/{id}` | 编辑草稿或退回状态术语 | `DE`、Owner、`ADMIN` |
| `POST` | `/terms/{id}/submit` | 提交审定 | Owner、`DE` |
| `POST` | `/terms/{id}/approve` | 审定通过 | `ADMIN` 或数据治理角色 |
| `POST` | `/terms/{id}/reject` | 审定退回 | `ADMIN` 或数据治理角色 |
| `POST` | `/terms/{id}/deprecate` | 标记废弃 | Owner、`ADMIN` |
| `GET` | `/terms/{id}/versions` | 查看历史版本 | 登录 |

### 5.2 绑定接口

| 方法 | 路径 | 用途 | 权限 |
| --- | --- | --- | --- |
| `GET` | `/terms/{id}/bindings` | 查询术语关联字段 | 登录 |
| `POST` | `/terms/{id}/bindings` | 新增字段绑定 | `DE`、Owner、`ADMIN` |
| `DELETE` | `/bindings/{bindingId}` | 移除绑定，软删除或标记 `STALE` | `DE`、Owner、`ADMIN` |
| `POST` | `/terms/{id}/binding-suggestions` | 根据 Catalog 字段生成建议绑定 | `DE` |
| `POST` | `/bindings/{bindingId}/confirm` | 确认推荐绑定 | `DE`、Owner |
| `POST` | `/bindings/{bindingId}/reject` | 拒绝推荐绑定 | `DE`、Owner |

### 5.3 Catalog 查询增强

建议新增或扩展：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/catalog/assets?term=GMV` | 按术语编码、名称、同义词、绑定字段检索资产 |
| `GET` | `/api/v1/catalog/assets/{id}/glossary` | 返回资产及字段绑定的术语 |
| `GET` | `/api/v1/catalog/glossary/terms/search` | 给全局搜索或字段绑定弹窗做轻量联想 |

如果保持职责纯粹，`CatalogController` 不直接维护术语，只读 `modeling` 查询结果或调用 `GlossaryQueryService`。

## 6. 事件设计

在 `DomainEvents` 中新增：

| 事件 | 生产者 | 消费者 | 用途 |
| --- | --- | --- | --- |
| `modeling.term.created` | GlossaryService | Notification、Catalog | 术语新建通知和搜索索引刷新 |
| `modeling.term.updated` | GlossaryService | Catalog、Quality、DaaS | 术语定义/口径变化 |
| `modeling.term.approved` | GlossaryService | Catalog、Quality、Security、DaaS | 审定后可被生产链路使用 |
| `modeling.term.deprecated` | GlossaryService | Catalog、DaaS、Orchestration | 标记下游影响 |
| `modeling.term.binding_changed` | GlossaryBindingService | Catalog、Quality、Security、DaaS | 字段绑定变化 |

`modeling.term.binding_changed` payload 建议：

```json
{
  "tenantId": "...",
  "termId": "...",
  "termCode": "GMV",
  "termName": "成交总额",
  "status": "APPROVED",
  "assetId": "...",
  "assetFqn": "ads.ads_sales_df",
  "columnName": "gmv",
  "relationType": "DEFINES",
  "bindingStatus": "ACTIVE",
  "changedBy": "...",
  "changedAt": "2026-06-23T..."
}
```

消费策略：

- Catalog：刷新资产详情中的字段术语投影和术语搜索索引。
- Quality：生成或更新质量规则建议，不自动启用生产 BLOCK 规则。
- Security：如果术语有 `sensitivityLevel` 或 PII 语义，生成 PII 扫描/脱敏建议。
- DaaS：标记受影响 API 文档待刷新，API 发布时继承术语定义。
- Orchestration/Modeling：口径变化时提示相关模型重新校验或重编译。

## 7. 跨模块联动设计

### 7.1 数据目录联动

目标：术语成为资产可发现、可理解的入口。

开发内容：

1. `CatalogSearch` 搜索关键字命中术语编码、名称、同义词时，返回绑定资产。
2. `AssetDetail` 的 Schema 表格新增“业务术语”列，展示术语 code/name。
3. 术语详情中的关联字段点击跳转到资产详情，并定位 Schema tab 和字段。
4. 资产字段删除、字段重命名或资产下线时，绑定标记为 `STALE`。
5. `CatalogAssetDetailService` 聚合术语绑定摘要，供资产详情使用。

验收：

- 新建并审定 `GMV`。
- 绑定 `ads.ads_sales_df.gmv`。
- 搜索 `GMV` 能找到 `ads.ads_sales_df`。
- 资产详情 Schema 能看到 `gmv` 字段绑定 `GMV`。

### 7.2 建模联动

目标：术语和指标口径成为建模字段、指标和模型产物的标准来源。

开发内容：

1. `Metric` 增加 `termId` 或 `termCode`，指标可以关联业务术语。
2. DWD/ADS 建模时，字段定义可选择业务术语，带出名称、描述、口径和建议字段名。
3. DWD 模型编译时，将字段术语关系写入 `business_term_binding`，source 为 `MODELING`。
4. 模型运行成功 `modeling.model.loaded` 后，Catalog 资产字段同步保留术语绑定。
5. 术语口径变更后，相关模型标记为需要重新校验。

验收：

- 在 DWD 建模向导选择术语生成字段。
- 编译模型后能看到字段绑定关系。
- 术语口径修改后，相关模型详情出现“需重新校验”提示或待办。

### 7.3 质量联动

目标：术语定义和口径驱动质量规则建议，而不是手工重复配置。

开发内容：

1. 术语可声明质量约束，如非空、范围、枚举、SQL 口径校验。
2. 字段绑定术语后，Quality 模块生成规则建议，状态为 `SUGGESTED`。
3. 质量规则创建弹窗支持按术语筛选字段。
4. 质量结果和门禁失败页展示术语上下文，便于业务解释。
5. 术语废弃时，相关质量规则标记为需复核。

验收：

- `GMV` 绑定字段后，质量规则页能按术语选择 `gmv` 字段。
- 创建或确认规则后，稽核结果能回写 Catalog 质量分。

### 7.4 安全联动

目标：术语的敏感属性和字段绑定触发安全治理。

开发内容：

1. 术语支持 `sensitivityLevel` 和 `piiType` 建议。
2. 绑定字段后，如果术语是敏感类型，触发 `security.pii.detected` 或生成待确认扫描记录。
3. 资产详情显示术语、PII 类型和建议密级的一致视图。
4. DaaS 发布 API 时，如果响应字段绑定 L3/L4 术语，则要求脱敏或审批。
5. 术语状态非 `APPROVED` 时，不允许作为高敏字段的正式治理依据，只能作为建议。

验收：

- 创建敏感术语“手机号”并绑定字段。
- Security PII 页面出现待确认记录。
- Catalog 字段显示术语和建议密级。

### 7.5 DaaS 联动

目标：API 对外口径和字段解释继承术语表，保证使用者理解一致。

开发内容：

1. API Wizard 从资产字段读取术语，自动生成响应字段说明。
2. API 详情页展示字段术语、定义、口径、密级和脱敏状态。
3. API 发布前校验：响应字段若绑定未审定术语，给出 warning；若绑定废弃术语，阻断或要求确认。
4. 术语口径变化后，受影响 API 标记“文档待刷新”。
5. DaaS runtime 调试结果可返回 `securityNotices` 和 `glossaryNotices`。

验收：

- 从 `ads.ads_sales_df` 发布 API 草稿时，`gmv` 返回字段自动携带 `GMV` 定义。
- 术语废弃后，API 详情能看到影响提示。

### 7.6 编排与 SQL 工作台联动

目标：术语变化能影响任务、SQL 查询和 DAG 运行的可观测性。

开发内容：

1. SQL 工作台结果列根据 Catalog 字段绑定显示术语提示。
2. 保存查询模板时，可记录使用的术语和资产。
3. 术语口径变更后，通过血缘找出下游 DAG/API，形成影响列表。
4. DWD DAG 触发前如相关术语处于 `REVIEWING` 或 `DEPRECATED`，给出可解释 warning。

验收：

- SQL 查询 `ads.ads_sales_df.gmv` 返回结果列能识别 `GMV`。
- 修改 `GMV` 口径后，术语详情展示下游 DaaS API 和 DAG。

### 7.7 数据集成联动

目标：源字段进入湖仓时，能形成术语推荐和绑定候选。

开发内容：

1. `integration.table.loaded` 已携带 `fieldMapping`，Catalog 建档后触发术语推荐扫描。
2. 推荐逻辑基于字段名、字段描述、资产标签、同义词匹配。
3. 推荐绑定默认 `PENDING`，需要数据 Owner 确认。
4. 采集任务创建向导可以把源字段映射到目标术语，进入 ODS/DWD 后自动建绑定。

验收：

- 新采集表字段 `gmv` 自动出现 `GMV` 推荐绑定。
- 确认后资产详情和术语详情都可见该绑定。

## 8. 前端页面计划

### 8.1 `Glossary.tsx` 生产化

改造点：

1. 接 `GlossaryAPI.listTerms`，按业务域构建树。
2. 详情区域读取真实 term detail。
3. 新建术语弹窗：编码、名称、业务域、定义、口径、同义词、负责人。
4. 编辑、提交审定、审定、废弃、归档动作。
5. 关联字段区支持新增/移除绑定。
6. 支持 loading、empty、error、retry。
7. 状态 badge 使用统一组件，避免手写状态色。

### 8.2 `CatalogSearch.tsx`

## 10. 验收用例

### P0 验收链路

1. 登录租户 A。
2. 创建术语 `GMV`，填写定义、口径、同义词。
3. 提交审定并审批为 `APPROVED`。
4. 绑定 `ads.ads_sales_df.gmv`。
5. 打开业务术语表，能看到 `GMV` 详情和关联字段。
6. 打开目录搜索，搜索 `GMV` 能找到 `ads.ads_sales_df`。
7. 打开资产详情 Schema，`gmv` 字段显示 `GMV`。
8. 打开 API 构建向导，选择该资产后返回字段继承 `GMV` 定义。
9. 租户 B 搜索 `GMV` 不应看到租户 A 的术语或资产绑定。

### 事件验收

1. 创建术语产生 `modeling.term.created`。
2. 审定术语产生 `modeling.term.approved`。
3. 新增绑定产生 `modeling.term.binding_changed`。
4. Catalog 消费后资产详情可读绑定。
5. 重复消费同一事件不产生重复绑定。

### 非功能验收

1. 所有写接口有权限控制。
2. 所有查询按 `TenantContext` 隔离。
3. 所有变更写 audit。
4. Outbox 失败可重试，重复事件幂等。
5. 前端所有关键操作有 loading、错误提示和重试。
6. `mvn -q -pl module-modeling,module-catalog -am test` 通过。
7. `cd onelake-app/web-console && pnpm build` 通过。

## 12. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| FQN 或字段名变化导致绑定失效 | 资产详情展示错误 | 使用 `asset_id + asset_fqn + column_name`，字段不存在时标记 `STALE` |
| 同义词误匹配 | 自动推荐错误 | 推荐默认 `PENDING`，人工确认后才 `ACTIVE` |
| 术语口径变化影响下游 API 和指标 | 生产口径漂移 | 已审定术语变更走版本和影响分析 |
| Catalog 和 Modeling 双向依赖 | 模块耦合 | 主数据在 Modeling，Catalog 只做只读聚合和投影 |
| 敏感术语直接影响权限 | 误阻断 | 安全策略先生成建议，发布/审批环节再强校验 |
| 前端一次性改动太大 | 回归风险 | 分页面切换，先 Glossary，再 CatalogSearch/AssetDetail，再其他模块 |
