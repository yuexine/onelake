# 全局任务条实施方案

> 日期：2026-06-22
> 范围：OneLake Web Console 全局任务条从前端原型走向真实运行任务视图。

## 1. 结论

全局任务条当前已经完成前端布局和原型组件，但仍停留在 mock 数据阶段。它已经出现在 `App.tsx` 顶栏触发器和底部 fixed 任务条中，数据来自 `web-console/src/mock/index.ts` 的 `runningTasks`，并由 `stores/app.ts` 初始化到 Zustand store。

下一步不应把它做成某个模块的局部组件，而应建设为 OneLake 的跨模块“长任务运行投影”：统一展示采集同步、SQL 查询、编排运行、质量稽核、存储优化、数据服务发布等异步任务的状态、进度、失败原因和跳转入口。

推荐第一版采用：

- 后端新增 `common.running_task` 投影表和 `/api/v1/tasks/running` 只读 API。
- 各业务模块先通过事件处理器或读模型同步写入任务投影。
- 前端新增 `TaskAPI` 和轻量轮询 hook，把 `TaskProgressBar` 从 mock 改为真实 API。
- 通知中心继续承担“结果通知/待处理事项”，全局任务条只承担“当前和近期运行状态”。

## 2. 整体设计要求对齐

项目设计文档要求全局布局包含 TopBar、SideNav、Content、右侧抽屉和底部全局任务条。任务条的设计语义是“底部/右下角常驻长任务进度，可最小化，点击展开实时进度，完成后 Toast + 通知中心红点”。

全局交互规范进一步要求：触发运行后先给即时 Toast，再在全局任务条展示进度，完成后给 Toast 和通知中心红点；详情页运行历史实时刷新。验收清单也明确要求“长任务有全局进度与通知”。

因此全局任务条不是独立视觉控件，而是跨模块连续性的基础设施。它要让用户在离开来源页面后仍知道任务是否在跑、是否失败、能否取消、该去哪里看日志或处理结果。

## 3. 开发现状

### 3.1 前端已完成

- `TaskProgressBar.tsx` 已有折叠触发器、展开态列表、状态图标、进度条、平均进度和空态。
- `App.tsx` 已把任务触发器放在 TopBar，并把展开态任务条 fixed 到页面底部。
- `stores/app.ts` 已有 `tasks` 状态和 `removeTask` 方法。
- `FRONTEND_VERIFICATION.md` 把“全局任务条”标记为原型完成。

### 3.2 前端未完成

- `tasks` 仍从 mock 初始化，没有真实 API。
- `RunningTask` 类型过窄，只支持 `COLLECT/COMPACTION/DAG/API/QUALITY` 和 `running/success/failed`，缺少 queued、cancelled、warning、详情链接、取消能力、起止时间、来源模块、关联对象。
- 触发采集、SQL 查询、DAG 运行等操作后，页面只处理局部状态或 message，没有统一向任务条登记运行。
- SQL 工作台当前只轮询当前页面内的 query；离开页面后全局不可见。
- 通知中心也是 mock，与任务条之间没有“完成/失败后入通知”的真实联动。

### 3.3 后端已具备的可接入来源

- 数据集成：`integration.sync_run` 和 `SyncTaskController` 已支持 run 详情、分页、日志、取消、reconcile。
- SQL 工作台：`catalog.sql_query_history` 记录 query 历史，异步查询 API 支持 submit/query/cancel，前端已轮询当前 query。
- 编排：`orchestration.job_run` 记录 DAG run，`DagController` 支持触发和 DAG 维度 run 列表。
- 质量：`quality.run_result` 和 `quality.alert` 可表达稽核结果和门禁失败。
- 公共层：`module-common` 已有 Outbox、Redis Stream 事件消费、TenantContext、common.alert。

### 3.4 后端缺口

- 没有统一任务表或统一任务 API。
- `common.notification` 只有表结构，没有对应实体、服务、Controller 或事件投影。
- `orchestration.job_run` 触发后写 RUNNING，但缺少 Dagster 终态回写。
- `quality.run_result` 更像结果记录，不表达运行中进度；第一版只能展示完成或失败结果，后续再补质量运行态。
- 存储优化、DWD/dbt/Dagster 等后续主线还没有统一运行契约。

## 4. 目标边界

### 4.1 P0 目标

把全局任务条从 mock 改成真实 API 驱动，至少覆盖：

- 采集同步运行：running、succeeded、failed、cancelled，支持跳 run 详情和取消。
- SQL 查询运行：running、succeeded、failed、cancelled，支持跳 SQL 工作台历史或当前 query。
- 编排运行：running 为主，支持跳 DAG 运行实例；终态回写可作为同阶段或下一阶段补齐。
- 质量稽核运行/告警：先展示最近失败或开放告警，后续补真实运行中进度。

### 4.2 非目标

- 不重做全局布局和视觉风格。
- 不把所有模块运行表重构成一个强耦合事务模型。
- 不在 P0 引入必须依赖 WebSocket/SSE 的推送链路。
- 不把通知中心和任务条合并成一个组件。
- 不伪造不可获得的进度百分比；缺进度时用阶段型状态。

## 5. 后端详细设计

### 5.1 数据模型

新增迁移：`common/V5__running_task.sql`。当前 common 目录已有 `V1__common.sql`、`V2__common_system_seed.sql`、`V3__alert.sql` 和 `V4__outbox_stream_contract.sql`，因此任务投影从 V5 开始。

建议表结构：

```sql
CREATE TABLE IF NOT EXISTS common.running_task (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL,
  user_id         UUID,
  source_module   VARCHAR(32) NOT NULL,
  task_type       VARCHAR(32) NOT NULL,
  ref_type        VARCHAR(64) NOT NULL,
  ref_id          VARCHAR(128) NOT NULL,
  parent_ref_id   VARCHAR(128),
  title           VARCHAR(256) NOT NULL,
  status          VARCHAR(16) NOT NULL,
  progress        INT,
  phase           VARCHAR(64),
  detail          VARCHAR(512),
  error_code      VARCHAR(64),
  error_message   TEXT,
  link            VARCHAR(512),
  cancellable     BOOLEAN NOT NULL DEFAULT false,
  cancel_endpoint VARCHAR(512),
  started_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  finished_at     TIMESTAMPTZ,
  expires_at      TIMESTAMPTZ,
  UNIQUE (tenant_id, ref_type, ref_id)
);

CREATE INDEX IF NOT EXISTS idx_running_task_tenant_status
  ON common.running_task (tenant_id, status, updated_at DESC);
CREATE INDEX IF NOT EXISTS idx_running_task_tenant_time
  ON common.running_task (tenant_id, updated_at DESC);
```

状态建议统一为：

- `QUEUED`
- `RUNNING`
- `SUCCEEDED`
- `FAILED`
- `CANCELLED`

前端可把 `QUEUED/RUNNING` 视为活跃任务，把 `SUCCEEDED/FAILED/CANCELLED` 保留一段时间作为近期结果。

### 5.2 DTO

```java
public record RunningTaskDTO(
    UUID id,
    String sourceModule,
    String taskType,
    String refType,
    String refId,
    String title,
    String status,
    Integer progress,
    String phase,
    String detail,
    String errorCode,
    String errorMessage,
    String link,
    Boolean cancellable,
    String cancelEndpoint,
    Instant startedAt,
    Instant updatedAt,
    Instant finishedAt
) {}
```

### 5.3 API

新增 `module-common` Controller：

- `GET /api/v1/tasks/running?includeRecent=true&limit=20`
  - 返回当前租户下活跃任务 + 最近 10 分钟终态任务。
- `GET /api/v1/tasks/{id}`
  - 返回单条任务投影，用于详情或调试。
- `POST /api/v1/tasks/{id}/dismiss`
  - 仅隐藏终态任务，不删除业务运行记录。
- `POST /api/v1/tasks/{id}/cancel`
  - 可选代理取消；P0 也可以只返回 `cancelEndpoint` 由前端调用业务 API。

权限：

- `DE/OPS/ADMIN` 可看本租户全量任务。
- 普通用户只看 `user_id = currentUser` 或被授权对象相关任务。
- P0 可先按租户展示，P1 再细化角色和对象授权。

### 5.4 投影写入策略

新增 `RunningTaskService`，提供幂等 upsert：

```java
upsert(refType, refId, payload)
markTerminal(refType, refId, status, detail, error)
dismiss(id)
```

数据来源分三类：

1. 事件驱动：
   - `integration.sync_run.started` upsert RUNNING。
   - `integration.table.loaded` mark SUCCEEDED。
   - `integration.sync.failed` mark FAILED。
   - `quality.check.completed/failed` 生成近期结果。
2. 业务读模型补齐：
   - 定时扫描 `integration.sync_run` 中 QUEUED/RUNNING 的记录，调用已有 reconcile 后 upsert。
   - 扫描 `catalog.sql_query_history` 最近 RUNNING/FAILED/SUCCEEDED 记录。
   - 扫描 `orchestration.job_run` RUNNING 记录。
3. 操作入口即时登记：
   - 触发同步、提交 SQL、触发 DAG 后，服务层在同一事务或返回前 upsert 任务，避免等待异步事件。

P0 推荐同时采用“操作入口即时登记 + 事件终态更新”。定时扫描作为兜底，防止事件丢失或重启期间任务条不准。

### 5.5 任务类型映射

| 来源 | refType | taskType | link | progress |
| --- | --- | --- | --- | --- |
| `integration.sync_run` | `sync_run` | `COLLECT` | `/integration/sync-tasks/{taskId}/runs/{runId}` 或失败诊断页 | Airbyte 无精确进度时 running=40，终态=100 |
| `catalog.sql_query_history` | `sql_query` | `SQL` | `/lakehouse/sql?queryId={historyId}` | running=50，终态=100 |
| `orchestration.job_run` | `dag_run` | `DAG` | `/orchestration/runs?runId={id}` | queued=10，running=50，终态=100 |
| `quality.run_result` | `quality_run` | `QUALITY` | `/quality/results?ruleId={ruleId}` | 终态=100 |
| `common.alert` | `alert` | `ALERT` | `/monitor/alerts` 或 `/quality/gate` | failed=100 |
| 后续 Compaction | `lakehouse_optimize` | `COMPACTION` | `/lakehouse/optimize?taskId={id}` | runtime 有 stats 后接真实进度 |

## 6. 前端详细设计

### 6.1 类型扩展

把 `RunningTask` 从组件内迁到 `types/index.ts`，扩展为：

```ts
export interface RunningTask {
  id: string;
  sourceModule: 'INTEGRATION' | 'LAKEHOUSE' | 'ORCHESTRATION' | 'QUALITY' | 'DATASERVICE' | 'SECURITY' | 'SYSTEM';
  taskType: 'COLLECT' | 'SQL' | 'DAG' | 'API' | 'QUALITY' | 'COMPACTION' | 'ALERT';
  refType: string;
  refId: string;
  title: string;
  status: 'QUEUED' | 'RUNNING' | 'SUCCEEDED' | 'FAILED' | 'CANCELLED';
  progress?: number;
  phase?: string;
  detail?: string;
  errorCode?: string;
  errorMessage?: string;
  link?: string;
  cancellable?: boolean;
  cancelEndpoint?: string;
  startedAt: string;
  updatedAt: string;
  finishedAt?: string;
}
```

### 6.2 API 与 store

新增：

- `TaskAPI.listRunning({ includeRecent: true })`
- `TaskAPI.dismiss(id)`
- `TaskAPI.cancel(id)`

`useAppStore` 从 mock 初始化改为：

- `tasks: []`
- `setTasks(tasks)`
- `upsertTask(task)`
- `dismissTask(id)`
- `setTaskPollingEnabled(enabled)`

新增 `useGlobalTasks` hook，在 `App.tsx` 中调用：

- 默认 5 秒轮询。
- 页面不可见时降为 30 秒。
- 任务条展开且有 RUNNING 时 2 秒轮询。
- 请求失败不弹全局错误，只在任务条内显示轻量“刷新失败，可重试”。

### 6.3 组件增强

`TaskProgressBar` 保持现有结构，只做真实化增强：

- 显示状态：排队中、运行中、已完成、失败、已取消。
- 每行支持“查看”和“取消”图标按钮。
- 失败任务显示 `errorMessage` tooltip。
- 终态任务可手动关闭。
- 空态保留“暂无运行中任务”。
- 进度缺失时显示阶段文案，不伪造精确百分比。

`TaskProgressTrigger` 调整统计：

- 活跃任务数优先展示。
- 有失败任务时用错误色。
- 无任务时显示 0，不再展示 mock 平均进度。

### 6.4 与通知中心联动

全局任务条只展示运行态和短时结果；通知中心展示需要用户事后处理的消息。

联动规则：

- 任务成功：可选生成 INFO 通知，但不强打扰；任务条保留 10 分钟。
- 任务失败：生成 TASK/ALERT 通知，通知中心红点增加；任务条保留到用户 dismiss。
- 需要审批/豁免/人工处理：进入通知中心和待办，不只停在任务条。

P0 可以先只完成任务条 API；通知中心真实化作为紧随其后的 P1，但数据模型应预留 `link/category/level`。

## 7. 分阶段实施计划

### 阶段 A：后端任务投影底座

目标：先让后端能返回统一任务列表。

改动：

- 新增 `common.running_task` 迁移。
- 新增 Entity、Repository、DTO、Service、Controller。
- 新增状态映射工具。
- 新增基于 `integration.sync_run` 的只读聚合或 upsert 逻辑。

验收：

- `GET /api/v1/tasks/running` 能返回当前租户采集运行。
- 活跃 run 会刷新后展示最新状态。
- 终态 run 返回 progress=100，失败包含 error message。

### 阶段 B：前端任务条真实化

目标：移除全局任务条 mock 数据。

改动：

- 新增 `RunningTask` 前端类型和 `TaskAPI`。
- `useAppStore` 改为 API 数据源。
- `App.tsx` 挂载 `useGlobalTasks`。
- `TaskProgressBar` 支持真实状态、查看、取消、关闭。

验收：

- 空库显示真实空态。
- 触发采集 run 后任务条出现任务。
- run 失败/成功后任务条状态变化，点击能跳转来源页。

### 阶段 C：接入 SQL 与编排运行

目标：覆盖当前最常用的开发长任务。

改动：

- SQL submit 时登记 `sql_query` 任务，query/cancel/终态写回投影。
- DAG trigger 时登记 `dag_run` 任务。
- 为编排补最小 Dagster run 状态刷新，避免 RUNNING 永久停留。

验收：

- SQL 查询离开工作台后仍可在任务条看到。
- 取消 SQL 后任务条显示 CANCELLED。
- 触发 DAG 后任务条可见并能跳运行实例。

### 阶段 D：质量、告警与通知中心真实化

目标：把失败任务和需处理事项闭环到通知中心。

改动：

- 新增 Notification Entity/Repository/Service/Controller。
- 任务失败事件生成 `common.notification`。
- NotificationCenter 从 mock 改真实 API。
- 质量稽核失败、采集失败、DaaS 发布失败统一生成通知。

验收：

- 失败任务在任务条和通知中心同时可见。
- 通知点击跳来源页并回写已读。
- 全部已读不影响任务条运行态。

### 阶段 E：生产化增强

目标：提升准确性和运行体验。

改动：

- SSE/WebSocket 推送可选增强，保留轮询兜底。
- 后端统一 TTL 清理终态任务。
- 加入用户级可见性、任务静默、批量关闭。
- 将 DWD/dbt/Dagster 模型运行事件纳入任务投影。

验收：

- 长任务状态延迟低于 5 秒。
- 重启后任务条可从数据库恢复。
- 不同租户/用户看不到无权任务。

## 8. 风险与约束

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 业务模块状态语义不一致 | 前端难统一展示 | 后端统一映射为 5 个全局状态 |
| Airbyte/Dagster 进度不可得 | 百分比不真实 | 用阶段进度和文案，不伪造精确值 |
| 编排终态未回写 | RUNNING 常驻 | 阶段 C 补 Dagster run 查询/回写 |
| 通知中心仍是 mock | 失败后红点不真实 | 阶段 D 独立真实化 |
| 轮询压力 | 顶栏常驻请求 | 只查当前租户活跃+近期任务，页面隐藏降频 |
| 权限边界过粗 | 用户看到不该看的任务 | P0 按租户，P1 补 user/object 授权过滤 |

## 9. 验证策略

后端：

- `mvn -q -pl module-common -am test -Djacoco.skip=true`
- `mvn -q -pl module-integration -am test -Djacoco.skip=true`
- 涉及 SQL/编排时追加 catalog/orchestration 模块测试。
- 直接调用 `GET /api/v1/tasks/running` 验证空态、运行态、终态、失败态。

前端：

- `cd onelake-app/web-console && pnpm exec tsc --noEmit`
- `cd onelake-app/web-console && pnpm build`
- 浏览器验证：
  - 登录后顶栏任务触发器无 mock。
  - 展开任务条无重叠、可收起。
  - 触发采集、SQL query、DAG run 后任务可见。
  - 成功、失败、取消三种状态有正确颜色、文案、跳转。

回归重点：

- 不破坏 `App.tsx` 全局布局。
- 不改变已有采集任务、SQL 工作台、DAG 页面主要布局。
- 不展示“检测到/完成/进度”等没有真实来源的文案。

## 10. 推荐立即实施切片

第一轮建议只做阶段 A + 阶段 B 的采集 run 覆盖：

1. 新增 common 任务投影表和 API。
2. 从 `integration.sync_run` 聚合真实任务。
3. 全局任务条接真实 API，移除 `runningTasks` mock。
4. 触发采集 run 后浏览器验证任务条出现、刷新、跳转、取消。

这条切片最贴合项目当前目标：在不重做样式的前提下，把一个已完成的原型能力接到真实控制面闭环，为后续 SQL、DWD/dbt/Dagster 和通知中心真实化提供统一底座。
