## 1. 现状盘点（V1 事实）

### 1.1 后端能力

| 层 | 现状 |
| --- | --- |
| 模块 | `module-orchestration`，控制面模块化单体的一个 Maven 子模块 |
| Controller | `PipelineController`、`PipelineBackfillController`、`OperatorController`、`JobRunController`、`ResourceGroupController`、`RuntimeContractController`、`DagController` |
| 数据模型 | `dag`、`job_run`、`task_run`、`pipeline_task`、`pipeline_task_edge`、`operator`/`operator_version`/`operator_install`、`resource_group`、`compute_profile`（共 11 张表） |
| 任务类型 | 仅 4 种：`SYNC_REF`、`SPARK_SQL`、`PYSPARK`、`QUALITY_GATE` |
| 触发类型 | `CRON`、`MANUAL`、`EVENT`（枚举存在） |
| 编译 | `PipelineCompileService`：单任务校验 + 图校验（端口/基数/环路）+ 拓扑排序 + 数据流入边推导（`from_tables`/`dataflow_inputs`）+ 自动生成 JOIN/DERIVE_COLUMN/SINK 的 Spark SQL |
| 运行 | `OrchestrationService.triggerPipelineRun`：编译 → 构建 Spark runConfig；默认 `LEGACY` 路径启动 `onelake_pipeline_run`，`onelake.orchestration.pipeline-execution-mode=GRAPH` 时启动 `onelake_pipeline_graph_run`，并按拓扑初始化/刷新 `task_run` 状态 |
| 状态机 | Pipeline：`DRAFT → VALIDATED → PUBLISHED`；`version` 自增整数 |
| 调度 | `PipelineSchedulerService`：Spring `@Scheduled` 60s tick，分钟粒度 cron 匹配；当前仍筛选 `PUBLISHED` 且 `onelake_pipeline_run` 的流水线，GRAPH 调度接入仍需生产化 |
| 事件 | Outbox：`pipeline.published`、`pipeline.run.succeeded/failed`、`pipeline.task.loaded` |

### 1.2 Dagster 执行链路（已落地基线与剩余短板）

存量 `LEGACY` 路径中的 `onelake_pipeline_run` 仍只有**一个 op `run_spark_task_op`**，它在一次 op 内**循环 for-loop 遍历所有 task 逐个 `spark-submit`**。这条路径继续保留，作为回退与兼容手段。

当前代码已新增 `onelake_pipeline_graph_run` / `run_pipeline_graph_op`，由 `onelake.orchestration.pipeline-execution-mode=GRAPH` 开关接入，具备节点状态回调、节点日志读取、`SINGLE` / `DOWNSTREAM` 重跑与运行取消能力。剩余短板集中在：

- GRAPH 仍是开关路径，默认化、灰度策略、调度器接入与生产回退演练还需要补齐。
- 节点观测已有日志查看/tail 与状态推进，但还缺甘特/时间线、指标趋势、SLA 告警与跨运行对比。
- `LEGACY` 路径仍按 run 终态粗粒度回刷 task_run，仅适合作为兼容回退，不宜继续承载节点级体验目标。

### 1.3 “回填”名不副实

`PipelineBackfillService` 实际是**把历史 `modeling.data_model` 迁移成 pipeline 实体**（一次性数据迁移），**不是按时间分区/业务日期的数据回填**。真正意义上的分区回填、区间重跑当前不存在。

### 1.4 算子市场与执行脱节

`OperatorService` 有丰富的 manifest 校验、字段 schema 闭合、敏感字段治理、图级资源契约校验，但：

- 算子（operator graph）校验链路与实际 `pipeline_task` 执行链路**是两套**；Spark 执行只看 `task_type` + `config`，**不消费 operator manifest/template 做代码生成**。
- 前端算子市场只是浏览/安装，**无法把算子直接拖到画布**成为可执行节点。

### 1.5 前端

- `UnifiedPipelineEditor`：三栏（Palette / DagCanvasSimple 拓扑列 / Inspector 弹窗）+ 校验向导（5 步动画）+ 运行横幅（5s 轮询）。
- `RunInstances`：运行列表 + 节点拓扑图 + 任务明细表。
- 已补齐：运行中高频轮询、逐节点状态推进、节点日志查看/tail 刷新、`SINGLE` / `DOWNSTREAM` 重跑、运行取消入口。
- 缺：甘特/时间线、指标趋势、SLA 告警、参数管理、版本对比、跨流水线依赖视图；节点日志目前是查看/刷新，不是服务端推送式实时流。

## 2. 竞品调研（2025–2026）

| 能力维度 | Dagster（asset-centric） | 阿里云 DataWorks Data Studio | Apache Airflow 3.0 |
| --- | --- | --- | --- |
| 编排范式 | 软件定义资产（SDA），围绕数据资产建 DAG，天然血缘 | DAG 画布拖拽连线，跨引擎工作流编排 | 任务 DAG + `@asset` 资产语法 |
| 声明式自动化 | Declarative Automation（`AutomationCondition`：`eager`/`on_cron`/自定义 Python），上游更新即物化 | 时间驱动周期调度 + 事件驱动触发 + 血缘解析上游触发 | 事件驱动调度（AssetWatcher 监听消息队列，如 SQS） |
| 数据质量 | Asset Checks 原生内嵌，阻断坏数据下传，统一健康视图 | 数据分类分级、敏感识别、脱敏、合规审计、风险阻断 | 依赖外部集成（dbt/Soda/GE） |
| 调度依赖 | 分区（partitions）+ 增量运行 | 同周期/跨周期/跨工作流/跨工作空间依赖；空跑/冻结/重跑/生效日期 | 统一 `schedule` 字段（时间+事件） |
| 回填 | 一等公民，大规模分区 backfill | 补数据、批量重跑 | Scheduler 托管 backfill，UI 可触发/监控/取消 |
| 版本 | 代码即配置 + 分支部署 + CI/CD 原生 | 发布到生产环境（开发/生产隔离）+ Git 集成 | DAG Versioning：run 绑定启动时版本，UI 可查历史 |
| 多引擎 | dbt/Spark/Snowflake/BigQuery/DuckDB/Kafka | MaxCompute/EMR/Flink/Hologres/ADB/PAI + Python/Notebook | 任意（Operator 生态） |
| 智能 | AI scaffolding | Copilot + Data Agent（自动化开发/自主治理/对话式分析） | — |
| 观测 | 血缘/run/分区/元数据/freshness 统一 UI | 千万级调度、智能运维监控、资源组编排 | React 新 UI、可中途暂停/取消 |

**对 OneLake 的启示（融入项目整体设计）：**

1. OneLake 已具备 Catalog 资产 + 血缘 + Outbox 事件底座，**最契合“资产感知式编排”（Dagster 路线）**——应把 `targetFqn` 资产升级为一等公民，用资产就绪/新鲜度驱动调度。
2. DataWorks 的**跨周期依赖、空跑/冻结/重跑、开发/生产发布隔离、资源组编排**是企业级调度刚需，是当前最大功能差距。
3. Airflow 3 的 **DAG 版本化 + Scheduler 托管回填 + 事件驱动**，指明版本与回填要做成一等公民。

## 4. V2 功能升级方案

按能力支柱拆分，每项标注优先级（P0 必做 / P1 重要 / P2 增强）与所属阶段。

### 4.1 支柱 A：节点级执行引擎（G1，P0）

- **A1｜Dagster 图执行生产化**：`onelake_pipeline_graph_run` / `run_pipeline_graph_op` 已接入；下一步是把 GRAPH 从开关路径推进到默认/调度路径，完善灰度、回退与运行时契约校验。
- **A2｜节点级状态回写生产化**：节点回调已更新 `task_run`；下一步补齐回调幂等审计、异常补偿看板与 `pipeline.task.run.*` Outbox 事件。
- **A3｜节点级日志增强**：后端已提供节点日志读取，前端 RunInstances 支持查看与 tail 刷新；下一步补下载体验、检索、错误摘要和实时推送。
- **A4｜节点级重试/续跑增强**：已提供 `POST /runs/{runId}/tasks/{taskKey}/rerun` 与 `SINGLE` / `DOWNSTREAM` 模式；下一步补 attempt 历史、重试策略与失败节点推荐续跑。
- **A5｜运行取消硬化**：已提供 `POST /runs/{runId}/cancel` → Dagster terminate，并将非终态 `task_run` 置 `CANCELLED`；下一步验证不同 Dagster 版本的 terminate 行为与 in-flight spark-submit 中断效果。

### 4.2 支柱 B：任务类型扩展（G2，P1）

新增 `TaskType`（保持 Spark-only 主线不破坏，新增类型分“执行型/控制型/观测型”）：

| 新类型 | 定位 | 优先级 |
| --- | --- | --- |
| `TRINO_SQL` | 轻量 SQL 转换/校验（走 Trino，不必起 Spark） | P1 |
| `PYTHON` / `SHELL` | 通用脚本节点（沙箱执行） | P1 |
| `BRANCH` / `CONDITION` | 条件分支、跳过下游 | P1 |
| `SENSOR` / `WAIT` | 等待外部资产/分区就绪、等待时间窗口 | P1 |
| `SUB_PIPELINE` | 引用并触发另一条流水线 | P2 |
| `NOTIFY` | 运行内发通知（接 `module-common` 通知总线） | P2 |
| `ASSERTION` | 通用断言（比 QUALITY_GATE 更轻，用于流程内校验） | P2 |

> 保留 `SYNC_REF/SPARK_SQL/PYSPARK/QUALITY_GATE` 不变，`TaskType` 增加 `category`（EXEC/CONTROL/OBSERVE）分组，前端 Palette 分类展示。

### 4.3 支柱 C：生产级调度（G3，P0）

- **C1｜数据日期/业务日期模型**：run 增加 `data_interval_start/end`、`logical_date`；参数支持 `${bizdate}`/`${cyctime}`。
- **C2｜跨周期依赖**：`pipeline_dependency` 表描述流水线间上下游（同周期/跨周期/偏移），调度器据此判定 ready。
- **C3｜调度增强**：每流水线 `timezone`、日历（工作日/节假日）、`catchup`（是否补历史）、misfire 策略、`max_active_runs`（单流水线并发）、优先级。
- **C4｜空跑/冻结/重跑（对标 DataWorks）**：`DRY_RUN`（不执行不阻塞）、`FROZEN`（不执行且阻塞下游）、失败自动重跑策略。
- **C5｜SLA/超时**：`sla_minutes`、`timeout_minutes`，超时/破 SLA 触发告警事件。
- **C6｜调度器演进**：中期从 Spring `@Scheduled` 分钟 tick 迁移到 **Dagster 原生 schedule/sensor** 或引入持久化调度队列，避免多实例重复触发与分钟粒度漂移（当前 tick 存在多副本重复触发风险，需加分布式锁/幂等）。

### 4.4 支柱 D：分区与回填（G4，P0）

- **D1｜分区模型**：`pipeline_task` 增加分区定义（时间分区键、粒度）；`task_run` 绑定分区。
- **D2｜真回填**：`POST /pipelines/{id}/backfill`（指定日期区间 + 分区），由**调度器托管**生成多个 run，UI 可视化进度、可暂停/取消/重试（对标 Airflow 3 / DataWorks 补数据）。
- **D3｜命名纠偏**：现 `PipelineBackfillService` 更名为 `ModelMigrationService`（历史模型迁移），把 `backfill` 语义让位给真正的数据回填，避免概念混淆。

### 4.5 支柱 E：资产感知的声明式自动化（G5，P1）

- **E1｜资产就绪触发**：复用现有 `pipeline.task.loaded` / `integration.table.loaded`，建立**资产 → 下游流水线**订阅（`PipelineSyncRefTriggerHandler` 泛化为通用触发器）。
- **E2｜新鲜度策略**：把边上已有的 `freshnessPolicy` 从“编译提示”升级为“运行触发条件”（`LATEST`/`SAME_FRESHNESS_WINDOW`/`SAME_BATCH` 真正参与 ready 判定）。
- **E3｜声明式条件**（对标 Dagster AutomationCondition）：流水线级“当上游资产更新且质量通过则自动运行”。

### 4.6 支柱 F：版本、发布与环境（G6，P1）

- **F1｜不可变版本快照**：发布时对 `dag + tasks + edges + config` 做整包快照（`pipeline_version` 表，JSON 存储），run 绑定版本（对标 Airflow DAG Versioning）。
- **F2｜版本对比/回滚**：前端 diff 视图，一键回滚到历史版本。
- **F3｜发布审批**：接 `module-security` 审批中心，`PUBLISHED` 前走审批（可配开关）。
- **F4｜开发/生产隔离**：引入 `environment`（DEV/PROD）或“发布到生产”语义，草稿在开发态迭代，发布后进入生产调度（对标 DataWorks）。

### 4.7 支柱 G：算子市场打通执行（G7，P2）

- **G1｜算子→Spark 代码生成**：`OperatorService` 的 manifest `template`（`SPARK_SQL`/`SELECT_EXPR`/`FILTER`/`JOIN`/`AGG`/`QUALITY_ASSERT` 等）在 `PipelineCompileService` 落地为 SQL 生成器，使算子成为真正可执行节点。
- **G2｜画布拖拽算子**：Palette 直接列出已安装算子，拖入即生成对应 `pipeline_task`（`operator_ref` + `config`）。
- **G3｜算子版本锁定**：run 记录算子版本，保证可复现。

### 4.8 支柱 H：参数体系（G8，P1）

- **H1｜三级参数**：全局（租户/项目）/ 流水线 / 节点参数，优先级覆盖。
- **H2｜动态时间参数**：`${bizdate}`、`${bizdate-1}`、`${cyctime}`、自定义格式。
- **H3｜节点间传参**：上游输出（行数/产物路径/自定义 KV）注入下游 config。

### 4.9 支柱 I：观测与告警（G9，P1）

- **I1｜运行大盘**：成功率、时长趋势、失败 Top、资源消耗（补 `/monitor/overview` 真实数据）。
- **I2｜甘特/时间线**：单 run 节点时间线视图。
- **I3｜失败/SLA 告警**：接 `module-common` 通知 + `module-quality` 告警，破 SLA/失败自动推送。
- **I4｜血缘联动**：run 视图直达 Catalog 资产与字段血缘。

## 5. 数据模型变更（增量迁移，`orchestration` schema）

```sql
-- task_run 节点级增强（A2/A3/A4）
ALTER TABLE orchestration.task_run
  ADD COLUMN attempt INT NOT NULL DEFAULT 1,
  ADD COLUMN max_retries INT NOT NULL DEFAULT 0,
  ADD COLUMN log_ref VARCHAR(512),
  ADD COLUMN data_interval_start TIMESTAMPTZ,
  ADD COLUMN data_interval_end TIMESTAMPTZ;

-- job_run 数据日期 + 回填标记（C1/D2）
ALTER TABLE orchestration.job_run
  ADD COLUMN logical_date TIMESTAMPTZ,
  ADD COLUMN data_interval_start TIMESTAMPTZ,
  ADD COLUMN data_interval_end TIMESTAMPTZ,
  ADD COLUMN backfill_id UUID,
  ADD COLUMN pipeline_version_id UUID;

-- 流水线级调度增强（C3/C4/C5）
ALTER TABLE orchestration.dag
  ADD COLUMN timezone VARCHAR(64) DEFAULT 'Asia/Shanghai',
  ADD COLUMN catchup BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN max_active_runs INT NOT NULL DEFAULT 1,
  ADD COLUMN priority INT NOT NULL DEFAULT 5,
  ADD COLUMN sla_minutes INT,
  ADD COLUMN timeout_minutes INT,
  ADD COLUMN schedule_mode VARCHAR(16) DEFAULT 'NORMAL'; -- NORMAL/DRY_RUN/FROZEN

-- 新增表
CREATE TABLE orchestration.pipeline_version (...);      -- F1 版本快照
CREATE TABLE orchestration.pipeline_dependency (...);   -- C2 跨流水线依赖
CREATE TABLE orchestration.backfill (...);              -- D2 回填批次
CREATE TABLE orchestration.pipeline_param (...);        -- H1 参数
```

> 迁移遵循 `bootstrap/.../db/migration/orchestration/V10+`，兼容存量数据（默认值不破坏现有流水线）。

## 8. 里程碑（建议 4 阶段）

| 阶段 | 主题 | 关键交付 | 支柱 |

| **M1（基石）** | 节点级执行 + 真回填 + 调度稳态 | A1–A5、D1–D3、C6（调度幂等/分布式锁） | A/D/C |

## 9. 风险与约束

- **R1 执行模型改造（A1）风险最高**：Dagster 从单 op 改为动态图，需同步改 `SparkRunConfigBuilder`、`task_run` 回写、事件契约，建议先在样例流水线灰度，保留旧路径回退。
- **R2 调度并发安全**：当前 Spring tick 在多实例下会重复触发，M1 必须先加分布式锁/触发幂等（可用 Outbox 幂等表或 Redis 锁）。
- **R3 概念对齐**：`backfill` 命名纠偏需同步更新 `docs/流水线模块重设计方案.md`、`docs/ODS到DWD标准闭环实施方案.md`，避免文档与代码再次漂移（遵循 `RTK.md` 第 10 节）。
- **R4 Spark-only 边界**：新增 `TRINO_SQL`/`PYTHON` 会打破“Spark-only 主线”表述，需在 ADR 明确多引擎口径，并同步 `RuntimeContractService` 与算子 `compileTarget` 校验。
- **R5 契约兼容**：`pipeline.run.*` / `pipeline.task.loaded` 事件被 catalog/quality/modeling 消费，扩展 payload 需保持向后兼容（参考 `docs/全局事件契约（Outbox 总线 · 事件清单 · 发布订阅 · 载荷 Schema）.md`）。
