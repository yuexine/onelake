# 流水线模块重设计方案（数据开发与治理编排中枢）

> 状态：实施修订 **v3**（Spark-only 主链路已落地；v2 设计过程保留为决策记录）
> 适用模块：`module-orchestration`、`module-modeling`、`web-console/src/pages/orchestration/*`、`dagster/`、数据面（Trino / 新增 Spark）
> 最新落地口径：
> 1. 流水线主链路统一使用 **Spark 引擎**，不再开放历史 `SQL_MODEL`、`FIELD_GOVERNANCE`、`TRINO_DBT` 或旧 DWD/dbt 模型运行入口。
> 2. `pipeline_task`、`pipeline_task_edge`、`task_run` 已成为主链路事实源；边记录端口、资产 FQN、别名、触发策略和新鲜度策略。
> 3. 当前可建模节点收敛为 `SYNC_REF`、`SPARK_SQL`、`PYSPARK`、`QUALITY_GATE`，并通过 `dataflow.nodeKind` 支持结构化 `JOIN`、`DERIVE_COLUMN`、`SINK`。
> 4. 下游输入由数据流边推导，Join 的 left/right、Spark 的 from_tables、Sink 的输入表不再要求用户重复手填。
> 5. `onelake_pipeline_run` 按 DAG 拓扑执行 Spark 任务，运行实例展示多节点、多边、端口、产物表、行数和上游失败传播。
>
> 下面 v2 设计中的 dbt-on-Trino、`SQL_MODEL`、`FIELD_GOVERNANCE`、`TRINO_DBT` 内容仅作为历史设计与迁移边界说明；当前新建流水线以 Spark-only 为准。
>
> v2 决策基线（用户确认）：
> 1. 产品形态：**统一编辑器**（合并通用画布与 DWD 工作台，一条流水线混放多类任务）
> 2. 执行引擎：以 dbt-on-Trino 为基线，**架构必须可扩展 Spark 引擎**（Spark 为真实执行路径，非契约态）
> 3. 任务持久化：**新建 `orchestration.pipeline_task` 一等实体表**
> 4. 本轮范围：**P1 → P4 完整闭环**
> 5. 先产出本设计文档，再开工

> **v2 澄清（评审后修订，实施前必读）**：
> - **C1 单源真相**：`pipeline_task` 仅持 DAG 语义；`SQL_MODEL`/`FIELD_GOVERNANCE` 的 SQL/治理配置以 `modeling.data_model` 为唯一真相，禁止写入 `pipeline_task.config`（见 §6.1）
> - **C2 三层依赖**：dbt 内部 `ref()` / `pipeline_task_edge` / Dagster 跨引擎 ops 三层职责分离，禁止重复表达（见 §6.3.1）
> - **C3 SPI 重命名**：`TaskExecutor` → `EngineRunConfigBuilder`，Java 侧只构建 runConfig，执行在 Dagster（见 §6.3）
> - **C4 事件触发迁移兼容**：删 `DwdOdsLoadedEventHandler` 前必须完成 `PipelineBackfillService` 回填且覆盖率 100%（见 §7 P1 / §7 P5）
> - **C5 资源画像分阶段**：Spark 资源画像在 P-Spark 落地，dbt/Trino resource group 真实路由推迟到后续阶段（见 §6.4）

## 1. 背景与问题诊断

### 1.1 现状（基于代码实测）

| 入口 | 文件 | 真实状态 |
|------|------|---------|
| 新建流水线 → 通用画布 | `DagCanvas.tsx` | 能存不能跑：`dagsterJob` 硬编码 `sql_workbench_draft`（L728），Dagster 无此 job，`triggerable` 永远 false。试运行/版本/发布均为写死假数据。 |
| 新建 DWD 治理流水线 | `DwdPipelineWorkbench.tsx` | 能跑但僵化：仅"单 ODS 表 → 单 DWD 表"，5 个固定阶段，本质是 `GovernanceFactory` 外包一层 Steps。 |

**核心矛盾**：灵活的路（画布）跑不了，能跑的路（DWD 工作台）不灵活 → 无法"灵活开展数据治理与开发"。

### 1.2 根因

1. **产品形态分裂**：两个入口、两套 definition（`operator_graph` vs `DWD_MODEL_DAG`）、两条触发路径。
2. **算子图与执行脱节**：65 个内置算子仅作 Manifest/校验/可视化；真正生成 SQL 的是 `DwdModelService.generateModelSql`（仅字段映射 + lookup join + 增量），算子 `template.sql` 从未逐节点编译。"图即程序"是空壳。
3. **执行运行时极窄**：Dagster 实际仅 2 个 job（`onelake_dbt_model_run` + log stub）。SPARK/PYTHON 全 contract-only，`scheduleCron` 无调度器。
4. **架构债**：跨 schema 直写（`ensureDwdDag` 直写 orchestration.dag、`createDwdModelRun` 直写 modeling.model_run）；`integration.table.loaded` 双重消费且匹配规则不一致；编排模块零 Outbox 生产；`JobRun.SUCCESS` 与 `model_run.SUCCEEDED` 枚举不一致；`run_dwd_model` op 忽略资源画像。

## 2. 竞品调研与借鉴

| 产品 | 核心范式 | 借鉴点 |
|------|---------|--------|
| 阿里 DataWorks | 数据开发 IDE + 业务流程 DAG，节点类型化（SQL/同步/PyODPS/虚拟节点），提交→发布到生产 | 节点类型化、开发态/生产态分离、提交发布工作流 |
| Databricks Workflows + DLT | Workflow=多 Task DAG（notebook/SQL/dbt/pipeline）；DLT 声明式 + expectations（质量） | 声明式依赖自动连边、质量即一等公民、Task 多引擎 |
| Dagster | 一切皆 asset，dbt 模型→asset，dbt tests→asset checks | 用 dbt ref 图作为真实执行 DAG，asset check=质量门禁 |
| dbt Cloud | model 即开发单元，`dbt build --select` 按 DAG 跑 | 流水线运行=按选择器跑一组 dbt 模型（OneLake 当前唯一真实可执行能力，放大它） |
| Azure Data Factory / Fabric | 可视化 pipeline + data flow（编译为 Spark） | 可视化转换必须编译成真实执行物；多引擎 |
| AWS Glue | Catalog + ETL job + Data Quality 内嵌 | 治理/质量与加工同图同运行 |

**结论**：
1. 不自造执行引擎，**复用 dbt-on-Trino 作为基线真实引擎**；流水线 = 一组带 `pipeline tag` 的 dbt 模型，运行 = `dbt build --select tag:pipeline_<id>`，dbt 负责 DAG/增量/依赖。
2. **执行层做成引擎可插拔（TaskExecutor SPI）**，dbt-on-Trino 与 Spark 为并列的真实 executor（满足"扩展 Spark"）。
3. **节点类型化**：一条 DAG 混放 SQL 模型、字段治理、质量门禁、同步引用、（后续）Spark/Python 任务。
4. **治理与开发统一在一张可执行 DAG**。

## 3. 重定义：功能目标

> **定位**：流水线是 OneLake 的**数据开发与治理编排中枢**——用一张可执行 DAG，把"取数 → 加工(SQL/Spark/模型) → 治理(字段处理/脱敏) → 质量门禁 → 产出(ODS/DWD/DWS/ADS)"串成可调度、可观测、可发布、可血缘的整体。

### 3.1 North Star 能力

1. **一种流水线，多种任务**：统一为 `Pipeline(Flow) = Tasks 的 DAG`。任务类型：
   - `SQL_MODEL`：核心开发单元，写 SQL/选字段，编译为 dbt 模型（任意分层）。
   - `FIELD_GOVERNANCE`：字段治理 Recipe，复用 `GovernanceFactory`。
   - `QUALITY_GATE`：质量门禁，编译为 dbt test / asset check。
   - `SYNC_REF`：引用采集任务，建立运行先后依赖。
   - `SPARK_SQL` / `PYSPARK`：Spark 引擎任务（真实执行路径，见 §6.4）。
   - （后续）`PYTHON`。
2. **真实可执行**：新增 Dagster `onelake_pipeline_run` job，按引擎分派 executor。
3. **依赖自动连边**：任务引用上游产出表 → 自动连边（dbt ref / 表级 FQN）。
4. **治理内嵌**：脱敏/字段处理/字典映射编译进 SQL，密级沿血缘继承。
5. **开发态/生产态分离**：草稿 → 校验(VALIDATED) → 发布(PUBLISHED) 真实状态机 + Outbox。
6. **可观测可调度**：真实运行实例、节点级状态、运行日志、cron 调度、失败重跑、最近运行聚合。
7. **架构合规**：消灭跨 schema 直写、统一 trigger、统一状态枚举、编排产出 Outbox。

### 3.2 明确边界（不做/后做）

- 不做多环境物理隔离（dev/test/prod）；本轮去掉假环境切换或降级为 tag。
- 提交审批/灰度发布本轮为占位并明确标注"后续"。
- `PYTHON` 任务本轮仅契约态（明确标注未接入运行时）。
- 算子市场不追求 65 算子全部逐节点编译；退化为"任务模板 + 字段处理器库"。

## 5. 交互设计要点

1. **从资产进入即预填**：分层表"治理成表/派生 DWD" → 统一编辑器并预置 ODS 输入 + 治理/SQL 任务（保留 `?template=ods-dwd&sourceAssetId=`）。
2. **依赖自动连边**：SQL 任务 `from dwd_xxx`，若 `dwd_xxx` 是同流水线另一任务产出 → 自动连边并提示；删除上游给出影响提示。
3. **校验即所得**：保存前调真实 `validateGraph`，错误定位到具体任务节点（节点标红 + 检查器显示）。去掉假"一键定位"。
4. **运行可观测**：点"运行"→ 真实创建运行实例 → 节点级状态实时刷新（轮询 Dagster run + dbt/Spark 节点结果），替换假试运行。
5. **发布工作流**：草稿 →（校验通过）发布 → 真实状态机 + `pipeline.published` Outbox。去掉假灰度弹窗（降级为后续占位）。
6. **质量门禁结果回流**：运行后门禁 PASS/FAIL 真实展示，FAIL 阻断发布（联动 quality 模块）。
7. **空态/错误态**：全部真实 loading/empty/error，不再 mock fallback。

## 6. 后端重构设计

### 6.1 统一数据模型

> **C1 单源真相原则**（实施红线）：
> - `SQL_MODEL` / `FIELD_GOVERNANCE` 任务的 SQL 文本、字段映射、治理规则**只存** `modeling.data_model`，`pipeline_task.model_id` 指向它。
> - `pipeline_task.config` (jsonb) **只用于无 `model_id` 的任务**：`SPARK_SQL` / `PYSPARK` / `QUALITY_GATE`（纯表达式）/ `SYNC_REF`（引用配置）。
> - 任何代码路径若发现 `pipeline_task.config` 与 `modeling.data_model` 内容冲突，以 `data_model` 为准并告警。
>
> 违反此原则会导致编辑器/dbt 编译器/运行视图三方数据漂移。

```sql
-- 流水线(Flow) 升级 orchestration.dag
ALTER TABLE orchestration.dag
  ADD COLUMN pipeline_kind   text DEFAULT 'BLANK',   -- BLANK | ODS_DWD | MULTI_LAYER
  ADD COLUMN status          text DEFAULT 'DRAFT',   -- DRAFT | VALIDATED | PUBLISHED
  ADD COLUMN engine          text DEFAULT 'TRINO_DBT',
  ADD COLUMN resource_group  text DEFAULT 'default',
  ADD COLUMN compute_profile text DEFAULT 'trino-small';
-- definition(jsonb) 仅保留画布布局快照(nodes 坐标/edges)，任务事实落 pipeline_task

-- 任务(Task) 一等实体（用户决策：task_table）
-- 仅持 DAG 语义 + 引用外键，不复制业务配置（C1）
CREATE TABLE orchestration.pipeline_task (
  id              uuid PRIMARY KEY,
  tenant_id       uuid NOT NULL,
  dag_id          uuid NOT NULL,
  task_key        text NOT NULL,           -- 画布节点 id，DAG 内唯一
  task_type       text NOT NULL,           -- SQL_MODEL|FIELD_GOVERNANCE|QUALITY_GATE|SYNC_REF|SPARK_SQL|PYSPARK
  name            text NOT NULL,
  engine          text NOT NULL DEFAULT 'TRINO_DBT', -- TRINO_DBT|SPARK_SQL|PYSPARK
  target_fqn      text,                    -- 产出表(用于跨引擎依赖/血缘)
  model_id        uuid,                    -- SQL_MODEL/FIELD_GOVERNANCE 关联 modeling.data_model（C1 真相源）
  sync_task_id    uuid,                    -- SYNC_REF 引用的采集任务
  config          jsonb NOT NULL DEFAULT '{}'::jsonb,
                  -- 仅当 task_type ∈ {SPARK_SQL, PYSPARK, QUALITY_GATE, SYNC_REF} 时写入
                  -- 严禁存放 SQL_MODEL/FIELD_GOVERNANCE 的 SQL/字段映射/治理规则
  compile_status  text NOT NULL DEFAULT 'DRAFT',    -- DRAFT|VALIDATED|FAILED
  compile_error   text,                    -- 编译失败时的节点级错误（§6.7 校验契约）
  executable      boolean NOT NULL DEFAULT false,
  position_x      int, position_y int,
  created_at      timestamptz DEFAULT now(),
  updated_at      timestamptz DEFAULT now(),
  UNIQUE (dag_id, task_key)
);
CREATE INDEX idx_pipeline_task_model ON orchestration.pipeline_task(model_id) WHERE model_id IS NOT NULL;
CREATE INDEX idx_pipeline_task_dag   ON orchestration.pipeline_task(dag_id);

-- 任务边（仅跨引擎/同步引用/显式手工连线，dbt ref 不入此表 — 见 §6.3.1 C2）
CREATE TABLE orchestration.pipeline_task_edge (
  id          uuid PRIMARY KEY,
  dag_id      uuid NOT NULL,
  source_key  text NOT NULL,
  target_key  text NOT NULL,
  edge_layer  text NOT NULL DEFAULT 'PIPELINE',  -- PIPELINE|CROSS_ENGINE (见 §6.3.1)
  source_port text DEFAULT 'out',
  target_port text DEFAULT 'in',
  auto        boolean DEFAULT false,       -- 是否由依赖自动连边生成（仅 CROSS_ENGINE 层）
  created_at  timestamptz DEFAULT now(),
  UNIQUE (dag_id, source_key, target_key, edge_layer)
);
CREATE INDEX idx_pipeline_task_edge_dag ON orchestration.pipeline_task_edge(dag_id);

-- 运行实例：统一状态枚举 + 节点级运行
-- RunStatus: QUEUED|RUNNING|SUCCEEDED|FAILED|CANCELLED  (统一 orchestration 与 modeling，C8 枚举迁移)
CREATE TABLE orchestration.task_run (
  id            uuid PRIMARY KEY,
  job_run_id    uuid NOT NULL,             -- 关联 orchestration.job_run
  task_key      text NOT NULL,
  status        text NOT NULL DEFAULT 'QUEUED',
  rows_written  bigint,
  scan_bytes    bigint,
  error_msg     text,
  artifact_path text,                      -- run_results.json / Spark event log 在 MinIO 的路径（见 §12）
  started_at    timestamptz, finished_at timestamptz
);
CREATE INDEX idx_task_run_job ON orchestration.task_run(job_run_id);
```

> **迁移文件**：
> - 新增 `orchestration/V4__pipeline_task.sql`（建表 + 索引）
> - `orchestration/V5__dag_pipeline_fields.sql`（dag 扩字段、job_run 状态枚举对齐 `SUCCESS → SUCCEEDED`，C8）
> - 历史 `DWD_MODEL_DAG` definition 由 `PipelineBackfillService`（P1）惰性回填到 pipeline_task；**不写历史 migration SQL 做批量回填**，避免迁移失败锁库（见 §7 P1）。

### 6.2 统一编译链路（PipelineCompileService）

把 `DwdModelService` 的单模型编译能力上提为"流水线编译器"：

```
compile(dagId):
  按拓扑序遍历 pipeline_task:
    SQL_MODEL / FIELD_GOVERNANCE (engine=TRINO_DBT)
        → 生成 dbt 模型: models/<layer>/<dbt_model>.sql  (含 config(tags=['pipeline_<id>']))
                         models/<layer>/<dbt_model>.yml  (含质量门禁 tests)
        → 上游引用用 dbt ref()/source() 表达 → dbt 自动连 DAG
    QUALITY_GATE → 合并为所属模型 .yml tests (复用现有 generateSchemaYaml)
    SPARK_SQL / PYSPARK (engine=SPARK_SQL/PYSPARK)
        → 生成 Spark 执行物: spark/jobs/pipeline_<id>/<task_key>.sql 或 .py
        → 跨引擎依赖用 Iceberg 表 FQN 表达(共享 Hive Metastore)，DAG 顺序由 Dagster 保证
    SYNC_REF → 不产执行物，仅记录运行先后依赖
  生成/更新 sources.yml(聚合所有 ODS 依赖，复用阶段87聚合逻辑)
  dbt parse 校验 (Trino 任务) / Spark 脚本静态校验
  回写 dag.status=VALIDATED，pipeline_task.compile_status/executable
```

要点：一条流水线 = 一组带同一 `pipeline tag` 的执行物；dbt 负责 Trino 子图依赖解析；跨引擎依赖在 Dagster 层按表级 DAG 排序。

### 6.3 统一执行链路（引擎可插拔 SPI）

> **C3 SPI 重命名**（评审后修订）：
> 旧名 `TaskExecutor` 让人误以为 Java 侧真的执行 dbt/Spark——实际 Java 只构建 `runConfig`，执行永远在 Dagster。重命名为 `EngineRunConfigBuilder`，方法签名改为返回 `RunConfig`，与 Dagster 侧 `run_dbt_subgraph_op` / `run_spark_task_op` 一一对应。

```java
// module-orchestration service/spi
interface EngineRunConfigBuilder {
  EngineType engine();                 // TRINO_DBT | SPARK_SQL | PYSPARK
  DagsterRunConfig build(TaskBundleContext ctx);
  //   TRINO_DBT  → { dbt_select: "tag:pipeline_<id>", profiles: ... }
  //   SPARK_SQL  → { tasks: [{task_key, sql, resource_profile}], iceberg_catalog: ... }
  //   PYSPARK    → { tasks: [{task_key, script_path, resource_profile}], ... }
}

record DagsterRunConfig(String jobName, Map<String,Object> opConfig) {}
```

- **职责边界**：
  - **Java 侧**：只构建 runConfig 并 launch Dagster job；轮询 Dagster run 状态；解析 run_results.json/event log 回写 task_run。
  - **Dagster 侧**：真正执行（dbt CLI / spark-submit）；产出 run_results.json / Spark event log 到 MinIO（见 §12）。
  - **不要** 在 Java 里执行 dbt/spark；不要在 Dagster 里解析业务语义（只看 runConfig 字面值）。

- 控制面 `OrchestrationService.triggerDag` 统一入口：
  ```
  triggerReadiness:
    status ∈ {VALIDATED, PUBLISHED}
    && 存在 executable=true 的任务
    && 引擎运行时就绪 (RuntimeContractService 查询)
  创建 JobRun(QUEUED) + 各 task_run(QUEUED)
  按 engine 分组任务 → 每个 engine 调用对应 EngineRunConfigBuilder.build()
  → 拼装成 Dagster runConfig（含 op 间显式 dep，见 §6.3.1 C2）
  dagster.launch("onelake_pipeline_run", runConfig, tags)
  → RUNNING / FAILED
  refreshRunStatus: 查 Dagster run + 解析 dbt run_results / Spark event log → 节点级 task_run 状态
  ```

- Dagster 侧新增 job（**固定 2 类 op，不要按 task 数动态生成**，避免与 dbt 内部 DAG 打架）：
  ```python
  @op(config_schema={"pipeline_id": str, "tenant_id": str, "run_id": str, "dbt_select": str, "profiles_dir": str})
  def run_dbt_subgraph_op(context):
      # 单次 dbt build --select tag:pipeline_<id>，dbt 内部 DAG 由 ref() 解析
      # 产物 run_results.json 上传 MinIO: artifacts/<run_id>/run_results.json
      return {"artifact_path": ...}

  @op(config_schema={"tasks": [{"task_key": str, "sql_or_script": str, "resource_profile": dict}], "iceberg_catalog": str})
  def run_spark_task_op(context):
      # 按 task_key 顺序 spark-submit，每任务完成后增量回写 task_run
      ...

  # job 定义里显式连 dep：
  # spark_root = run_spark_task_op(run_spark_task_op, ...) 取决于 dbt 与 spark 的拓扑（C2）
  onelake_pipeline_run = graph_with_dependencies(...)  # 见 §6.3.1
  ```

**消灭跨 schema 直写**：
- 运行记录统一落 orchestration（`job_run` + `task_run`）。modeling 不再被 orchestration JDBC 直写。
- modeling 通过消费 `pipeline.task.loaded` 事件更新自身模型状态/血缘。
- 编排发 Outbox：`pipeline.published`、`pipeline.run.succeeded`、`pipeline.run.failed`、`pipeline.task.loaded`，由 catalog/quality/security/通知消费（血缘、质量分、PII、全局任务条）。
- **payload schema 必须在 P1 启动前先在 `docs/全局事件契约（Outbox 总线 · 事件清单 · 发布订阅 · 载荷 Schema）.md` 定义**（见 §7 P1 前置工作）。

#### 6.3.1 三层依赖语义（C2 实施红线）

> **核心问题**：原方案把 dbt ref、pipeline_task_edge、Dagster 跨引擎 dep 混在一起，会导致 edge 表变成 dbt manifest 的拷贝，同步是噩梦。拆为三层，每层各司其职。

| 层次 | 何时解析 | 由谁负责 | 落到哪 |
|------|---------|---------|--------|
| **L1: dbt 模型内部 `ref()`** | 编译期 | dbt 自身 | `models/*.sql` 里 `{{ ref('upstream') }}`；**不入** `pipeline_task_edge` |
| **L2: 流水线任务先后**（PIPELINE 层） | 编译期/触发期 | `pipeline_task_edge` (`edge_layer='PIPELINE'`) | SYNC_REF 上游、显式手工连线、跨 dbt 模型的非 ref 依赖 |
| **L3: 跨引擎表级**（CROSS_ENGINE 层） | 运行期 | Dagster op 显式 dep | `pipeline_task_edge` (`edge_layer='CROSS_ENGINE'`) 仅记录 auto=true 的边（target_fqn 匹配） |

**写入规则**：
- `SQL_MODEL A from {{ ref('B') }}`（B 同流水线）→ **只写 dbt ref**，不写 edge。
- `SYNC_REF` 任务前置 `SQL_MODEL` → **写 PIPELINE edge**。
- dbt 模型产出 Iceberg 表 X，`SPARK_SQL` 任务读 X → **写 CROSS_ENGINE edge**（auto=true，由编译期 target_fqn 匹配推导）。
- Spark 任务产出表 Y，下游 dbt 模型 source('iceberg','Y') → **写 CROSS_ENGINE edge**。

**Dagster 层固定 op 拓扑**（**不动态生成**）：
```
run_dbt_subgraph_op
   ↓ (仅当存在 CROSS_ENGINE edge: dbt→spark)
run_spark_task_op × N (按 PIPELINE + CROSS_ENGINE 拓扑序串行)
   ↓ (仅当存在 CROSS_ENGINE edge: spark→dbt，下一轮 dbt subgraph)
run_dbt_subgraph_op_2  # 后续 dbt 任务（当前轮简化：限制 spark 不能反向喂回同一 pipeline 的 dbt）
```

> **实施约束**（P1 范围）：
> - v1 只支持 **单向 dbt → Spark** 跨引擎依赖（覆盖 95% 场景）。
> - 若同一流水线出现 Spark → dbt 反向依赖，编译期报错并提示用户拆分流水线。
> - v2（P6 或后续）支持双向时再扩展 op 拓扑。

### 6.4 Spark 引擎扩展（真实路径，非契约态）

**数据面**：在 `docker-compose.yml` 新增 Spark 运行时（Spark + Iceberg runtime + S3A/MinIO + 共享 Hive Metastore catalog），与 Trino 共用同一 Iceberg catalog，保证 Trino/Spark 读写同一批 Iceberg 表（跨引擎依赖在表级闭合）。
- 形态选型（P-Spark 阶段定）：Spark Thrift Server（Spark SQL，JDBC 提交）或 Dagster 内 PySpark op（`pyspark` + `iceberg-spark-runtime`）。优先 Dagster PySpark op，部署最简、与现有 op 模式一致。

**执行**：Dagster 新增 `run_spark_task_op`（见 §6.3）：对 `SPARK_SQL` 跑 Spark SQL，对 `PYSPARK` 跑提交的 PySpark 脚本；产出写回 Iceberg；解析结果回写 `task_run`。

**C5 资源画像分阶段落地**（评审后修订）：
原方案"修复现状 op 忽略资源画像"过于乐观——dbt/Trino 与 Spark 的资源画像是两套基础设施工作，不能在一个阶段一并落地。拆为：
- **P-Spark 阶段落地**：Spark 资源画像真实生效（`run_spark_task_op` 读 `compute_profile`，映射为 `spark-submit --executor-memory/--executor-cores/--num-executors`）。Spark runtime 本身就要起，资源画像一并落地成本低。
- **后续阶段（P7+ 或独立基础设施 ticket）**：dbt/Trino resource group 真实路由。这依赖：
  - Trino resource group 配置体系（需 coordinator 端 `resource-groups.properties`）
  - dbt `profiles.yml` 按 `resource_group` 字段生成对应 `x-trino-resource-group` session property
  - 不在 P1-P4 范围内，避免承诺无法兑现的能力。
- **过渡期**：`run_dbt_subgraph_op` 读 `resource_group` 但只写日志（与现状一致），UI 上明确标注"Trino 资源组待基础设施就绪"，不假装生效。

**校验**：`OperatorService.validateGraph` 与 Manifest 校验放开 `engine=SPARK`/`compileTarget=SPARK` 的图级执行（依赖 P-Spark Dagster op 就绪后由 `RuntimeContractService` 标记 READY）。

### 6.5 调度与事件

- `scheduleCron`：控制面 Spring `@Scheduled` 扫描到期且 PUBLISHED 的流水线 → `triggerDag(CRON)`（最小实现；后续可迁 Dagster schedule）。
- **C4 事件触发迁移**（评审后修订）：
  - **目标终态**：**统一由 orchestration** 消费 `integration.table.loaded`，按 DAG 内 `SYNC_REF` 任务的 `target_fqn` 匹配触发；删除 modeling 侧 `DwdOdsLoadedEventHandler` 的重复触发路径，统一匹配规则。
  - **迁移路径**（不可一步到位）：
    1. **P1**：先在 orchestration 实现 `PipelineSyncRefTriggerHandler`（按 SYNC_REF.target_fqn 匹配），**与** modeling 侧 handler 共存。

### 6.6 算子市场重定位

### 6.7 校验 API 契约（评审后新增）

> **背景**：原方案只提"`validateGraph` 增强为任务级 + 依赖闭合 + 引擎契约校验"，但未定义 API 形状。前端"校验即所得"（§5 第 3 条）需要可定位到具体节点的错误返回。

## 8. 风险与应对

| 风险 | 应对 |
|------|------|
| Spark 数据面引入成本高 | P-Spark 独立阶段；执行层 SPI 先就位，Spark runtime 后接，期间 SPARK 任务明确"未就绪"而非假可执行 |
| 历史 `DWD_MODEL_DAG` 数据兼容 | `PipelineBackfillService`（P1）惰性回填 + dry-run 验证；保留兼容读取（老 `onelake_dbt_model_run` 不删），不破坏现有已编译模型；P5 才清理 |
| 跨引擎依赖一致性 | 三层依赖语义（§6.3.1）：dbt 子图内用 ref（L1，不入 edge 表），跨引擎用 CROSS_ENGINE edge + Dagster 显式 op dep（L3），同步/手工用 PIPELINE edge（L2） |
| 一次性重写破坏现有闭环 | 渐进式：P1 先打底座不改前端，P2 再合并前端；P3 完成前老 workbench 不删；每阶段独立可验证 |
| 编排/建模职责边界 | modeling 仅负责"模型定义/字段映射/dbt 产物语义"（C1 真相源），orchestration 负责"DAG/任务/运行/调度"，跨模块只走 Outbox |
| **C1 双源真相漂移**（新） | DB 约束 + 代码 review：`pipeline_task.config` 在 SQL_MODEL/FIELD_GOVERNANCE 任务上强制为 `{}`；编译器只读 `modeling.data_model`；CI 加 grep 检查 |
| **C4 事件触发迁移期断流**（新） | P1 起 orchestration 与 modeling handler 共存；P5 删 modeling handler 前必须验证 backfill 覆盖率 = 100% |
| **C8 枚举迁移破坏前端**（新） | P0 审计 + P5 DB UPDATE 迁移 + Outbox payload schema 版本化（v1=SUCCESS, v2=SUCCEEDED），消费方按版本兼容 |
| **dbt manifest 规模膨胀**（新） | 启用 partial parsing；500+ model 时按 schema 拆分子项目；见 §11 性能预算 |
| **统一编辑器巨型组件**（新） | P2 按组件分层硬约束（shell < 300 行，inspector 各 < 500 行），CI 加文件行数检查 |

## 11. 性能预算（评审后新增）

> 初始 SLA，超出时触发性能优化工作。

| 指标 | 目标 | 触发优化的阈值 | 应对 |
|------|------|---------------|------|
| L1 任务级校验 | < 1s | > 2s | SQL 静态解析走缓存；避免每次 dbt parse |
| L2 图级校验（≤20 任务） | < 5s | > 10s | 异步化；拓扑序并行校验 |
| L2 图级校验（>20 任务） | 异步 < 30s | > 60s | 拆分校验；partial parsing |
| 流水线编译（≤20 任务） | < 5s | > 10s | 增量编译；按 task_key 哈希 diff |
| `dbt build --select tag:pipeline_<id>` 启动 | < 3s | > 8s | 启用 `--use-paths-cache` + partial parsing |
| Dagster launch → op 开始 | < 2s | > 5s | Dagster daemon 资源调整 |
| 单流水线最大任务数 | 50（软），100（硬） | > 100 | UI 拆分流水线提示；dbt 子项目拆分 |
| 并发运行流水线数 | 20（软），50（硬） | > 50 | Dagster queue；concurrency slot |
| dbt manifest 总模型数 | < 500 | > 500 | 按 schema 拆 dbt 子项目（`packages.yml` 本地 install） |
| task_run 状态刷新延迟 | < 5s | > 15s | Dagster run notifier 推送替代轮询 |

**监控**：所有指标打点 Prometheus，Grafana 看板 `onelake-pipeline-latency`。超出阈值告警 OPS 角色。

## 12. 日志与产物归档（评审后新增）

> **背景**：原方案提"运行日志"但未定义存储；§6.3 SPI 改造后 Java 侧不直接执行，必须设计产物归档路径。

### 12.1 存储布局（MinIO）

```
s3://onelake-artifacts/
└── pipelines/
    └── <tenant_id>/
        └── <run_id>/
            ├── manifest.json           # 本次运行的 runConfig 快照（含 task 列表、引擎、资源）
            ├── dbt/
            │   ├── run_results.json    # dbt 原生产物
            │   ├── logs/dbt.log        # dbt 完整日志
            │   └── target/manifest.json (可选, 大文件)
            └── spark/
                └── <task_key>/
                    ├── stdout.log
                    ├── stderr.log
                    └── eventlog        # Spark event log (JSON)
```

### 12.2 上传时机

- `run_dbt_subgraph_op` 完成 → 上传 `dbt/run_results.json` + `dbt/logs/dbt.log`，返回 `artifact_path` 写入 `task_run`（首个 dbt 任务的 artifact_path 代表整个子图）。
- `run_spark_task_op` 每任务完成 → 上传 `spark/<task_key>/`，更新对应 `task_run.artifact_path`。

### 12.3 Java 侧读取

- `OrchestrationService.refreshRunStatus` 解析 `run_results.json` → 节点级 `task_run.status / rows_written / scan_bytes`。
- 运行视图前端调 `/api/v1/orchestration/runs/{runId}/tasks/{taskKey}/logs` → 控制面从 MinIO stream。

### 12.4 保留策略

- 成功运行：保留 7 天。
- 失败运行：保留 30 天。
- 发布过的流水线最近一次成功运行：永久保留（用于回归对照）。
- 通过 `@Scheduled` 定期清理；超 10GB 触发告警。

### 12.5 租户隔离

- MinIO 路径含 `tenant_id`；通过 STS token 限制访问范围。
- 跨租户读取必须走控制面 API，不直接给前端 MinIO presigned URL。
