# 湖仓与建模模块 V2 · M1 详细任务拆解、迁移脚本与提示词

> 上游文档：`docs/湖仓与建模模块V2升级计划.md`（M1 = 湖仓现代化 A1-A4 + 维护自动化 B1-B5 + Dimension 骨架 G1）
>
> 前置：编排 V2 M4 已完成或并行；catalog schema Flyway 迁移头为 **V8**；modeling schema 迁移头为 **V8**。M1 迁移必须从 **V9+** 续接。
>
> 口径：基于当前代码事实（2026-07-14）。对齐现状：`CatalogTableService.createTable` 直发 Trino `CREATE TABLE`，无 catalog 后端抽象；`CatalogMaintenanceService` 仅支持手动触发 OPTIMIZE/EXPIRE_SNAPSHOTS/REMOVE_ORPHAN_FILES；`TableCreateRequest.ttlDays` 已存在但仅写 tags 字符串无执行器；`Dimension` 实体存在但无 Repository/Service/Controller。
>
> 使用方式：每步含「当前状态 / 目标 / 涉及文件 / 实现要点 / 验收标准 / Agentic 开发提示词 / 用户模拟测试提示词」。

## 0. M1 范围与执行顺序

```text
Epic 1 湖仓基础设施现代化（A1-A4）
  1.1 Catalog 后端抽象（IcebergCatalogBackend 接口 + HMS 实现）
  1.2 多 catalog 注册表（catalog.iceberg_catalog + API）
  1.3 Iceberg REST Catalog 引入（Trino 配置 + 双 catalog 并行）
  1.4 前端多 catalog 管理入口

Epic 2 维护自动化与生命周期（B1-B5）
  2.1 维护策略注册表（catalog.maintenance_policy）
  2.2 维护运行历史（catalog.maintenance_run）
  2.3 调度化维护（Catalog 域独立调度器）
  2.4 TTL 执行器（消费 TableCreateRequest.ttlDays）
  2.5 快照保留策略 + Sort Order（Z-Order 延后）

Epic 3 Dimension 骨架补齐（G1，为 M2 SCD/层级打底）
  3.1 Dimension Repository/Service/Controller
  3.2 前端维度管理入口（最小可用）
```

依赖关系：`1.1 → 1.2 → 1.3 → 1.4`；`2.1 → 2.2 → 2.3 → 2.4/2.5`；`3.1 → 3.2`。Epic 1/2/3 之间无强依赖，可并行；建议顺序：1.1 → 2.1 → 3.1（数据模型一次迁移），再 1.2/2.2/3.2，再 1.3/2.3，最后 2.4/2.5/1.4。

> **架构边界（与 V2 计划一致）**：
> - 不破坏 Spark-only 主线（V110 决策）；湖仓写闭环（D 支柱）留给 M2。
> - 不引入 dbt-on-Trino 回潮；dbt 产物继续作为「文档化 SQL」存在。
> - 维护调度属于 Catalog 域：由 `module-catalog` 内的 `CatalogMaintenanceScheduler` 负责，只复用通用 CRON/重试思路，不依赖编排域调度服务。
> - 复用算子市场、Outbox、TenantContext、AuditLogger 等既有基础设施。

## 1. 迁移脚本草案（合并预览）

> 落位：`onelake-app/bootstrap/src/main/resources/db/migration/`，catalog 续接 V8，modeling 续接 V8。全部 `ADD COLUMN ... DEFAULT` / `CREATE TABLE IF NOT EXISTS`，兼容存量数据。

### catalog/V9__iceberg_catalog_backend.sql

```sql
-- M1-A：Catalog 后端抽象与多 catalog 管理
CREATE TABLE IF NOT EXISTS catalog.iceberg_catalog (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL,
  code        VARCHAR(64) NOT NULL,
  name        VARCHAR(128) NOT NULL,
  backend_type VARCHAR(16) NOT NULL,                -- HMS|REST|JDBC|POLARIS
  config      JSONB NOT NULL,                       -- 凭证引用 security.secret，不含明文
  is_default  BOOLEAN NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, code)
);
CREATE UNIQUE INDEX IF NOT EXISTS uq_iceberg_catalog_one_default
  ON catalog.iceberg_catalog (tenant_id)
  WHERE is_default = true;

-- 默认 HMS catalog seed（每个租户一条；本地默认 iceberg）
INSERT INTO catalog.iceberg_catalog (tenant_id, code, name, backend_type, config, is_default)
SELECT t.id, 'default', 'Default HMS Iceberg Catalog', 'HMS',
       jsonb_build_object('trino_catalog', 'iceberg', 'hive_metastore_uri', 'thrift://hive-metastore:9083'),
       true
FROM common.tenant t
WHERE NOT EXISTS (
  SELECT 1 FROM catalog.iceberg_catalog ic WHERE ic.tenant_id = t.id
);

-- 资产关联到具体 catalog，并补齐逻辑归档所需的生命周期状态
ALTER TABLE catalog.asset
  ADD COLUMN IF NOT EXISTS catalog_code VARCHAR(64) NOT NULL DEFAULT 'default',
  ADD COLUMN IF NOT EXISTS status VARCHAR(16) NOT NULL DEFAULT 'ACTIVE'; -- ACTIVE|ARCHIVED
CREATE INDEX IF NOT EXISTS idx_asset_tenant_active
  ON catalog.asset (tenant_id, id)
  WHERE status = 'ACTIVE';
```

### catalog/V10__maintenance_policy.sql

```sql
-- M1-B1/B2：维护策略 + 运行历史
CREATE TABLE IF NOT EXISTS catalog.maintenance_policy (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL,
  asset_id    UUID NOT NULL REFERENCES catalog.asset(id) ON DELETE CASCADE,
  small_file_threshold_bytes BIGINT DEFAULT 134217728,
  small_file_risk_count      INT DEFAULT 10,
  min_snapshots_to_keep      INT DEFAULT 5,
  max_snapshot_age_hours     INT DEFAULT 168,
  ttl_days                   INT,                   -- NULL=永不归档
  optimize_strategy          VARCHAR(16) DEFAULT 'BIN_PACK', -- BIN_PACK|SORT；Z_ORDER 需 Spark executor，M1 不开放
  sort_columns               JSONB DEFAULT '[]',
  schedule_cron              VARCHAR(64),           -- NULL=仅手动
  enabled                    BOOLEAN NOT NULL DEFAULT true,
  created_at                 TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at                 TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (asset_id)
);

CREATE TABLE IF NOT EXISTS catalog.maintenance_run (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL,
  asset_id    UUID NOT NULL,
  operation   VARCHAR(32) NOT NULL,                 -- OPTIMIZE|EXPIRE_SNAPSHOTS|REMOVE_ORPHAN_FILES|VACUUM|ARCHIVE
  status      VARCHAR(16) NOT NULL,                 -- QUEUED|RUNNING|SUCCEEDED|FAILED|SKIPPED
  triggered_by VARCHAR(32) NOT NULL,                -- MANUAL|SCHEDULED|AUTO
  before_state JSONB,
  after_state  JSONB,
  duration_ms  BIGINT,
  error        TEXT,
  started_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  finished_at  TIMESTAMPTZ
);
CREATE INDEX IF NOT EXISTS idx_maint_run_asset ON catalog.maintenance_run (asset_id, started_at DESC);
CREATE INDEX IF NOT EXISTS idx_maint_run_status ON catalog.maintenance_run (tenant_id, status, started_at DESC);

-- Catalog 域独立调度锁，与流水线调度锁隔离
CREATE TABLE IF NOT EXISTS catalog.maintenance_scheduler_lock (
  lock_key     VARCHAR(64) PRIMARY KEY,
  holder       VARCHAR(128) NOT NULL,
  locked_until TIMESTAMPTZ NOT NULL,
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 表生命周期事件（B3 TTL 触发归档/删除）
CREATE TABLE IF NOT EXISTS catalog.lifecycle_event (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL,
  asset_id    UUID NOT NULL,
  event_type  VARCHAR(32) NOT NULL,                 -- CREATED|WRITTEN|OPTIMIZED|ARCHIVED|DELETED|ROLLBACK|EXPIRING
  payload     JSONB,
  occurred_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_lifecycle_asset ON catalog.lifecycle_event (asset_id, occurred_at DESC);
```

## Epic 1 · 湖仓基础设施现代化（A1-A4）

### 步骤 1.1 — Catalog 后端抽象（IcebergCatalogBackend 接口 + HMS 实现）

- **当前状态**：`CatalogTableService.executeCreateTable:183` 直接拼 `CREATE SCHEMA + CREATE TABLE` SQL 经 Trino JDBC 执行；`CatalogMaintenanceService.runMaintenance:63` 直接 `ALTER TABLE ... EXECUTE`；`CatalogSchemaChangeService.executeApproved:43` 直接 ALTER。所有路径耦合 HMS + Trino，无法切换 REST/JDBC/Polaris 后端。
- **目标**：抽出 `IcebergCatalogBackend` SPI 接口，HMS 作为首个实现，未来可加 REST/JDBC/Polaris；现有所有 DDL/DQL/Maintenance 调用经接口而非直发 SQL。
- **涉及文件**：
  - 新增 `module-catalog/src/main/java/com/onelake/catalog/service/catalogbackend/IcebergCatalogBackend.java`（接口）
  - 新增 `HmsCatalogBackend.java`（实现，封装现有 Trino SQL 逻辑）
  - 修改 `CatalogTableService.java`、`CatalogMaintenanceService.java`、`CatalogSchemaChangeService.java`（注入 backend，替换直发 SQL）
  - 新增 `module-catalog/src/main/java/com/onelake/catalog/config/CatalogBackendConfig.java`（按 catalog.iceberg_catalog 表配置实例化）
- **实现要点**：
  - 接口方法：`createTable/createSchema/dropTable/listTables/getTable/runMaintenance/alterTable/executeQuery/expireSnapshots/rewriteManifests` 等。
  - 接口实现持有 `CatalogConfig`（含 trino_catalog_name、warehouse、可选 REST endpoint）。
  - `CatalogBackendFactory` 按 `tenant_id + catalog_code` 实例化 backend，缓存（同 tenant 同 catalog 共享）。
  - 不引入新依赖；HMS 实现继续走 Trino JDBC。
  - 保持现有错误码（50061/50062/50063）与异常类型不变。
- **验收标准**：现有建表/维护/Schema 变更/SQL 工作台行为完全不变；`mvn -q -pl module-catalog -am test` 通过；新增 `HmsCatalogBackendTest` 覆盖建表、维护、Schema 变更。

> **Agentic 开发提示词**
> ```

> 1. 新增 com.onelake.catalog.service.catalogbackend.IcebergCatalogBackend 接口，方法:createTable/createSchema/dropTable/listTables/getTable/runMaintenance/alterTable/executeQuery/expireSnapshots/rewriteManifests。接口持有 CatalogConfig 记录 trinoCatalog/warehouse/credentialsRef。

> 验证：mvn -q -pl module-catalog -am test 全绿；浏览器/真实 API 验证建表、OPTIMIZE、Schema 变更三类操作行为不变。

### 步骤 1.2 — 多 catalog 注册表与 API

- **当前状态**：无 catalog 管理实体；所有资产默认属于 HMS `iceberg` catalog。
- **目标**：暴露 `/api/v1/catalog/catalogs` CRUD 接口；资产关联到具体 `catalog_code`；新增 catalog 时校验后端连通性；无资产租户与迁移后新建租户也必须幂等获得 `default` catalog。
- **涉及文件**：
  - 新增 `IcebergCatalog` entity、`IcebergCatalogRepository`、`IcebergCatalogDTO`、`CatalogConfigRequest`
  - 新增 `CatalogBackendService`（CRUD + 连通性测试 + `ensureDefaultCatalog`）
  - 新增 `CatalogCatalogController`（命名注意不与 `CatalogController` 冲突，建议路径 `/api/v1/catalog/catalogs`）
  - 修改 `Asset.java` 增加 `catalogCode` 字段（迁移已加）
  - 修改 `CatalogService.list/getAsset` 返回 `catalogCode`
- **实现要点**：
  - 凭证不入 `config` 明文，引用 `security.secret`（复用 `module-security` 的 SecretService）；`config = {secret_ref: "...", trino_catalog: "...", warehouse: "..."}`。
  - 创建 catalog 时调用 `IcebergCatalogBackend.testConnection()` 校验。
  - 删除 catalog 前校验是否有资产引用；级联拒绝或先迁移资产。
  - `is_default = true` 由迁移中的部分唯一索引强制（per tenant 只能一个默认）；切换默认 Catalog 必须在同一 PostgreSQL 事务中先取消旧默认再设置新默认。
  - 迁移从权威租户表 `common.tenant` seed 存量租户；`CatalogBackendService.ensureDefaultCatalog(tenantId)` 以 `(tenant_id, code)` 幂等创建默认 HMS catalog，并在 catalog 列表查询与首次建表前调用，覆盖迁移后新增租户。并发冲突时重读已有记录，不生成第二个默认项。
- **验收标准**：CRUD 接口可用；新增 REST/JDBC catalog 时（步骤 1.3 后）可切换；多 catalog 切换查询不串租户；存量无资产租户和迁移后新建租户首次访问时都只有一条 `default` catalog。

### 步骤 1.3 — Iceberg REST Catalog 引入

- **当前状态**：`trino/catalog/iceberg.properties` 仅配置 `iceberg.catalog.type=hive_metastore`；无 REST Catalog 支持。
- **目标**：引入 Iceberg REST Catalog（本地用 `tabularkeycatalog` 或 `polaris` 简化版），Trino 增加 `iceberg_rest` catalog；Java 侧 `RestCatalogBackend` 实现；HMS 与 REST 双 catalog 并行。
- **涉及文件**：
  - `onelake-app/docker-compose.yml`（新增 `iceberg-rest` 服务，建议用 `tabulario/iceberg-rest` 镜像）
  - `onelake-app/trino/catalog/iceberg_rest.properties`（新增）
  - 新增 `module-catalog/src/main/java/com/onelake/catalog/service/catalogbackend/RestCatalogBackend.java`
  - `CatalogBackendFactory` 按 `backendType=REST` 实例化
- **实现要点**：
  - 本地默认 `iceberg-rest` 服务用 PostgreSQL 作为 catalog 元数据后端（复用 `hive-metastore-db` 或新增专用库）；`warehouse=s3://warehouse/rest/`。
  - Trino `iceberg_rest.properties`：`connector.name=iceberg`、`iceberg.catalog.type=rest`、`uri=http://iceberg-rest:8181`、`warehouse=s3://warehouse/rest/`、S3 配置同 `iceberg.properties`。
  - `RestCatalogBackend` 与 HMS 共享接口；REST 不支持的 procedure（如某些 Iceberg v2 操作）需要 fallback 提示。
  - 不强制迁移现有 HMS 表；REST catalog 作为新 catalog 选项，新表可选 REST。
- **验收标准**：`make up` 启动 iceberg-rest 成功；Trino `SHOW CATALOGS` 包含 `iceberg` 与 `iceberg_rest`；新建 REST catalog 表可在 Trino/Spark 查询；连通性测试 API 通过。

> **Agentic 开发提示词**
> ```

> 验证：make up 启动 iceberg-rest healthy；Trino SHOW CATALOGS 含 iceberg_rest；通过 API 创建 REST catalog + testConnection + 建表 + Trino 查询成功。

### 步骤 2.5 — 快照保留策略 + Sort Order（Z-Order 延后）

- **当前状态**：`EXPIRE_SNAPSHOTS` 使用固定 `retention_threshold='7d'`（写死）；Trino `OPTIMIZE` 只提供文件合并参数，当前路径没有 Z-Order executor。
- **目标**：在 Trino 服务端能力支持时，用 `retention_threshold + retain_last` 同时落实 max_snapshot_age_hours/min_snapshots_to_keep；SORT 通过 Iceberg `sorted_by` 表属性管理新写入顺序。Z-Order 和指定快照保护不在 M1 Trino 路径中伪装实现。
- **涉及文件**：
  - 修改 `CatalogMaintenanceService.runMaintenance`：读策略，按经能力探测的 Trino 参数构造 SQL
  - 新增 `TrinoIcebergCapabilityService`：启动或首次运行时探测 `expire_snapshots.retain_last`，缓存服务端能力而不依赖 JDBC 驱动版本猜测
  - M2 Branch/Tag 完成后，需要永久保护的快照用 Iceberg Tag 承载；M1 不增加无法影响 expire 行为的本地 CANARY_TAGGED 旁路。
- **实现要点**：
  - 服务端支持时执行 `ALTER TABLE <fqn> EXECUTE expire_snapshots(retention_threshold => '<hours>h', retain_last => <min>)`。若 `retain_last` 不可用且 min_snapshots_to_keep > 1，本次维护必须写 `SKIPPED/UNSUPPORTED`并保留所有快照，禁止降级成仅 retention 后继续删除。
  - SORT 先校验 sort_columns，再用 `ALTER TABLE <fqn> SET PROPERTIES sorted_by = ARRAY[...]` 更新写入排序属性；Trino `OPTIMIZE` 只用其支持的 `file_size_threshold/WHERE` 参数做小文件合并。
  - `Z_ORDER` 需要 Spark Iceberg `rewrite_data_files` 的 sort/zorder strategy，在对应 executor 和运行契约落地前 API 返回 UNSUPPORTED，不回退成 SORT/BIN_PACK。
- **验收标准**：retain_last 支持时年龄与最小数量策略同时生效；不支持时无快照被误删；SORT 属性生效且 OPTIMIZE 只使用已支持参数；Z_ORDER 被明确拒绝直到 Spark executor 上线。

> **Agentic 开发提示词**
> ```

> 2. CatalogMaintenanceService.runMaintenance 读 maintenance_policy；EXPIRE_SNAPSHOTS 在能力就绪时执行 ALTER TABLE <fqn> EXECUTE expire_snapshots(retention_threshold => '<max_snapshot_age_hours>h', retain_last => <min_snapshots_to_keep>)。
> 3. retain_last 不可用且 min_snapshots_to_keep > 1 时写 maintenance_run.status=SKIPPED/error=UNSUPPORTED_RETAIN_LAST，不执行删除；禁止用仅 retention 的降级破坏策略。
> 4. SORT 用 ALTER TABLE <fqn> SET PROPERTIES sorted_by = ARRAY[...] 配置新写入顺序，再仅用 Trino 支持的 optimize(file_size_threshold)/WHERE 参数合并小文件。

## 3. M1 风险与回退

- **R1｜IcebergCatalogBackend 抽象引入回归**：HMS 实现必须保持现有错误码与异常类型不变；任何行为差异都会破坏 catalog/integration/security 模块联动。回退方案：保留旧直发 SQL 路径作为 feature flag，必要时切回。
- **R2｜REST Catalog 部署失败**：本地 `tabulario/iceberg-rest` 镜像若不可用，可暂时跳过步骤 1.3，只完成 1.1/1.2 的抽象与注册表，REST 推到 M1 末或 M2 初。
- **R3｜维护调度冲突**：多副本若无独立锁可能重复维护。隔离方式：Catalog 域使用独立 `maintenance_scheduler_lock` 表及不同 lock key，不共享、不影响流水线调度锁。
- **R4｜TTL 误删数据**：默认逻辑归档（ARCHIVED 状态，保留原表/FQN），不真删；物理冷存储必须经复制/重写校验后原子切换，硬删除模式必须显式环境变量开启。
- **R5｜Dimension 表已有种子数据**：modeling/V1 建 dimension 表时可能有 seed；激活前确认表为空或种子数据可保留。
