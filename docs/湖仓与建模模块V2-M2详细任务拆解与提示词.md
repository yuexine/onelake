## Epic 4 · 度量语义层引擎（F1-F7）【M2 核心】

### 步骤 4.1 — MetricFlow 兼容 schema 扩展

- **当前状态**：`Metric` 实体只有 `code/name/metricType/caliberSql/dbtModel/version/domainId`；无 entity 关联、无 measure 表达式、无 dimensions、无 time_dim；M1 未触动 Metric。
- **目标**：扩展 Metric 为 MetricFlow 兼容（semantic_entity_id + entity_fqn 快照 + measure_expr + dimensions + time_dim + status）；保留 `caliberSql` 作为旧字段（向后兼容），新增字段为权威。
- **涉及文件**：
  - 修改 `module-modeling/.../domain/entity/Metric.java`（新增字段对齐迁移 V10）
  - 修改 `MetricRepository`、`ModelingService.createMetric/getMetric/listMetricsByDomain`
  - 修改 `MetricDTO`、`MetricRequest`
  - 修改 `ModelingController` 现有 `/metrics` 端点接收新字段
- **实现要点**：
  - measure_type 枚举强校验（SUM/COUNT/COUNT_DISTINCT/AVG/MIN/MAX/MEDIAN）。
  - measure_expr 必须是合法 Spark/Trino 聚合表达式（如 `sum(amount)`、`count(distinct user_id)`）；服务端用 JSqlParser 校验。
  - semantic_entity_id 必须属于当前租户；entity_fqn 由 SemanticEntity.sourceFqn 写入快照，且目标必须存在于 catalog.asset（layer=DWD/DWS）。
  - dimensions 数组每项是 `{column, type, description}`；可选列存在性校验。
- **验收标准**：API 接收新字段；旧 caliberSql 字段保留兼容；现有 metric 数据不受影响。

> **Agentic 开发提示词**
> ```
> 目标：扩展 Metric 为 MetricFlow 兼容。约束：
> 1. Metric.java 增加字段：semanticEntityId/entityFqn/measureExpr/measureType/dimensions(jsonb)/defaultFilters(jsonb)/timeDim/timeGrain/semanticVersion/status/publishedAt/ownerId/ownerName/description/createdAt/updatedAt。
> 2. ModelingService.createMetric/getMetric 强校验：measure_type 枚举；measure_expr 用 JSqlParser 验证是聚合表达式；semantic_entity_id 属于当前租户，entity_fqn 从 SemanticEntity.sourceFqn 同步且 catalog.asset layer ∈ {DWD, DWS}；dimensions 列存在性校验。
> 3. MetricDTO/MetricRequest 同步对齐；ModelingController /metrics 端点接收新字段；旧 caliberSql 保留为兼容字段，不强制。
> 4. status 默认 DRAFT；publish 时校验所有必填字段（semanticEntityId/entityFqn/measureExpr/measureType/timeDim）。
> 5. 事件：DomainEvents 增加 MODELING_METRIC_CREATED/UPDATED/PUBLISHED/DEPRECATED。
> 验证：mvn -q -pl module-modeling -am test；真实 API 创建一个度量（sum(amount) on dwd.dwd_trade_order_df），校验通过。
> ```

### 步骤 4.2 — SemanticEntity/Measure/Dimension 模型 + CRUD

- **目标**：在 Metric 上层抽象 SemanticEntity（事实表注册）；Measure 复用 Metric 表；Dimension 复用 modeling.dimension（M1 已建）。
- **涉及文件**：
  - 新增 `SemanticEntity` entity、`SemanticEntityRepository`、`SemanticEntityService`
  - 新增 `SemanticEntityController`（`/api/v1/modeling/semantic-entities`）
  - 复用 `Metric`（作为 Measure）、`Dimension`（作为 Dimension）
- **实现要点**：
  - SemanticEntity 字段：tenantId/code/name/sourceFqn（DWD/DWS 表 FQN）/columns(jsonb)/primaryKey/timeColumn/description/status。
  - SemanticEntity 自动从 catalog.asset 推导 columns（refresh 接口）；columns 来自 `asset.columns`。
  - Measure（Metric）创建时必须关联 SemanticEntity。
  - 不重写 Metric，只是给它「挂在 Entity 上」的语义。
- **验收标准**：SemanticEntity CRUD 可用；Measure 可关联 Entity；Dimension 可关联 Entity（作为可用维度）。

> **Agentic 开发提示词**

### 步骤 4.3 — SemanticQueryCompiler（直接生成 Trino/Spark SQL）

> **关键约束**：**不走 dbt 中间产物**，避免重蹈 D1（DWD 编译产物与运行时脱钩）覆辙。

- **当前状态**：无 SQL 生成器；Metric.caliberSql 是裸 SQL 字符串。
- **目标**：`SemanticQueryCompiler` 接收 `{metric, dimensions, filters, time_range, time_grain, top_n, compare_to}`，直接生成可执行 Trino/Spark SQL。
- **涉及文件**：
  - 新增 `module-modeling/.../service/semantic/SemanticQueryCompiler.java`
  - 新增 `module-modeling/.../dto/semantic/SemanticQueryRequest.java`、`SemanticQueryResult.java`
  - 新增 `module-modeling/.../service/semantic/MeasureExpressionParser.java`（解析 measure_expr 为 AST 节点）
- **实现要点**：
  - 输入：`metric_id` + `group_by_dimensions[]` + `filters[]` + `time_range` + `time_grain` + `top_n` + `compare_to`（同环比）。
  - 输出：Spark SQL 字符串（用于物化）+ Trino SQL 字符串（用于查询）。
  - SQL 结构：`SELECT dims, measure FROM entity WHERE filters AND time_range GROUP BY dims ORDER BY ...`。
  - 同环比：生成 JOIN 当前周期与上一周期的 SQL（基于 time_dim + time_grain）。
  - **不调用 dbt**，不走 Jinja 模板，直接 String 拼接 + JSqlParser 校验生成结果。
- **验收标准**：20 条典型查询（聚合、过滤、同环比、TopN、时间粒度切换）生成的 SQL 在 Trino 可执行且语义正确。

> **Agentic 开发提示词**
> ```
> 目标：SemanticQueryCompiler 直接生成可执行 SQL（不走 dbt）。约束：
> 1. 新增 com.onelake.modeling.service.semantic.SemanticQueryCompiler；方法 compile(SemanticQueryRequest) 返回 CompiledQuery{trinoSql, sparkSql, bindings}。
> 2. SemanticQueryRequest 字段：metricId/groupByDimensions[]/filters[]/timeRange{start,end}/timeGrain/day|week|month/topN?/compareTo?{previous_period_type}。
> 3. SQL 结构（Trino）：SELECT <dims>, <measure_expr> AS metric_value FROM <entity_fqn> WHERE <filters> AND <time_dim> BETWEEN ? AND ? GROUP BY <dims> [ORDER BY metric_value DESC LIMIT <topN>]。
> 4. 同环比：生成两个 CTE（current_period / previous_period），按 dims JOIN；输出 metric_value 与 prev_metric_value 与 growth_rate。
> 5. time_grain 转换：date_trunc('day', time_dim)；week 用 date_trunc('week', ...)；month 用 date_trunc('month', ...)。
> 6. MeasureExpressionParser 用 JSqlParser 解析 measure_expr，校验是合法聚合表达式（含 sum/count/avg/min/max/count_distinct）。
> 7. 用 ReadOnlySqlValidator（已有）校验生成的 SQL 是只读 SELECT，避免误生成写操作。
> 8. 不引入 dbt 依赖；不生成 Jinja 模板；不写文件。
> 验证：mvn -q -pl module-modeling -am test；20 条典型查询生成 SQL 在本地 Trino 执行通过。
> ```

> **用户模拟测试提示词**

### 步骤 4.4 — 指标查询 API（Headless BI）

- **当前状态**：无指标查询 API；Metric 不可被查询。
- **目标**：`POST /api/v1/modeling/metrics/query` 接受 `SemanticQueryRequest`，编译 SQL 后通过 Trino 执行，返回结构化结果；API shape 兼容 Cube.dev。
- **涉及文件**：
  - 新增 `module-modeling/.../api/SemanticQueryController.java`
  - 新增 `SemanticQueryService`（调 SemanticQueryCompiler + 复用 SqlAssetSecurityService 脱敏 + Trino 执行）
  - 修改 `module-catalog/.../config/TrinoConnectionFactory` 暴露给 module-modeling（或新增独立 TrinoConnection）
- **实现要点**：
  - 调编译器生成 SQL → SqlAssetSecurityService 校验资产授权 → SecurityService.maskRowsWithNotices 脱敏 → Trino 执行 → 返回 `{columns, rows, maskedColumns, securityNotices, meta: {durationMs, scanBytes}}`。
  - Cube.dev 兼容：`POST /api/v1/modeling/metrics/query` 接收 `{query: {...}, queryType: 'multi'}`，返回 `{data: [...], annotation: {...}}`。
  - 大查询拦截：复用现有 SqlWorkbenchService 的扫描阈值与 confirmLargeQuery 机制。
  - 限流：per-tenant QPS（基于 QuotaUsage 表）。
- **验收标准**：API 可用；脱敏生效；与 Superset/Cube 集成通过。

> **Agentic 开发提示词**
> ```
> 目标：指标查询 API（Headless BI）。约束：
> 1. SemanticQueryController POST /api/v1/modeling/metrics/query；接收 SemanticQueryRequest；返回 {columns, rows, maskedColumns, securityNotices, meta}。
> 2. SemanticQueryService 流程：调 SemanticQueryCompiler 编译 → 用 module-common ReadOnlySqlValidator 二次校验 → SqlAssetSecurityService.requireQueryAccess 校验 entity_fqn 资产授权 → Trino 执行 → SecurityService.maskRowsWithNotices 脱敏 → 返回。

### 步骤 7.1 — 采集侧自动 captureSnapshot

- **当前状态**：`SourceSchemaSnapshotServiceImpl.publishDriftIfChanged` 存在但**不自动 capture**；只暴露 `POST /integration/snapshot/...` 等外部调用。

- **实现要点**：
  - 流程：reconcile 成功 → captureAndCompare → 比对上次 snapshot → 有差异时发 `integration.schema.drift` 事件。

  - 事件 payload：`{tenantId, datasourceId, tableFqn, changeType, oldColumns, newColumns, breakingChange: bool}`。

> **Agentic 开发提示词**
> ```

> 1. SyncTaskServiceImpl.refreshRunSnapshot 在 SUCCEEDED 终态后调 sourceSchemaSnapshotService.captureAndCompare(datasourceId, sourceTable)。
> 2. SourceSchemaSnapshotService.captureAndCompare：用 DatabaseDiscoveryClient 查当前 columns；与最新 snapshot 比对；差异时发 INTEGRATION_SCHEMA_DRIFT 事件。

### 步骤 7.2 — DWD 自动 revalidate/recompile

- **当前状态**：`DwdSchemaDriftEventHandler` 监听 drift 后只标记 `modeling.data_model.status=NEEDS_REVIEW`；**不自动 revalidate/recompile**。
- **目标**：drift 事件触发后自动调 `DwdModelService.validate(id)`；NON_BREAKING 自动 recompile；BREAKING 标记 NEEDS_REVIEW、通知 owner，并通过跨域事件立即冻结关联 pipeline 调度。
- **涉及文件**：
  - 修改 `DwdSchemaDriftEventHandler.handleEvent`
  - 调用 `DwdModelService.validate(id)` 与可选 `compileArtifacts(id)`
  - Modeling 通过 Outbox 发 `modeling.model.drift_detected`、`modeling.pipeline.freeze.requested` 和 `modeling.pipeline.freeze.resolved`
  - Orchestration 新增冻结/解冻事件消费者与 `PipelineFreezeOverride` entity/repository，并修改 `PipelineSchedulerService` 检查 ACTIVE 覆盖
- **实现要点**：
  - NON_BREAKING（新增字段）：自动 recompile，原字段映射不变；新字段默认不纳入 mapping，除非显式配置。
  - BREAKING（删字段/类型变）：标记 NEEDS_REVIEW + 通知 owner + 发布 `modeling.pipeline.freeze.requested`；由 orchestration 按 `(dagId, sourceType=MODEL_SCHEMA_DRIFT, sourceRef=modelId)` upsert ACTIVE 覆盖，不改写 `dag.schedule_mode`。

  - recompile 失败时不阻塞采集，只发告警。
- **验收标准**：NON_BREAKING 自动 recompile；BREAKING 通知 + 立即暂停；接受/忽略后仅解除对应 drift 覆盖，手工冻结与其他 ACTIVE 原因仍生效。

> **Agentic 开发提示词**
> ```
> 目标：DWD 自动响应 drift。约束：
> 1. DwdSchemaDriftEventHandler.handleEvent 解析 payload {tenantId, tableFqn, changeType, breakingChange}；查所有 sourceFqn = tableFqn 的 DWD 模型。
> 2. NON_BREAKING：自动调 DwdModelService.compileArtifacts(id)（recompile）；发通知给 owner（level=INFO，"字段已自动同步"）。

> 6. recompile 失败：不抛异常，记日志 + 发告警。事件 payload 保留兼容。
> 7. 单测覆盖 requested 幂等、多 model 覆盖并存、accept/dismiss 定向 resolve、手工 FROZEN 不被解除。
> 验证：mvn -q -pl module-modeling,module-orchestration -am test；模拟源表 BREAKING 变更，确认 DWD 进入 NEEDS_REVIEW，已发布 pipeline 无需重新发布即停止后续 cron；接受变更后对应覆盖变为 RESOLVED 并恢复触发。

## 3. M2 风险与回退

- **R1｜度量编译器 SQL 正确性**：20 条查询覆盖不全可能漏 case；建议先支持核心 5 类（聚合/过滤/时间粒度/同环比/TopN），其他作为迭代。
- **R2｜Trino 写权限误开放**：DEV 模式只能在显式本地 profile 由服务端启用，客户端不得覆盖；接口始终认证 + DE 角色 + 租户隔离 + 白名单 + 审计，非本地 profile 配置 DEV 启动失败。
- **R3｜Schema Drift 自动 recompile 失败率**：NON_BREAKING 也可能因新字段类型冲突 recompile 失败；失败时降级为 BREAKING 处理。
- **R4｜SCD2 维度表膨胀**：高频变更维度导致表快速膨胀；配套 expire_snapshots 与归档策略。
- **R5｜Iceberg Branch/Tag 版本依赖**：Trino/Iceberg 1.5+ 才完整支持；本地若版本过低需升级。
