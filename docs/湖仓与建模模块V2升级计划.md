**核心启示**（融入 V2 设计）：

1. **REST Catalog 是未来 3 年护城河**：Polaris 已进入 Apache 孵化，HMS 长期会过渡到 REST Spec，跨引擎互操作是关键。
2. **语义层是 NL2SQL 的地基**：Snowflake 实测显示「Arctic-Text2SQL 单轮 ~45%，配合语义层调优可达 90%+」——OneLake 的字典/术语表/血缘是天然优势。
3. **Iceberg procedures 工程化最快见效**：直接影响查询性能与存储成本。
4. **Stream-Table Duality（Paimon）补齐实时**：对抗 Hudi 在高频 upsert 场景的强势地位。
5. **Data Contract（ODCS）成为跨域契约载体**：把 ODS/DWD/DWS 链路产物发布成契约化数据产品。
6. **多模态湖仓（Lance + Iceberg）是下一前沿**：图像/音频/向量入湖。

## 4. V2 功能升级方案

按能力支柱拆分，每项标注优先级（P0/P1/P2）与所属阶段。

### 4.1 支柱 A：湖仓基础设施现代化（H1，P0）

- **A1｜Iceberg REST Catalog 引入**：新增 `onelake-app/polaris/` 或集成 `tabular` REST server；Trino catalog 增加 `iceberg.rest` 配置；HMS 作为 fallback 后端保留兼容；提供 `catalog.catalog_backend = HMS|REST|JDBC` 配置。
- **A2｜Catalog 后端抽象**：`CatalogTableService` 抽出 `IcebergCatalogBackend` 接口，HMS/REST 实现可切换；建表/删表/列表通过接口而非直发 SQL。
- **A3｜Polaris 兼容**：REST Catalog 实现遵循 [Iceberg REST Spec](https://iceberg.apache.org/docs/latest/api/rest-catalog/)，长期目标支持 Polaris 接入（同一表被 Spark/Trino/Flink/外部引擎访问）。
- **A4｜Catalog 联邦底座**：暴露 `/catalogs` 多 catalog 管理接口，每个 catalog 独立配置（type/warehouse/credentials），为支柱 E 做铺垫。

### 4.2 支柱 B：维护自动化与表生命周期（H2，P0）

- **B1｜维护策略注册表**：`catalog.maintenance_policy`（per-table 配置：`small_file_threshold`、`min_snapshots_to_keep`、`max_snapshot_age_hours`、`ttl_days`、`optimize_strategy=BIN_PACK|SORT`、`sort_columns`）；`Z_ORDER` 只能在 Spark `rewrite_data_files` executor 上线后扩展，不经 Trino OPTIMIZE 伪实现。
- **B2｜调度化维护**：由 `module-catalog` 内的 `CatalogMaintenanceScheduler` 扫描到期策略，按 SLA/小文件阈值触发 `OPTIMIZE/EXPIRE_SNAPSHOTS/REMOVE_ORPHAN_FILES`；使用 Catalog 域独立分布式锁，写 `catalog.maintenance_run` 历史，不新增流水线 TaskType。
- **B3｜TTL 与生命周期**：表级 `lifecycle_policy`（`ARCHIVE/DELETE` after N days），过期前 7 天发 `catalog.lifecycle.expiring` 通知；归档可走对象存储 tier；**消费 `TableCreateRequest.ttlDays` 已有字段**。
- **B4｜快照保留策略**：在 Trino 能力支持时用 `retention_threshold + retain_last` 落实最近 N 个 + 最近 M 天；不支持 `retain_last` 时 fail closed。需要永久保护的快照在 M2 用 Iceberg Tag 承载，不建只存本地数据库的伪保护标记。
- **B5｜Z-Order/Sort Order**：M1 用 Iceberg `sorted_by` 表属性 + Trino BIN_PACK 支持 Sort Order；Z-Order 需单独的 Spark `rewrite_data_files` executor 才开放，后续可接入查询历史推荐 sort 列。

### 4.3 支柱 C：表版本管理与时间旅行（H3，P1）

- **C1｜Iceberg Branch/Tag**：`CatalogTableService` 增加 `createBranch/tag/mergeBranch/rollbackTo` 接口；底层使用与 Spark 数据面版本对齐的 Iceberg Java Catalog/ManageSnapshots API，不通过 Trino JDBC 执行 Spark DDL。
- **C2｜时间旅行 UI**：`TableDetail` 增加「快照历史」tab（**前端骨架已预留**，`TableDetail.tsx:905-916`），列出 `snapshot_id/parent_id/operation/timestamp/summary`；支持「查询此快照」「回滚到此快照」「创建分支」。
- **C3｜ML/实验工作流**：典型场景「在 dev 分支做训练数据生成→merge 到 main」，与编排 V2 联动（pipeline 输出可指定 branch）。
- **C4｜GDPR/审计快照**：`tag` 历史时点（如「2025-Q3 末」），用于合规审计。

### 4.4 支柱 D：湖仓写闭环（H4，P1）

- **D1｜SQL 工作台写模式**：在 ADR-001 能力矩阵下放开 `TRINO_SQL` 写：`CTAS`、`MERGE INTO`、`INSERT`、`UPDATE`、`DELETE`；新增 `WriteSqlValidator`（白名单写操作 + 资源/超时限制），与 `ReadOnlySqlValidator` 共存。
- **D2｜写权限分级**：复用 `security.access_grant.permissions` JSONB 的 `write=true`，不扩展仅有单值 `permission` 的 `acl_resource`；写接口始终需要认证、DE 角色、租户隔离与服务端白名单。DEV 只能在显式本地 profile 跳过 WRITE ACL，客户端不得控制模式/白名单；PROD 必须审批。
- **D3｜写操作审计与回滚**：已解析目标的所有写执行尝试记录到 `catalog.write_audit`；CTAS 用 `target_fqn` 先审计、资产注册后回填可空 `asset_id`。快照必须通过 committed_at + parent_id 与本次写唯一关联，不得用 `max(snapshot_id)` 猜测；只有 `CONFIRMED` 才可回滚。
- **D4｜CTAS 模板**：「查询结果保存为新表」一键入口，自动建 Iceberg 表 + 注册 Catalog 资产 + 发 `catalog.asset.created` 事件。

### 4.5 支柱 E：逻辑数据仓库与 Catalog 联邦（H5，P2）

- **E1｜Trino 联邦 catalog 管理**：在 `trino/catalog/` 下支持动态加载 `mysql/postgres/kafka/redis/elasticsearch/clickhouse` 等 connector；`CatalogSyncService` 自动发现外部表并登记为 `catalog.asset`（`source_type=FEDERATED`）。
- **E2｜跨源逻辑视图**：基于 Trino 联邦，建模可定义跨源逻辑模型（如 MySQL 维度表 + Iceberg 事实表 JOIN），编译为 Trino view。
- **E3｜外部表加速**：对热点外部表提供「Trino 物化视图」选项（基于 `REFRESH MATERIALIZED VIEW`），纳入维护调度。
- **E4｜Apache XTable 互操作**：若未来引入 Delta/Hudi，通过 XTable 翻译元数据到 Iceberg catalog，统一治理面。

### 4.6 支柱 F：度量语义层引擎（M1，P0）

> **关键约束**：直接生成可执行 Spark SQL / Trino SQL，**不走 dbt 中间产物**，避免 D1 重蹈。

- **F1｜MetricFlow 兼容语义模型**：重构 `Metric` 为 `SemanticEntity + Measure + Dimension`（对齐 dbt Semantic Layer）；YAML 定义 + DB 持久化双写。
- **F2｜度量口径统一**：`caliberSql` 拆为 `expression`（声明式，如 `sum(amount)`）+ `dimensions`（可用维度集）+ `filters`（默认过滤）+ `time_dim` + `time_grain`；多版本管理（继承术语表版本机制）。
- **F3｜SQL 自动生成器（`SemanticQueryCompiler`）**：把「指标 + 维度 + 过滤 + 时间窗口」直接编译为 Trino/Spark SQL（非 dbt）；支持 `top_n`、`compare_to`（同环比）、`time_grain=day/week/month`。
- **F4｜指标 API（Headless BI）**：`POST /api/v1/modeling/metrics/query` 接受 JSON 查询体，返回结构化结果；兼容 Cube.dev API shape；可被 analytics/Superset/外部 BI 消费。
- **F5｜指标血缘**：度量定义与底层 DWD/DWS 资产绑定，发 `modeling.metric.evaluated` 事件，Catalog 维护「指标→资产→字段」血缘链。
- **F6｜指标版本与发布**：度量变更走审定→发布→版本（对齐 Glossary 现有机制），避免口径漂移。
- **F7｜与算子市场协同**：新增 `semantic.compile` 算子注册到 `BuiltInOperatorCatalog`，复用编排 V2 G2 的「算子→Spark 代码生成」链路。

### 4.7 支柱 G：完整维度建模（M2，P0）

> **首刀**：Dimension 完全空壳，必须先补 Repository/Service/Controller 骨架。

- **G1｜Dimension 骨架补齐**：新增 `DimensionRepository/DimensionService/DimensionController`；CRUD + 列表 + 详情接口。
- **G2｜SCD2 支持**：`Dimension` 增加 `scd_type`（1/2/3）；SCD2 表自动增加 `valid_from/valid_to/is_current`；关闭旧版本与插入新版本必须由单个 Spark MERGE/Iceberg 快照原子提交并支持幂等重放；DWD 派生时自动 JOIN 当前有效版本。
- **G3｜维度层级**：`Dimension.hierarchy`（如 `省→市→区`），UI 树形展示；下钻/上卷 API 支持层级聚合。
- **G4｜事实-维度关联**：`modeling.fact_dimension_binding` 描述事实表外键与维度表主键的关联；`role_name` 支持同一 ROLE_PLAYING 维度以 order_date/ship_date 等多角色重复绑定，并自动生成星型拓扑图。
- **G5｜星型/雪花建模画布**：新建 `/modeling/workbench` 页面，可视化拖拽事实/维度表，自动生成 DWD/DWS 模型与外键约束。
- **G6｜一致性维度（Conformed Dimension）**：跨主题域共享维度（如 `dim_date`、`dim_user`），版本统一管理，避免口径分裂。
- **G7｜与算子市场协同**：新增 `dimension.scd2_join` / `dimension.hierarchy_rollup` 算子，复用编排 V2 G2 链路。

### 4.8 支柱 H：DWS/ADS 汇总层建模（M3，P1）

- **H1｜DWS 模型驱动**：DWS 模型 = 度量 + 维度 + 时间粒度，由支柱 F 度量语义层自动推导物化 SQL；生成 `dws_<domain>_<metric>_<grain>` 表。
- **H2｜ADS 应用层**：ADS = DWS + 业务加工（TOP N、同环比、目标对比），由编排 V2 触发 Spark pipeline 生成。
- **H3｜物化策略**：`EAGER`（每次上游更新触发）、`LAZY`（查询时计算）、`SCHEDULED`（按 SLA 调度）；与编排 V2 自动化触发联动。
- **H4｜预聚合缓存**：对高频查询的 DWS 表，可选 Cube.dev 风格的预聚合缓存层（Trino `REFRESH MATERIALIZED VIEW` 或独立加速引擎）。

### 4.9 支柱 I：数据契约与数据产品（M4，P1）

> **Schema Drift 闭环提前**：三段链路已部分存在，整合成本低。

- **I1｜Schema Registry**：所有 ODS/DWD/DWS/ADS 表的 schema 强契约；变更走 PR 式审批（已有 Schema 变更审批，扩展为契约版）。
- **I2｜Breaking Change 检测**：`SchemaDriftDetector` 对比新旧 schema，分类 `BREAKING/NON_BREAKING/COMPATIBLE`；Breaking 必须审批 + 下游通知。
- **I3｜Schema Drift 闭环收口**：采集侧自动 `captureSnapshot`（解决 `SourceSchemaSnapshotServiceImpl:85` 不自动 capture）；DWD 侧自动 revalidate/recompile；BREAKING 通过原因隔离的 `pipeline_freeze_override` 立即暂停调度，接受/忽略后定向解除对应覆盖，不改写或误解除手工 `schedule_mode=FROZEN`。
- **I4｜数据契约发布**：基于 [ODCS](https://github.com/bitolio/open-data-contract-standard) 标准，发布 `data_contract.json`（含 schema/SLA/owner/quality/price）；契约版本化。
- **I5｜数据产品目录**：契约化的资产上升为「数据产品」，可在 `/catalog/products` 浏览、订阅、申请；与 DaaS API 联动（数据产品可发布为 API）。
- **I6｜下游订阅与 SLA**：契约包含 SLA（freshness/quality/availability），违约触发 `catalog.contract.sla_breached` 事件。

### 4.10 支柱 J：AI 辅助建模与治理（M5，P2）

- **J1｜NL2Model**：自然语言描述 → DWD 模型草稿（字段映射、命名规范、密级推断）；复用 OneLake 字典/术语表作为知识库，比通用 NL2SQL 准确率高（参考 Snowflake Cortex 90%+ 数据）。
- **J2｜字段智能归类**：基于字段名 + 采样值 + 上下游，自动推荐业务术语/密级/SCD 类型；接入 Security PII 扫描结果。
- **J3｜Text2SQL Copilot（SQL 工作台）**：基于当前 Catalog schema + 字典 + 度量语义层，生成只读 SQL；用户可微调；记录到 `sql_query_history`。
- **J4｜命名规范检测**：基于 `modeling.data_standard`（**当前空壳，需先激活**），扫描 DWD 草稿字段命名违规；发警告。
- **J5｜AI 治理**：自动 PII 分类升级（从命名规则 → 采样值 + 模型）、敏感数据发现、policy-as-code。

### 4.11 支柱 K：多模湖仓与 AI 检索（M6，P2）

- **K1｜向量列支持**：建表支持 `VECTOR(FLOAT, dim)` 类型（Trino + Iceberg）；Puffin 文件存放向量索引。
- **K2｜向量索引**：基于 Lance 或 Iceberg Puffin stats，建立 ANN 索引（HNSW/IVF）；通过 SQL 函数 `vector_search(table, query_vec, k)` 查询。
- **K3｜RAG 入湖**：文档/半结构化数据通过 pipeline 切片 + embed 入湖；表结构 `id/chunk/content/embedding/metadata`。
- **K4｜AI 检索 API**：`/api/v1/lakehouse/vector-search` 接受查询向量 + 过滤条件，返回 top-k；与 Security 脱敏联动。
- **K5｜多模资产治理**：`catalog.asset.asset_type` 扩展 `VECTOR_INDEX/MODEL/DOCUMENT`；纳入统一权限/血缘。

### 4.12 支柱 L：湖仓深度观测（H6，P1）

- **L1｜快照时间线**：`TableDetail` 增加「快照历史」可视化（甘特图 + diff）。
- **L2｜表生命周期看板**：聚合「建表→写入→维护→归档→删除」全生命周期事件；可观测性指标（小文件趋势、压缩比、查询 P95）。
- **L3｜新鲜度深度**：现有 `freshnessStatus` 升级为「上游链路新鲜度」（多级上游任一滞后即报警）。
- **L4｜Cost Insights**：基于 Trino `EXPLAIN` + 历史扫描量，给出表/查询成本画像；推荐优化策略。
- **L5｜湖仓大盘**：`/lakehouse/overview` 总览存储/查询/维护/契约健康度。

  status VARCHAR(16) NOT NULL DEFAULT 'ACTIVE',      -- ACTIVE|RESOLVED
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  resolved_at TIMESTAMPTZ,
  UNIQUE (dag_id, source_type, source_ref)
);
CREATE INDEX idx_pipeline_freeze_active
  ON orchestration.pipeline_freeze_override (tenant_id, dag_id)
  WHERE status = 'ACTIVE';

## 8. 里程碑（建议 4 阶段，每阶段 6-8 周）

| 阶段 | 主题 | 关键交付 | 依赖 |
| --- | --- | --- | --- |
| **M1（基石）** | 湖仓现代化 + 维护自动化 | A1-A4（REST Catalog 抽象）、B1-B5（维护自动化 + TTL 执行器）、G1（Dimension 骨架补齐） | 编排 V2 M4 完成或并行 |
| **M2（度量层 + 版本管理 + 写闭环）** | 度量语义层 + 表版本管理 + 写闭环 + Schema Drift 闭环 | F1-F7（MetricFlow 兼容，直接生成可执行 SQL）、C1-C4（Branch/Tag/Time Travel）、D1-D4（写闭环）、I3（Schema Drift 闭环）、L1-L5（深度观测）、G2-G7（SCD/层级/星型） | M1 |
| **M3（契约与汇总层）** | 数据契约 + DWS/ADS + 联邦 | I1/I2/I4/I5/I6（契约/数据产品）、H1-H4（DWS/ADS）、E1-E4（Catalog 联邦/XTable） | M2 |
| **M4（智能化与多模）** | AI 辅助 + 向量检索 | J1-J5（NL2Model/Copilot）、K1-K5（向量列/Puffin/RAG） | M3 |

### 8.1 与编排 V2 的对齐

- **不破坏 Spark-only 主线**：湖仓写闭环（支柱 D）通过 ADR-001 能力矩阵管控；度量查询（支柱 F4）默认走 Trino 只读，物化走 Spark。
- **按域分离调度**：维护自动化（B2）由 Catalog 域独立调度并使用独立锁；流水线调度只处理 DAG 发布快照与 I6 契约触发，不跨域扫描 `catalog.maintenance_policy`。
- **复用算子市场**：度量编译器（F3）和维度建模（G2/G3）作为新算子（`semantic.compile`、`dimension.scd2_join`、`dimension.hierarchy_rollup`）注册到 `BuiltInOperatorCatalog`，复用编排 V2 G2 已落地的「算子→Spark 代码生成」链路。

### 8.2 关键架构决策（避免重蹈 D1 覆辙）

- **度量语义层直接生成可执行 SQL**：`SemanticQueryCompiler` 不走 dbt 中间产物，避免「文档化 SQL」陷阱。
- **现有 dbt 产物降级为文档导出**：DWD compile 链路保留 dbt Jinja 输出，但仅作为「文档/审计」用途，不作为运行依赖；运行入口仍是 Spark `transform.spark_sql` 节点。
- **TTL/生命周期作为执行器消费已有字段**：B3 直接消费 `TableCreateRequest.ttlDays`，不新建字段。

## 9. 风险与约束

- **R1｜REST Catalog 迁移影响面大**：HMS→REST 涉及所有引擎（Spark/Trino/Flink），必须保留 HMS 兼容后端，灰度迁移；建议先做双 catalog 并行。
- **R2｜度量语义层投入大**：完整 MetricFlow 实现是 6-8 人周工作量；建议先做最小集（3-4 种 measure type + 时间维度），后续迭代扩展。
- **R3｜Spark-only 边界**：度量查询若放开 Trino 直查，需在 ADR-001 矩阵中明确「Trino 查询不写表、不消耗 Spark 资源」；与编排 V2 决策对齐。
- **R4｜契约化发布阻力**：现有 ODS→DWD 链路非契约化，强制契约会阻塞现有迭代；建议新表强制、老表渐进迁移。
- **R5｜AI 辅助准确率**：NL2Model/NL2SQL 依赖字典/术语表质量；早期需要人工审核兜底，避免错误模型流入生产。
- **R6｜向量列存储成本**：Puffin 索引和向量列会显著增加存储；必须配套 TTL 与归档策略。
- **R7｜多 catalog 复杂度**：联邦 catalog 增加权限/脱敏复杂度；外部表必须有独立的 ACL 评估路径。
- **R8｜契约兼容性**：扩展 Outbox 事件 payload（新增 `catalog.contract.*`、`modeling.metric.*`、`catalog.lifecycle.*`、`catalog.maintenance.*`）需保持向后兼容（参考全局事件契约文档）。
- **R9｜Dimension/DataStandard 空壳激活风险**：M1/M2 激活空壳表时，需考虑存量数据（应为空）与未来扩展字段；建议直接按 V2 schema 建表，避免反复迁移。

## 10. 非目标（V2 暂不做）

- **流批一体实时数仓（Paimon/Flink）**——列入 V3；当前 Spark-only 主线 + Iceberg 已满足批场景。
- **多 OLAP 加速引擎（Hologres/ByteHouse/Dremio Reflections）**——列入 V3；当前 Trino 作为唯一查询引擎。
- **完整 Delta/Hudi 表格式支持**——XTable 互操作作为 V3 探索；V2 保持 Iceberg 单格式深化。
- **完整 Data Mesh 平台**——契约与数据产品（支柱 I）作为 V2 起点，跨域联邦治理列入 V3。
- **完整 AI Observability**——J5 仅做基础 PII/分类自动化，模型监控/可观测列入 V3。
- **Lakehouse + LLM 训练**——K 支柱仅做检索底座，模型训练/微调列入 V3。
- **多租户资源隔离增强**——V2 复用现有 `resource_group/compute_profile`，不重做资源调度器。
- **重写 DWD 编译为 dbt-on-Trino**——V110 已硬删该路径；V2 度量语义层绕开 dbt，DWD compile 链路保持现状（dbt 产物仅作文档）。

## 13. 总结

OneLake 的湖仓建模 V1 已完成「业务建模 → Spark 物化 → 元数据治理」纵向切片，V2 的核心命题是**横向能力补全**与**面向未来的现代化**：

1. **横向补全**：度量语义层（M1）、维度建模深化（M2）、DWS/ADS 汇总层（M3）、数据契约（M4）——把「单线 ODS→DWD」升级为「完整 Kimball 维度建模 + Headless BI 度量平台」。
2. **现代化**：Iceberg REST Catalog（H1）、维护自动化（H2）、表版本管理（H3）、写闭环（H4）——对齐 Polaris/Unity Catalog/Iceberg v4 路线。
3. **面向未来**：AI 辅助建模（J）、多模向量湖仓（K）——为 NL2SQL/RAG/AI 应用打底。

V2 与编排 V2 并行不冲突：**编排 V2 解决「怎么跑」，湖仓建模 V2 解决「跑什么、跑出来怎么用」**。两者通过算子市场、调度器、事件总线协同，最终把 OneLake 从「数据开发平台」升级为「数据开发 + 数据产品 + AI 检索」一体化平台。

V2 必须直面的架构债务（D1-D5）已显式纳入设计：度量语义层绕开 dbt 中间产物（避免 D1 重蹈）、Dimension 骨架首刀补齐（解决 D2）、TTL 直接消费已有字段（利用 D3）、Schema Drift 闭环收口（解决 D4）。
