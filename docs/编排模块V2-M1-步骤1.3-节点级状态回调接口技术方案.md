# M1 · 步骤 1.3 — 节点级状态回调接口技术方案

> 上游：`docs/编排模块V2-M1详细任务拆解与提示词.md` 步骤 1.3；依赖步骤 1.2（`docs/编排模块V2-M1-步骤1.2-Dagster动态图执行技术方案.md`）在执行器内埋的 `_callback(...)` 调用点。
>
> 口径：基于当前代码（2026-07-09）。涉及 `OrchestrationService`、`TaskRun`/`TaskRunStatus`、`TaskRunRepository`、`SecurityConfig`、`PipelineController.listTaskRuns`、Flyway `orchestration`（当前最新 V10，1.3 后续迁移需从 V11 起）。
>
> 目标读者：实现该步骤的编码 Agent / 工程师。

## 1. 目标与范围

步骤 1.2 让 Dagster 执行器在每个节点开始/结束时向后端回调。1.3 负责后端**接收端**：一个受信任的内部接口，把节点级状态实时写入 `task_run`，并做幂等、状态机保护、实时短路传播。

**目标（M1 必达）**

- G-1 提供内部回调接口：`POST /api/v1/internal/orchestration/runs/{runId}/tasks/{taskKey}/status`。
- G-2 幂等 + 单调状态机：重复/乱序/迟到回调不破坏已终态节点。
- G-3 实时短路：节点 FAILED 时，立即把其下游 QUEUED 节点置 `UPSTREAM_FAILED`（不必等 Dagster run 终态）。
- G-4 机器鉴权：无 JWT 的 Dagster 子进程用共享内部令牌鉴权；租户从 `runId` 反查，不信任请求体。
- G-5 与 1.2 的回退兼容：LEGACY 模式下不发回调，接口不被调用；GRAPH 模式下 `syncTaskRunsFromTerminalRun` 降级为兜底补偿。

**非目标（M2+）**

- N-1 回调走消息队列（M1 用同步 HTTP，失败仅告警）。
- N-2 mTLS / 服务网格级鉴权（M1 用共享令牌 + 内网）。
- N-3 节点级指标聚合面板（步骤 1.6/M2）。

## 2. 现状与差距

- 现状 `task_run` 状态由 `OrchestrationService.syncTaskRunsFromTerminalRun` 在 **Dagster run 终态后**统一翻牌（SUCCEEDED/FAILED），运行中全程停在初始状态。
- 已有的 `propagateUpstreamFailures(run, taskRuns, failedOrBlocked)` 只在终态补偿时运行——**逻辑可复用**到实时回调。
- 无任何 inbound 内部接口；`SecurityConfig` 对非白名单路径一律 `authenticated()`（OAuth2 JWT）。
- 前端 `PipelineController.listTaskRuns`（`GET /{dagId}/runs/{runId}/tasks`）已存在，前端轮询即可看到实时状态——**无需新前端接口**。

## 3. 接口设计

### 3.1 单节点状态回调（主接口）

```
POST /api/v1/internal/orchestration/runs/{runId}/tasks/{taskKey}/status
Header: X-Onelake-Internal-Token: <shared-secret>
Content-Type: application/json
```

请求体（`TaskRunCallbackRequest`）：

```jsonc
{
  "status": "RUNNING",              // 必填：RUNNING|SUCCEEDED|FAILED|UPSTREAM_FAILED|SKIPPED|CANCELLED
  "startedAt": "2026-07-09T02:30:00Z",   // 可选（RUNNING 时应带）
  "finishedAt": "2026-07-09T02:31:20Z",  // 可选（终态时应带）
  "errorMsg": "…",                 // 可选（FAILED 时带，后端截断 4000）
  "artifactPath": "table:onelake.dwd.user_wide", // 可选
  "rowsWritten": 12000,            // 可选
  "scanBytes": 34567890,           // 可选
  "logRef": "logs/<tenant>/<run>/<taskKey>.log", // 可选（1.4 使用）
  "attempt": 1,                    // 可选（1.5 重试次数）
  "dagsterStepKey": "spark_dwd_sink" // 可选（可观测关联）
}
```

响应（统一 `ApiResponse`）：

```jsonc
{ "code": 0, "message": "ok", "data": { "applied": true, "currentStatus": "RUNNING" } }
```

- `applied=false` 表示因状态机保护被忽略（幂等/乱序），仍返回 200（回调方无需重试）。

### 3.2 可选：批量回调（性能优化，M1 可不实现）

```
POST /api/v1/internal/orchestration/runs/{runId}/tasks:batchStatus
Body: { "items": [ {"taskKey": "...", ...同上}, ... ] }
```

> M1 建议先只做单节点接口；节点数少、并发低，够用。批量留作后续优化。

### 3.3 幂等与状态机

状态优先级（rank）：

| 状态 | rank | 是否终态 |
| --- | --- | --- |
| QUEUED | 0 | 否 |
| RUNNING | 1 | 否 |
| SUCCEEDED / FAILED / CANCELLED / UPSTREAM_FAILED / SKIPPED | 2 | 是 |

写入规则（`applyTaskRunCallback`）：

1. 当前已是**终态** → 忽略（`applied=false`），保护先到的真实终态不被迟到回调覆盖。
2. 入参 rank **小于**当前 rank → 忽略（防 RUNNING→QUEUED 回退、SUCCEEDED 后又来 RUNNING）。
3. 同 rank（如 RUNNING→RUNNING）→ 仅补齐时间戳/attempt，不改状态。
4. rank 前进（QUEUED→RUNNING、RUNNING→终态）→ 应用状态及附带字段。
5. 进入终态时：`finishedAt` 缺省用服务端时间；FAILED 且 `errorMsg` 为空则填默认文案。
6. FAILED/UPSTREAM_FAILED/SKIPPED → 触发实时下游短路（§5.2）。

## 4. 鉴权设计

Dagster 子进程没有用户 JWT，用**共享内部令牌 + 内网**。

1. `SecurityConfig` 放行内部路径前缀：在 `permitAll(...)` 列表加 `"/api/v1/internal/**"`。
2. 令牌校验放 `OncePerRequestFilter`，挂到 Spring Security filter chain，在请求体反序列化前拒绝非法令牌，并用常量时间比较：

```java
// InternalApiTokenFilter（骨架）
private boolean validInternalToken(String provided) {
    if (!StringUtils.hasText(internalToken)) {
        return false;
    }
    byte[] expected = sha256(internalToken);
    byte[] actual = sha256(provided == null ? "" : provided);
    return MessageDigest.isEqual(expected, actual);
}

// 校验失败时 filter 直接返回 HTTP 403 + ApiResponse code=40300。
```

3. **租户从 `runId` 反查、不信任请求体**：`jobRun → dag.tenantId`。回调不依赖 `TenantContext`（无 JWT），服务方法按 `runId`/`taskKey` 直接定位行。

> 当前实现：使用 `OncePerRequestFilter` 拦截 `/api/v1/internal/**`，在请求体反序列化前完成内部令牌校验。

## 5. 服务层设计

### 5.1 `applyTaskRunCallback`

```java
// OrchestrationService（骨架）
@Transactional
public TaskRunCallbackResult applyTaskRunCallback(UUID runId, String taskKey,
                                                  TaskRunCallbackRequest req) {
    JobRun run = runRepo.findById(runId)
            .orElseThrow(() -> new BizException(40400, "运行实例不存在"));
    TaskRun tr = taskRunRepo.findByJobRunIdAndTaskKeyForUpdate(runId, taskKey)
            .orElseThrow(() -> new BizException(40400, "任务实例不存在"));

    TaskRunStatus incoming = req.status();
    if (!canTransition(tr.getStatus(), incoming)) {
        return new TaskRunCallbackResult(false, tr.getStatus());
    }

    // same-rank RUNNING refresh: only timestamps/attempt
    if (rank(incoming) == rank(tr.getStatus()) && incoming == TaskRunStatus.RUNNING) {
        applyTimestamps(tr, req);
        taskRunRepo.save(tr);
        return new TaskRunCallbackResult(true, tr.getStatus());
    }

    tr.setStatus(incoming);
    applyTimestamps(tr, req);
    if (incoming == TaskRunStatus.RUNNING && tr.getStartedAt() == null) {
        tr.setStartedAt(Instant.now());
    }
    if (isTerminalTaskRunStatus(incoming)) {
        if (tr.getFinishedAt() == null) tr.setFinishedAt(Instant.now());
        if (incoming == TaskRunStatus.FAILED && !StringUtils.hasText(req.errorMsg())
                && !StringUtils.hasText(tr.getErrorMsg())) {
            tr.setErrorMsg("Task failed (no detail reported)");
        }
    }
    if (StringUtils.hasText(req.errorMsg())) tr.setErrorMsg(truncate(req.errorMsg(), 3900));
    if (StringUtils.hasText(req.artifactPath())) tr.setArtifactPath(truncate(req.artifactPath(), 512));
    if (req.rowsWritten() != null) tr.setRowsWritten(req.rowsWritten());
    if (req.scanBytes() != null) tr.setScanBytes(req.scanBytes());
    if (StringUtils.hasText(req.logRef())) tr.setLogRef(truncate(req.logRef(), 512)); // V10 新列
    if (req.attempt() != null) tr.setAttempt(req.attempt());                          // V10 新列
    tr.setUpdatedAt(Instant.now());
    taskRunRepo.save(tr);

    // 实时短路
    if (incoming == TaskRunStatus.FAILED || incoming == TaskRunStatus.UPSTREAM_FAILED
            || incoming == TaskRunStatus.SKIPPED) {
        propagateUpstreamFailuresRealtime(run, taskKey);
    }
    return new TaskRunCallbackResult(true, incoming);
}

private boolean canTransition(TaskRunStatus current, TaskRunStatus incoming) {
    if (isTerminalTaskRunStatus(current)) return false;      // 终态不可变
    return rank(incoming) >= rank(current);                  // 不回退
}

private int rank(TaskRunStatus s) {
    return switch (s) {
        case QUEUED -> 0;
        case RUNNING -> 1;
        default -> 2; // terminal
    };
}
```

### 5.2 实时短路 `propagateUpstreamFailuresRealtime`

复用现有终态版 `propagateUpstreamFailures` 的 BFS 逻辑，但以单个失败节点为种子、只作用于 `QUEUED` 下游：

```java
private void propagateUpstreamFailuresRealtime(JobRun run, String failedTaskKey) {
    List<TaskRun> taskRuns = taskRunRepo.findByJobRunIdForUpdate(run.getId());
    Set<String> seed = new HashSet<>();
    seed.add(failedTaskKey);
    // 直接复用现有 propagateUpstreamFailures（它已按 PIPELINE 边做 BFS，
    // 只把 QUEUED 下游置 UPSTREAM_FAILED，幂等）
    propagateUpstreamFailures(run, taskRuns, seed);
}
```

> 说明：实时短路先锁定当前 run 的 task_run 行，和 GRAPH 终态兜底保持一致的锁获取顺序；现有 `propagateUpstreamFailures` 只把 `QUEUED` 下游改成 `UPSTREAM_FAILED`，已在运行中的下游不受影响，天然幂等。

### 5.3 与 JobRun 聚合状态的关系

- **JobRun 聚合状态仍以 Dagster run 为准**（`refreshRunStatus` + `mapDagsterStatus`）：GRAPH 模式下执行器在有节点失败时 `raise`，Dagster run 转 FAILED。
- 回调只负责**节点级实时刷新**；不直接改 `JobRun.status`，避免与 `refreshRunStatus` 竞态。
- `syncTaskRunsFromTerminalRun` 在 GRAPH 模式降级为**兜底补偿**：先锁定当前 run 的 `task_run` 行，再仅对回调丢失、仍非终态的记录按 run 终态补齐，避免并发节点回调被兜底覆盖。

## 6. 数据模型变更（Flyway V11）

当前代码库已用 `V10__task_run_node_execution.sql` 承载节点级执行字段；回调字段补齐与

## 7. 控制器骨架

```java
@RestController
@RequestMapping("/api/v1/internal/orchestration")
@RequiredArgsConstructor
@Tag(name = "编排内部回调", description = "Dagster 执行器回写节点级状态（内部令牌鉴权）。")
public class InternalTaskRunCallbackController {

    private final OrchestrationService service;

    @PostMapping("/runs/{runId}/tasks/{taskKey}/status")
    public ApiResponse<TaskRunCallbackResult> updateStatus(
            @PathVariable UUID runId,
            @PathVariable String taskKey,
            @RequestBody @Valid TaskRunCallbackRequest req) {
        return ApiResponse.ok(service.applyTaskRunCallback(runId, taskKey, req));
    }
}
```

  `onelake.orchestration.internal-token`。Dagster 侧 `_callback` 仍读取

- 建议把运行中轮询间隔调至 ~2–3s，使节点级实时状态体验顺滑（非必须）。

## 10. 测试策略

**Java 单测（`OrchestrationServiceTest`）**：
- 幂等/状态机：SUCCEEDED 后收到 RUNNING → `applied=false`、状态不变；RUNNING 后收到 QUEUED → 忽略；QUEUED→RUNNING→SUCCEEDED 正常前进。
- 短路：菱形拓扑，回调某节点 FAILED → 其 QUEUED 下游变 UPSTREAM_FAILED、旁路分支不受影响。
- 附带字段：SUCCEEDED 带 rowsWritten/artifactPath/logRef/attempt 正确落库。
- 与终态兜底并存：先回调部分节点终态，再触发 `syncTaskRunsFromTerminalRun`，已终态行不被覆盖。

**Web 层测试（`@WebMvcTest` 或 MockMvc）**：
- 缺令牌/错令牌 → 403/40300；正确令牌 → 200。
- 路径参数与 body 校验（非法 status 枚举 → 400）。

**集成（可选）**：GRAPH 模式跑样例流水线，观察 `task_run` 在运行中实时推进；人为让中间节点失败，确认下游实时 UPSTREAM_FAILED。

验证命令：`cd onelake-app && mvn -q -pl module-orchestration -am test -Djacoco.skip=true`。

## 12. 风险与回退

| 风险 | 缓解 |
| --- | --- |
| 令牌泄露 → 伪造回调 | 内网 + 常量时间比较；令牌用密钥管理注入；`runId`/`taskKey` 需真实存在才生效 |
| 并发回调改同一行竞态 | 单调状态机 + `@Transactional`；必要时对该行加乐观/悲观锁；不同节点为不同行天然无冲突 |
| 迟到/乱序回调覆盖真实终态 | rank 保护：终态不可变、不回退 |
| 回调风暴影响主库 | M1 节点/并发少可忽略；后续可批量接口 + 异步 |
| 回调丢失导致节点停在 RUNNING | `syncTaskRunsFromTerminalRun` 终态兜底补偿 |
| 回退需求 | 切 `pipeline-execution-mode=LEGACY`：不再发回调，接口闲置，行为回旧版 |
