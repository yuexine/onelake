```text
Epic 1 节点级执行引擎（A1–A5）
```

```sql
-- M1-A：节点级执行与数据日期字段

ALTER TABLE orchestration.job_run
  ADD COLUMN IF NOT EXISTS logical_date        TIMESTAMPTZ,

CREATE INDEX IF NOT EXISTS idx_job_run_backfill
    ON orchestration.job_run (backfill_id) WHERE backfill_id IS NOT NULL;
```

### V12__pipeline_partition_backfill.sql

```sql
-- M1-D：分区定义 + 真回填批次
-- 关联 §4.4（D1/D2）

ALTER TABLE orchestration.pipeline_task
  ADD COLUMN IF NOT EXISTS partition_key     VARCHAR(64),   -- 分区列，如 dt
```

### V13__scheduler_idempotency.sql

```sql
-- M1-C6：调度触发幂等 + 分布式锁
-- 关联 §4.3（C6）、§9 R2

-- 同一 DAG 同一 logical_date 的 CRON 触发只允许一条 job_run
CREATE UNIQUE INDEX IF NOT EXISTS uq_job_run_cron_logical
    ON orchestration.job_run (dag_id, logical_date)
    WHERE trigger_type = 'CRON' AND logical_date IS NOT NULL;

-- 调度器多副本互斥锁（也可改用 Redis / pg_advisory_lock，二选一）

  holder      VARCHAR(128) NOT NULL,
  acquired_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at  TIMESTAMPTZ NOT NULL
);
```

### 步骤 1.2 — Dagster 动态图执行

- **目标**：把 `onelake_pipeline_run` 从「单 op 循环所有 task」改为「按拓扑生成的 op 图」，使引擎侧节点独立执行、独立成败、失败节点阻断下游。
- **涉及文件**：`onelake-app/dagster/definitions.py`；`service/spi/SparkRunConfigBuilder.java`（runConfig 结构从单 op 的 `tasks[]` 改为 per-op config）；`OrchestrationService.buildPipelineRunConfig` 调用处。
- **实现要点**：
  - `definitions.py` 用 Dagster `@graph` + 动态构建，或保留单 job 但用 `dynamic_output` / 多 op 依赖；每个 pipeline_task 对应一个 `spark_task_op` 实例，边对应 op 依赖。
  - 单节点失败 → Dagster 自动跳过下游（对齐 `UPSTREAM_FAILED` 语义）。
  - runConfig 仍由 Java 侧生成（C3 契约不变），但结构改为按 step_key 组织。
- **验收标准**：一条含 SYNC_REF→SPARK_SQL→QUALITY_GATE 的样例流水线触发后，Dagster UI 显示 3 个独立 step；人为让中间节点失败时，下游 step 为 skipped 而非全部 failed。

> **Agentic 开发提示词**
> ```
> 目标：升级 onelake-app/dagster/definitions.py 的 onelake_pipeline_run，使 Spark 流水线在 Dagster 侧按节点级 op 图执行，而非当前单 op run_spark_task_op 内 for-loop。约束：
> 1. 先读 onelake-app/dagster/definitions.py、module-orchestration 的 SparkRunConfigBuilder.java、OrchestrationService.triggerPipelineRun/buildPipelineRunConfig，理解现有 runConfig 结构（ops.run_spark_task_op.config.tasks[]）。
> 2. 保持 spark-submit 的执行细节（iceberg catalog、resource_profile → spark-submit flags）完全不变，只改「编排粒度」：每个 task 成为独立 op，边成为 op 依赖，上游失败则下游 skip。
> 3. 采用 Dagster 官方推荐方式（graph-backed job 或 dynamic op），保证 dagster_step_key 稳定可回传（等于 pipeline_task.task_key）。
> 4. 同步调整 SparkRunConfigBuilder.build，使 runConfig 结构与新 op 图匹配；OrchestrationService 调用处随之更新。保持「Java 生成 runConfig、Dagster 执行」的 C3 契约。
> 5. 提供最小回退开关（如 dag/dagster job 名或配置项），可切回旧单 op 路径。
> 验证：cd onelake-app && make dagster-up 后在 Dagster UI(http://localhost:3000) 用样例 runConfig 跑一条 3 节点流水线，确认 3 个独立 step 与失败阻断行为。报告：改动文件、Dagster UI 截图或 step 列表、回退方式。
> ```

### 步骤 1.3 — 节点级状态回写通道

- **目标**：替换 `syncTaskRunsFromTerminalRun` 的「终态粗刷」，让每个 op 完成即回写对应 `task_run` 的真实状态、行数、产物、错误。
- **涉及文件**：`onelake-app/dagster/definitions.py`（op 内调用后端回写）；新增后端 `InternalTaskRunCallbackController`（内部回调：`POST /api/v1/internal/orchestration/runs/{runId}/tasks/{taskKey}/status`）；`OrchestrationService`；`DomainEvents` 增 `PIPELINE_TASK_RUN_STARTED/SUCCEEDED/FAILED`（可选）。
- **实现要点**：
  - Dagster op 完成时 HTTP 回调后端（带内部鉴权/共享密钥），后端幂等更新 `task_run`（按 job_run_id + task_key）。
  - 保留终态兜底：Dagster run 整体终态时对未回写节点做补偿（防回调丢失）。
  - 回调鉴权走内部 token，避免暴露到公网。
- **验收标准**：运行中前端每 5s 轮询能看到节点状态逐个从 QUEUED→RUNNING→SUCCEEDED 推进，而非结束后一次性翻转。

> **Agentic 开发提示词**
> ```
> 目标：为编排 M1 建立 Dagster→后端 的节点级状态回写通道，替换 OrchestrationService.syncTaskRunsFromTerminalRun 的终态粗刷。约束：
> 1. 后端新增内部回调接口（POST /api/v1/internal/orchestration/runs/{runId}/tasks/{taskKey}/status），入参含 status(RUNNING/SUCCEEDED/FAILED)、rowsWritten、scanBytes、artifactPath、errorMsg、logRef、dagsterStepKey、startedAt/finishedAt。用共享密钥/内部 token 鉴权（参考 module-common 现有安全配置），不要放入需要用户 JWT 的链路。

> 验证：mvn -q -pl module-orchestration -am test；补一个单测覆盖回调幂等与 UPSTREAM_FAILED 传播。报告改动文件与测试结果。

### 步骤 1.4 — 节点级日志

- **目标**：每个节点的 spark-submit stdout/stderr 落到 MinIO，`task_run.log_ref` 记录路径，前端可查看/下载。
- **涉及文件**：`definitions.py`（把 stdout/stderr 写 MinIO 并回传 log_ref）；后端 `GET /api/v1/orchestration/pipelines/{dagId}/runs/{runId}/tasks/{taskKey}/log`（从 MinIO 拉取，支持 tail/下载）；`OrchestrationService`。
- **实现要点**：日志对象命名 `{tenantId}/{jobRunId}/{taskKey}/attempt-{n}.log` 与 `{tenantId}/{jobRunId}/{taskKey}/latest.log`；接口支持 `?tail=500` 返回尾部，`download=true` 流式下载。
- **验收标准**：运行结束后能在前端查看某节点的 spark 日志；失败节点日志含报错栈。

> **Agentic 开发提示词**

### 步骤 1.5 — 节点级重试 / 从失败续跑

- **目标**：支持对单个失败节点重跑，或从失败节点起对整条流水线续跑，不必整体重来。
- **涉及文件**：`PipelineController` 增 `POST /pipelines/{dagId}/runs/{runId}/tasks/{taskKey}/rerun`；`OrchestrationService.rerunTask`；`SparkRunConfigBuilder`（构造子图 runConfig）。
- **实现要点**：
  - 重跑生成新的 attempt（`task_run.attempt+1`）或新 job_run（记录 `parent_run_id`，M1 可先用同 job_run 递增 attempt）。
  - 「从失败续跑」= 以失败节点为根的下游子图重新编译+触发。
  - 校验：仅允许对终态（FAILED/UPSTREAM_FAILED）节点重跑。
- **验收标准**：失败节点重跑成功后，其下游被重新调度；成功节点不重复执行。

> **Agentic 开发提示词**
> ```
> 目标：为编排 M1 增加节点级重跑与「从失败续跑」。约束：
> 1. PipelineController 新增 POST /api/v1/orchestration/pipelines/{dagId}/runs/{runId}/tasks/{taskKey}/rerun，body 支持 mode=SINGLE|DOWNSTREAM（默认 SINGLE）。@PreAuthorize("hasRole('DE')")。
> 2. OrchestrationService.rerunTask：校验目标 task_run 处于终态失败态（FAILED/UPSTREAM_FAILED）；SINGLE 仅重跑该节点，DOWNSTREAM 以该节点为根重编译下游子图并触发；复用 PipelineCompileService 与 SparkRunConfigBuilder 构造子图 runConfig。
> 3. 重跑写入 task_run.attempt+1（保留历史 attempt 语义），并复用 1.3 回调更新状态。成功节点不得重复执行。
> 4. 触发失败要正确置 task_run 状态与 errorMsg，遵循现有 BizException/ApiResponse 约定。
> 验证：补单测覆盖「非终态节点重跑被拒」「DOWNSTREAM 子图选择正确」；mvn -q -pl module-orchestration -am test。报告改动文件与测试结果。
> ```

> **用户模拟测试提示词**
> ```
> 作为数据工程师，验证失败重跑闭环：

> 报告：每周期 run 数、logical_date 正确性、多副本下是否去重成功。
