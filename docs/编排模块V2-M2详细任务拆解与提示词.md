## 1. 迁移脚本草案（合并预览）

> 落位：`onelake-app/bootstrap/src/main/resources/db/migration/orchestration/`，续接仓库当前的 `V15`。全部 `ADD COLUMN ... DEFAULT` / `CREATE TABLE IF NOT EXISTS`，兼容存量数据。执行前 `make up && make migrate`。

### V16__dag_scheduling_enhancements.sql

```sql
-- M2-C：生产级调度增强（时区/日历/catchup/并发/优先级/空跑冻结/SLA）
-- 关联 docs/数据开发与编排模块V2升级计划.md §4.3（C1/C3/C4/C5）、§5

ALTER TABLE orchestration.dag
  ADD COLUMN IF NOT EXISTS timezone         VARCHAR(64)  NOT NULL DEFAULT 'Asia/Shanghai',
  ADD COLUMN IF NOT EXISTS catchup          BOOLEAN      NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS max_active_runs  INT          NOT NULL DEFAULT 1,
  ADD COLUMN IF NOT EXISTS priority         INT          NOT NULL DEFAULT 5,
  ADD COLUMN IF NOT EXISTS sla_minutes      INT,
  ADD COLUMN IF NOT EXISTS timeout_minutes  INT,
  ADD COLUMN IF NOT EXISTS schedule_mode    VARCHAR(16)  NOT NULL DEFAULT 'NORMAL',   -- NORMAL|DRY_RUN|FROZEN
  ADD COLUMN IF NOT EXISTS misfire_policy   VARCHAR(16)  NOT NULL DEFAULT 'FIRE_ONCE',-- FIRE_ONCE|SKIP
  ADD COLUMN IF NOT EXISTS calendar_id      UUID,
  ADD COLUMN IF NOT EXISTS schedule_start   TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS schedule_end     TIMESTAMPTZ;

ALTER TABLE orchestration.job_run
  ADD COLUMN IF NOT EXISTS run_mode   VARCHAR(16) NOT NULL DEFAULT 'NORMAL',  -- NORMAL|DRY_RUN
  ADD COLUMN IF NOT EXISTS sla_missed BOOLEAN     NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS priority   INT         NOT NULL DEFAULT 5;

-- 调度日历（工作日/节假日）C3
CREATE TABLE IF NOT EXISTS orchestration.schedule_calendar (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id  UUID NOT NULL,
  name       VARCHAR(128) NOT NULL,
  timezone   VARCHAR(64)  NOT NULL DEFAULT 'Asia/Shanghai',
  created_at TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE TABLE IF NOT EXISTS orchestration.schedule_calendar_day (
  calendar_id UUID NOT NULL REFERENCES orchestration.schedule_calendar(id) ON DELETE CASCADE,
  day         DATE NOT NULL,
  day_type    VARCHAR(16) NOT NULL,   -- HOLIDAY|WORKDAY
  PRIMARY KEY (calendar_id, day)
);

DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1
    FROM pg_constraint
    WHERE conname = 'fk_dag_schedule_calendar'
      AND conrelid = 'orchestration.dag'::regclass
  ) THEN
    ALTER TABLE orchestration.dag
      ADD CONSTRAINT fk_dag_schedule_calendar
      FOREIGN KEY (calendar_id)
      REFERENCES orchestration.schedule_calendar(id)
      ON DELETE SET NULL;
  END IF;
END $$;

CREATE INDEX IF NOT EXISTS idx_calendar_tenant ON orchestration.schedule_calendar (tenant_id);
CREATE INDEX IF NOT EXISTS idx_job_run_active ON orchestration.job_run (dag_id, status);
```

### V17__pipeline_dependency.sql

```sql
-- M2-C2：跨流水线 / 跨周期依赖
CREATE TABLE IF NOT EXISTS orchestration.pipeline_dependency (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id         UUID NOT NULL,
  downstream_dag_id UUID NOT NULL REFERENCES orchestration.dag(id) ON DELETE CASCADE,

  dependency_type   VARCHAR(16) NOT NULL DEFAULT 'SAME_CYCLE',  -- SAME_CYCLE|CROSS_CYCLE

CREATE INDEX IF NOT EXISTS idx_dep_downstream ON orchestration.pipeline_dependency (downstream_dag_id);
CREATE INDEX IF NOT EXISTS idx_dep_upstream   ON orchestration.pipeline_dependency (upstream_dag_id);
```

### 步骤 4.2 — 数据日期/业务日期模型（C1）

- **目标**：建立 `logical_date` 与 `data_interval_[start,end]` 的统一计算，触发时写入 `job_run`，并构造供参数体系与 SQL 使用的 `RunContext`。
- **涉及文件**：新增 `RunContext.java`（run 级上下文：logicalDate/dataInterval/timezone/runMode/backfillId）；`OrchestrationService.triggerPipelineRun`（接受/计算数据日期并写 job_run）；新增 `DataIntervalCalculator`（按 cron + grain + timezone 推算区间）。
- **实现要点**：
  - CRON 触发：`logical_date` = 本周期计划时间；`data_interval` 按调度粒度推算（如日调度 `[bizdate 00:00, +1d)`）。

  - 语义对标 Airflow：`logical_date` 是“数据所属周期”，非“真实执行时刻”。
- **验收标准**：CRON/手动/回填三种触发的 `job_run.logical_date/data_interval_*` 均正确落库；单测覆盖日/时/月粒度区间推算。

### 步骤 4.3 — 调度增强（C3：时区/日历/catchup/misfire/并发/优先级）

- **目标**：`PipelineSchedulerService` 从“系统时区分钟 tick”升级为“按 dag 时区评估 cron + 日历过滤 + catchup 补历史 + max_active_runs 限流 + priority 排序 + misfire 策略”。
- **涉及文件**：`PipelineSchedulerService.java`（核心）；`OrchestrationService`（并发计数校验）；`ScheduleCalendar*` 查询；复用 M1 的 `scheduler_lock`（V12）与回填派发器。
- **实现要点**：
  - **时区**：`isCronDue` 用 `dag.timezone` 而非 `systemDefault`。
  - **日历**：命中 cron 后，若 `calendar_id` 指定且当日为 HOLIDAY 则跳过（或按配置顺延）。
  - **catchup**：`enabled` 从关到开、或长时间未跑时，若 `catchup=true`，计算自 `last logical_date` 起遗漏的周期，交由 M1 回填派发器补跑；`catchup=false` 只跑当前周期。
  - **max_active_runs**：触发前统计该 dag 处于 RUNNING/QUEUED 的 job_run 数，达上限则本周期跳过（记 misfire）。
  - **priority + misfire**：多条同时到期按 `priority` 排序；`misfire_policy=SKIP` 时错过不补，`FIRE_ONCE` 补一次。
- **验收标准**：跨时区流水线按其本地时间触发；节假日跳过；关停再启用且 catchup=true 时补跑遗漏周期；并发不超过 `max_active_runs`。

> **Agentic 开发提示词**
> ```
> 目标：升级 PipelineSchedulerService 为生产级调度（C3）。约束：
> 1. isCronDue 改为按 dag.timezone 评估 CronExpression（用 ZoneId.of(dag.getTimezone())），替换 systemDefault。
> 2. 命中后依次过滤：调度窗口(schedule_start/end)；日历(若 calendar_id 命中当日 HOLIDAY 则跳过)；schedule_mode=FROZEN 跳过(见 4.4)。
> 3. max_active_runs：触发前统计该 dag 非终态 job_run 数，达上限则跳过并记 misfire 日志。
> 4. catchup：新增 CatchupPlanner，enabled 由关转开或距上次 logical_date 有缺口时，若 dag.catchup=true，展开遗漏周期并交给 M1 回填派发器(BackfillDispatcher)补跑；catchup=false 只跑当前周期。
> 5. 多条到期按 dag.priority 排序触发；沿用 M1 的 scheduler_lock 抢锁与 (dag_id,logical_date) 幂等。
> 6. 保留 PUBLISHED + onelake_pipeline_run + enabled 过滤与租户上下文。
> 验证：补单测覆盖 时区评估/日历跳过/并发限流/catchup 展开；mvn -q -pl module-orchestration -am test。报告改动文件与测试结果。
> ```

> **用户模拟测试提示词**
> ```

### 步骤 4.4 — 空跑/冻结/失败自动重跑（C4）

- **目标**：支持 `schedule_mode = NORMAL | DRY_RUN | FROZEN`，以及失败自动重跑策略。
- **涉及文件**：`PipelineSchedulerService`、`OrchestrationService.triggerPipelineRun`、`SparkRunConfigBuilder`（DRY_RUN 短路）、`Dag`/`JobRun`。
- **实现要点**：
  - **DRY_RUN（空跑）**：照常生成 job_run/task_run 与拓扑，但不真正 spark-submit（Dagster 侧走 no-op 或 Java 侧直接标记 SUCCEEDED），`job_run.run_mode=DRY_RUN`；**不阻塞下游依赖**（对标 DataWorks 空跑）。
  - **FROZEN（冻结）**：调度器跳过触发，且下游跨周期依赖判定视其为“未就绪/阻塞”（对标 DataWorks 冻结）。
  - **失败自动重跑**：`dag` 级重跑策略（复用 `task_run.max_retries` 思路，或 dag 级 `run_retry`），run 整体失败后按策略自动重触发一次（区别于 M1 的节点级重试）。
- **验收标准**：DRY_RUN 生成 run 但不产数据、下游不被阻塞；FROZEN 不触发且阻塞下游；失败按策略自动重跑一次。

> **Agentic 开发提示词**
> ```
> 目标：实现空跑/冻结/失败自动重跑（C4）。约束：
> 1. DRY_RUN：triggerPipelineRun 读 dag.schedule_mode，DRY_RUN 时照常建 job_run(run_mode=DRY_RUN)/task_run 与拓扑，但跳过真实 spark-submit（推荐 Dagster runConfig 加 dry_run=true，op 内直接标记各节点 SUCCEEDED；或 Java 侧不 launch 直接置成功）。DRY_RUN 不得阻塞下游依赖判定。
> 2. FROZEN：PipelineSchedulerService 对 schedule_mode=FROZEN 跳过触发；跨周期依赖(4.6)判定把 FROZEN 上游视为阻塞(未就绪)。
> 3. 失败自动重跑：dag 级策略(如 run_retry 次数/间隔，可存 dag 或复用现有字段)，run 进入 FAILED 终态且未超次数时自动重触发一次，记录重跑来源，避免与 M1 节点级重试冲突。
> 4. 保持 NORMAL 行为不变。
> 验证：补单测覆盖 DRY_RUN 不落数据/不阻塞、FROZEN 不触发、失败自动重跑次数上限；mvn -q -pl module-orchestration -am test。报告改动文件与测试结果。
> ```

> **用户模拟测试提示词**
> ```
> 作为数据工程师，验证空跑/冻结/自动重跑：
> 1. 把某流水线设为“空跑”后触发，确认生成运行记录且节点为成功，但目标表数据未变化；其下游流水线不被阻塞。
> 2. 把某流水线设为“冻结”，确认到点不触发；其下游跨周期依赖显示为“等待上游/被阻塞”。
> 3. 配置失败自动重跑=1，制造一次整体失败，确认系统自动重跑一次；再失败则不再重跑。
> 报告：空跑是否产数据/是否阻塞下游、冻结是否阻塞、自动重跑次数是否受控。

### 步骤 4.5 — SLA / 超时（C5）

- **目标**：`sla_minutes` 破线与 `timeout_minutes` 超时检测，触发告警事件并标记 run。
- **涉及文件**：新增 `SlaMonitorService`（`@Scheduled` 巡检进行中 run）；`OrchestrationService`（超时取消，复用 M1 cancelRun）；`DomainEvents` 增 `PIPELINE_RUN_SLA_MISSED`/`PIPELINE_RUN_TIMEOUT`；`JobRun.slaMissed`。
- **实现要点**：
  - 巡检非终态 run：`now - startedAt > timeout_minutes` → 取消并发 `PIPELINE_RUN_TIMEOUT`；`> sla_minutes` 未完成 → 置 `sla_missed=true` 并发 `PIPELINE_RUN_SLA_MISSED`（仅一次）。
  - 事件对接 `module-common` 通知总线 / `module-quality` 告警（M2 先发事件，通知消费在 M4 I3 完善）。
- **验收标准**：超时 run 被自动取消并发事件；破 SLA run 被标记并发事件（幂等不重复发）。

### 步骤 4.6 — 跨周期依赖（C2）

- **目标**：`pipeline_dependency` 描述流水线间上下游（同周期/跨周期/偏移），调度器据此判定下游 ready。
- **涉及文件**：新增 `V17__pipeline_dependency.sql`；`PipelineDependency.java` + Repository；新增 `DependencyReadinessService`；`PipelineSchedulerService`（触发前校验依赖就绪）；`PipelineController` 或新 `PipelineDependencyController`。
- **实现要点**：
  - 下游到期时，对每条依赖计算目标上游周期：`SAME_CYCLE` 取同 `logical_date`；`CROSS_CYCLE` 按 `offset_grain/offset_n` 偏移。
  - 就绪 = 目标上游周期存在 SUCCEEDED 的 job_run（FROZEN 上游视为阻塞、DRY_RUN 视为已满足）。
  - 未就绪：本周期等待（下一 tick 重判），可配等待超时。
- **验收标准**：下游在所有上游对应周期成功前不触发；上游成功后下一 tick 触发；跨周期偏移正确。

## 3. 风险提示（M2 专项）

- **R-M2-1 调度器复杂度上升**：时区/日历/catchup/依赖判定叠加，务必保持 M1 的 `scheduler_lock` + `(dag_id, logical_date)` 幂等，所有新分支都要在锁内且可跳过重入。
- **R-M2-2 catchup 放大效应**：长时间停用后开 catchup 可能瞬间展开大量周期，需复用 M1 回填派发器的 `max_parallel` 限流，并设展开上限保护。
- **R-M2-3 数据日期语义一致性**：`logical_date` 必须在调度触发、回填、参数渲染三处口径一致（同一 `DataIntervalCalculator`/`RunContext`），否则回填与调度结果不可复现。
