# 编排模块 V2 · M3 详细任务拆解、迁移脚本与提示词

> 上游文档：`docs/数据开发与编排模块V2升级计划.md`（M3 = 版本快照/回滚/审批 F1–F4 + 资产驱动自动化 E1–E3）
>
> 前置：M1、M2 已落地并收口，数据库迁移头为 **V23**。M3 必须从 **V24** 续接；当前 `job_run` 尚无 `pipeline_version_id`，由 V24 正式增加，不得复用已占用的 V17–V23 编号。
>
> 口径：基于当前代码事实（2026-07-11）。对齐现状：`PipelineService`（发布状态机 DRAFT→VALIDATED→PUBLISHED，`dag.version` 整数自增，发 `pipeline.published`）、`PipelineSyncRefTriggerHandler`（消费 `integration.table.loaded` 触发含 `SYNC_REF` 的流水线，含就绪 barrier 雏形）、`PipelineTaskEdge.freshnessPolicy`（默认 `LATEST`，当前仅编译提示）、`module-security` 的 `ApprovalRequest`（`requestType` 已含 `PUBLISH`，审批 API 齐备）。
>
> 使用方式：每步含「目标 / 涉及文件 / 实现要点 / 验收标准 / Agentic 开发提示词 / 用户模拟测试提示词」。

## 0. M3 范围与执行顺序

```text
Epic 6 版本、发布与环境（F1–F4）
  6.1 DB 迁移 V24（pipeline_version 表 + job_run.pipeline_version_id + dag.published_version_id） [基石·先做]
  6.2 发布即快照（F1）：publish 整包快照，run 绑定版本，调度/触发跑「已发布快照」
  6.3 版本对比 / 回滚（F2）：diff + rollback API
  6.4 发布审批（F3）：接 module-security ApprovalRequest(PUBLISH)，可配开关
  6.5 开发 / 生产隔离（F4）：DEV 编辑态 + PROD 快照运行态
  6.6 前端：版本历史 / 对比 / 回滚 + 审批入口

Epic 7 资产感知的声明式自动化（E1–E3）
  7.1 DB 迁移 V25（pipeline_subscription + asset_readiness）
  7.2 通用资产订阅触发（E1）：泛化 PipelineSyncRefTriggerHandler，兼容 integration.table.loaded + pipeline.task.loaded
  7.3 新鲜度策略参与就绪判定（E2）：LATEST / SAME_FRESHNESS_WINDOW / SAME_BATCH
  7.4 声明式条件（E3）：ON_UPDATE_AND_QUALITY_PASS（消费 quality.check.completed）
  7.5 前端：自动化 / 订阅配置面板
```

依赖关系：`6.1 → 6.2 → {6.3, 6.4, 6.5}`；`6.2` 是 7.x 的前提（事件/调度触发的 run 必须绑定并执行「已发布快照」）；`7.1 → 7.2 → 7.3 → 7.4`；`6.6/7.5` 前端最后接入。建议先 Epic 6 再 Epic 7。

## 1. 迁移脚本草案（合并预览）

> 落位：`onelake-app/bootstrap/src/main/resources/db/migration/orchestration/`，续接 M2 的 `V23__schedule_wait_lifecycle.sql`。全部 `ADD COLUMN ... DEFAULT` / `CREATE TABLE IF NOT EXISTS`，兼容存量数据。执行前 `make up && make migrate`。

### V24__pipeline_version.sql

```sql
-- M3-F：不可变版本快照 + run 绑定版本 + 开发/生产隔离
-- 关联 docs/数据开发与编排模块V2升级计划.md §4.6（F1/F4）、§5

CREATE TABLE IF NOT EXISTS orchestration.pipeline_version (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL,
  dag_id          UUID NOT NULL REFERENCES orchestration.dag(id) ON DELETE CASCADE,
  version         INT  NOT NULL,                 -- 每个 dag 单调递增
  snapshot        jsonb NOT NULL,                -- {dag 核心配置, tasks[], edges[], params[], schedule}
  checksum        VARCHAR(64),                   -- snapshot 内容哈希，用于去重/对比
  status          VARCHAR(16) NOT NULL DEFAULT 'PUBLISHED',  -- PUBLISHED|ARCHIVED
  note            VARCHAR(512),
  published_by      UUID,
  published_by_name VARCHAR(128),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_pipeline_version UNIQUE (dag_id, version)
);
CREATE INDEX IF NOT EXISTS idx_pipeline_version_dag ON orchestration.pipeline_version (dag_id);

-- run 绑定版本（M1 §5 计划的预留位，此处正式落地）
ALTER TABLE orchestration.job_run
  ADD COLUMN IF NOT EXISTS pipeline_version_id UUID;

-- 开发/生产隔离：dag 为 DEV 编辑态，published_version_id 指向当前 PROD 快照
ALTER TABLE orchestration.dag
  ADD COLUMN IF NOT EXISTS published_version_id UUID,
  ADD COLUMN IF NOT EXISTS has_unpublished_changes BOOLEAN NOT NULL DEFAULT false;

CREATE INDEX IF NOT EXISTS idx_job_run_version ON orchestration.job_run (pipeline_version_id)
  WHERE pipeline_version_id IS NOT NULL;
```

### V25__pipeline_automation.sql

```sql
-- M3-E：资产订阅 + 新鲜度就绪持久化
-- 关联 §4.5（E1/E2/E3）

CREATE TABLE IF NOT EXISTS orchestration.pipeline_subscription (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL,
  dag_id        UUID NOT NULL REFERENCES orchestration.dag(id) ON DELETE CASCADE, -- 被触发的下游
  source_type   VARCHAR(16) NOT NULL,     -- ASSET|PIPELINE
  source_ref    VARCHAR(256) NOT NULL,    -- 资产 FQN 或上游 dag_id
  condition     VARCHAR(32)  NOT NULL DEFAULT 'ON_UPDATE', -- ON_UPDATE|ON_UPDATE_AND_QUALITY_PASS
  freshness_policy VARCHAR(32) NOT NULL DEFAULT 'LATEST',  -- LATEST|SAME_FRESHNESS_WINDOW|SAME_BATCH
  enabled       BOOLEAN NOT NULL DEFAULT true,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_pipeline_subscription UNIQUE (dag_id, source_type, source_ref)
);
CREATE INDEX IF NOT EXISTS idx_subscription_source ON orchestration.pipeline_subscription (tenant_id, source_type, source_ref);

-- 多输入就绪持久化（替换 PipelineSyncRefTriggerHandler 的内存 readinessByDag，支持多副本 + 新鲜度窗口）
CREATE TABLE IF NOT EXISTS orchestration.asset_readiness (
  tenant_id   UUID NOT NULL,
  dag_id      UUID NOT NULL,
  task_key    VARCHAR(128) NOT NULL,
  asset_fqn   VARCHAR(256) NOT NULL,
  batch_id    VARCHAR(128),               -- SAME_BATCH 用
  freshness_window VARCHAR(64),           -- SAME_FRESHNESS_WINDOW 用（如 logical_date 桶）
  ready_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (dag_id, task_key)
);
```

## Epic 6 · 版本、发布与环境（F1–F4）

### 步骤 6.1 — DB 迁移 V24（版本表 + run 绑定 + 环境隔离）

- **目标**：落地不可变版本快照表，`job_run` 绑定版本，`dag` 增加发布版本指针与「有未发布变更」标记。
- **涉及文件**：新增 `V24__pipeline_version.sql`；新增实体 `PipelineVersion.java` + `PipelineVersionRepository`；`JobRun.java` 增 `pipelineVersionId`；`Dag.java` 增 `publishedVersionId`、`hasUnpublishedChanges`。
- **实现要点**：`snapshot` 用 jsonb（参考 `Dag.definition` 的 `@ColumnTransformer(write="?::jsonb")`）；`checksum` 用 snapshot 规范化后哈希，便于对比与「无变更不产生新版本」。
- **验收标准**：`make migrate` 成功；实体映射正确；`mvn -q -pl module-orchestration -am test -Djacoco.skip=true` 通过。

> 2. 新增 PipelineVersion 实体(dagId/version/snapshot(jsonb,用 @ColumnTransformer(write="?::jsonb"))/checksum/status/note/publishedBy/publishedByName) + Repository(findByDagIdOrderByVersionDesc、findByDagIdAndVersion)。

### 步骤 6.2 — 发布即快照（F1）

- **目标**：进入 `PUBLISHED` 时对 `dag + tasks + edges + params + schedule` 做整包不可变快照；run 绑定 `pipeline_version_id`；调度/事件触发执行「已发布快照」而非实时草稿。
- **涉及文件**：`PipelineService.updateStatus`（发布分支写快照）；新增 `PipelineSnapshotService`（组装/校验/哈希 snapshot、按快照重建运行输入）；`OrchestrationService.triggerPipelineRun`（读取 `published_version_id` 快照编译，而非实时 dag）；`PipelineSchedulerService`/`PipelineSyncRefTriggerHandler`（触发绑定版本）。
- **实现要点**：
  - 发布时：序列化当前 dag/tasks/edges/params → `pipeline_version`（version = 上一版本+1），置 `dag.published_version_id`、`has_unpublished_changes=false`。
  - 触发时：`triggerPipelineRun` 优先按 `published_version_id` 的 snapshot 编译执行，并写 `job_run.pipeline_version_id`。手动「试跑草稿」可显式走 DEV。
  - 任何对已发布流水线的编辑 → 置 `has_unpublished_changes=true`（提示需重新发布）。
  - checksum 相同则不产生新版本（幂等发布）。
- **验收标准**：发布生成一条 `pipeline_version`；调度/事件触发的 run 绑定该版本并按快照执行；发布后改草稿不影响已发布快照的运行结果。

> **Agentic 开发提示词**
> ```
> 目标：实现「发布即不可变快照 + run 绑定版本 + 跑已发布快照」（F1）。约束：
> 1. 新增 PipelineSnapshotService：snapshot(dagId) 组装 {dag 核心字段, tasks[], edges[], pipeline_param[], schedule 配置} 为规范化 JSON 并算 checksum；publishSnapshot 写 pipeline_version(version=max+1)，置 dag.published_version_id 与 has_unpublished_changes=false；checksum 未变则复用旧版本(幂等)。
> 2. PipelineService.updateStatus 的 PUBLISHED 分支调用 publishSnapshot；保留现有 pipeline.published 事件并补 versionId 到 payload(向后兼容)。
> 3. OrchestrationService.triggerPipelineRun 增可选参数 useVersionId/或默认读 dag.published_version_id：按 snapshot 编译执行(而非实时 dag)，写 job_run.pipeline_version_id。若无已发布版本(仅 DEV 试跑)保持现有实时编译路径。
> 4. PipelineService 编辑类写操作对已 PUBLISHED 的 dag 置 has_unpublished_changes=true。
> 5. PipelineCompileService 需能从 snapshot 重建编译输入(抽出「从 tasks/edges 集合编译」的纯函数，避免只能读库)。
> 验证：补单测覆盖 快照生成/幂等/run 绑定版本/改草稿不影响已发布快照；mvn -q -pl module-orchestration -am test。报告改动文件、快照结构与测试结果。
> ```

> **用户模拟测试提示词**
> ```
> 作为数据工程师，验证版本快照：
> 1. 编辑并发布一条流水线，确认生成「版本 1」；查看版本详情能看到当时的节点/边/参数快照。
> 2. 手动或定时触发一次，确认该 run 标注绑定「版本 1」。
> 3. 发布后修改某节点 SQL(先不重新发布)，再触发调度，确认运行结果仍按「版本 1」快照(草稿改动不生效)，且界面提示「有未发布变更」。
> 4. 重新发布 → 生成「版本 2」，之后触发绑定版本 2。
> 报告：版本生成/绑定、草稿隔离、未发布变更提示是否正确。
> ```

### 步骤 6.3 — 版本对比 / 回滚（F2）

- **目标**：列出历史版本、两版本 diff、一键回滚到历史版本。
- **涉及文件**：新增 `PipelineVersionController`（`GET /pipelines/{id}/versions`、`GET /pipelines/{id}/versions/{v}`、`GET /pipelines/{id}/versions/diff?from=&to=`、`POST /pipelines/{id}/versions/{v}/rollback`）；`PipelineVersionService`（diff/rollback）；`DomainEvents` 增 `PIPELINE_ROLLED_BACK`（可选）。
- **实现要点**：
  - diff：对两版本 snapshot 做结构化比较（新增/删除/修改的 task/edge/param 字段级差异）。
  - rollback：把目标历史版本 snapshot 回写为当前 DEV 草稿（不直接改历史版本），再走正常发布生成新版本（对标「回滚=以旧内容产生新版本」，保持版本线性不可变）。
- **验收标准**：版本列表/详情/diff 正确；回滚后草稿等于目标版本内容，重新发布得到新版本号。

> **Agentic 开发提示词**
> ```
> 目标：实现版本对比与回滚（F2）。约束：
> 1. PipelineVersionService.diff(dagId, fromV, toV)：对两 snapshot 做结构化 diff，输出 task/edge/param 的 added/removed/changed(字段级)。
> 2. rollback(dagId, targetV)：把目标版本 snapshot 还原为当前 DEV 草稿(覆盖 dag/tasks/edges/params)，置 has_unpublished_changes=true；不修改历史 pipeline_version(不可变)。回滚后由用户再走发布生成新版本。可选发 PIPELINE_ROLLED_BACK 事件。
> 3. PipelineVersionController：GET /versions(列表)、GET /versions/{v}(详情)、GET /versions/diff?from&to、POST /versions/{v}/rollback；@PreAuthorize hasRole('DE')；ApiResponse 包装 + Swagger 注解。
> 验证：补单测覆盖 diff 三类差异 + rollback 后草稿等值 + 历史版本不可变；mvn -q -pl module-orchestration -am test。报告接口清单与测试结果。
> ```

> **用户模拟测试提示词**
> ```
> 作为数据工程师，验证版本对比/回滚：
> 1. 发布 v1 → 改动 → 发布 v2，打开版本历史确认两版本都在。
> 2. 对 v1 与 v2 做对比，确认差异(改了哪个节点的哪个字段/新增删除的边)展示准确。
> 3. 对 v1 执行「回滚」，确认当前草稿变回 v1 内容且提示「有未发布变更」；重新发布得到 v3(内容=v1)。
> 4. 确认历史 v1/v2 未被改动。
> 报告：版本列表、diff 准确性、回滚语义(生成新版本而非改历史)。
> ```

### 步骤 6.4 — 发布审批（F3）

- **目标**：发布前经 `module-security` 审批中心审批，审批通过后才进入 `PUBLISHED`；可配开关关闭。
- **涉及文件**：`PipelineService.updateStatus`（PUBLISHED 前创建 `ApprovalRequest(requestType=PUBLISH)`）；新增跨模块调用（复用 `module-security` 的 `SecurityService` 审批 API 或经事件）；审批通过回调（消费审批通过事件或轮询）完成发布；配置开关 `onelake.orchestration.publish-approval.enabled`。
- **实现要点**：
  - 复用现有 `ApprovalRequest`（`requestType="PUBLISH"`、`targetRef=dagId`、`payload=待发布 snapshot 摘要`）。
  - 开关关闭：保持现状直接发布。开关开启：`updateStatus(PUBLISHED)` 改为「提交审批」→ 状态置 `PENDING_APPROVAL`（新增流水线子状态或用审批记录判定），审批通过后由回调执行真正发布（6.2 快照）。
  - 审批拒绝：保持 VALIDATED，记录原因。
- **验收标准**：开关开启时，发布需审批通过才生效；拒绝则不发布；开关关闭时行为不变。

> **Agentic 开发提示词**
> ```
> 目标：为流水线发布接入 module-security 审批中心（F3），可配开关。约束：
> 1. 新增配置 onelake.orchestration.publish-approval.enabled(默认 false)。关闭时保持现有直接发布行为。
> 2. 开启时：PipelineService 发布入口改为「提交发布审批」——创建 ApprovalRequest(requestType='PUBLISH', targetRef=dagId, payload=snapshot 摘要)，流水线进入等待审批态(可用 dag 子状态或依据 pending 审批记录判定，勿破坏现有 DRAFT/VALIDATED/PUBLISHED 主状态机)。
> 3. 审批通过 → 执行 6.2 的 publishSnapshot 并置 PUBLISHED；审批拒绝 → 保持 VALIDATED 并回填原因。通过/拒绝的衔接优先走事件或 module-security 现有回调，避免编排模块直接写 security 库(遵循模块边界)。
> 4. 幂等：同一 dag 同一待发布内容不重复建审批。
> 验证：补单测覆盖 开关开/关两条路径 + 通过/拒绝结果；mvn -q -pl module-orchestration -am test。报告改动文件、跨模块衔接方式与测试结果。
> ```

> **用户模拟测试提示词**
> ```
> 作为数据工程师与审批人，验证发布审批：
> 1. 打开审批开关。以工程师身份对已 VALIDATED 流水线点「发布」，确认变为「等待审批」而非直接发布。
> 2. 以审批人身份在 系统/审批 页看到该 PUBLISH 申请，点通过，确认流水线真正进入 PUBLISHED 并生成版本快照。
> 3. 再发起一次并「拒绝」，确认流水线停留 VALIDATED 且显示拒绝原因。
> 4. 关闭审批开关，确认发布恢复为直接生效。
> 报告：审批门控是否生效、通过/拒绝结果、开关行为。
> ```

### 步骤 6.5 — 开发 / 生产隔离（F4）

- **目标**：明确「DEV 草稿（dag 实时定义）」与「PROD 运行（已发布快照）」分离：日常编辑在 DEV，调度/事件触发跑 PROD 快照。
- **涉及文件**：`OrchestrationService`/`PipelineSchedulerService`/`PipelineSyncRefTriggerHandler`（生产触发一律用 `published_version_id`）；`PipelineService`（DEV 试跑接口 `trigger?env=DEV`）；前端标识。
- **实现要点**：
  - 生产触发（CRON/EVENT）：无 `published_version_id` 则不触发（未发布不进入生产调度），有则按快照执行。
  - 手动触发支持 `env=DEV|PROD`：DEV 试跑当前草稿（不写版本），PROD 跑已发布快照。
  - 与 6.2 一致：这是 6.2「跑已发布快照」的显式化 + 生产准入校验。
- **验收标准**：未发布流水线不进入 CRON/EVENT 生产调度；DEV 试跑用草稿、PROD 用快照，两者互不影响。

> **Agentic 开发提示词**
> ```
> 目标：落地开发/生产隔离（F4）。约束：
> 1. 生产触发准入：PipelineSchedulerService(CRON) 与 PipelineSyncRefTriggerHandler(EVENT) 仅触发有 published_version_id 的流水线；无已发布版本则跳过并记日志(未发布不进入生产)。
> 2. triggerPipelineRun 增 env 语义：PROD(默认，跑 published 快照，写 pipeline_version_id)；DEV(试跑当前草稿，不绑定版本、结果标记为试跑)。手动触发接口透传 env。
> 3. 与 6.2 保持一致(不重复实现快照编译)；DEV 试跑不得污染生产 run 统计(可在 job_run.run_mode 或 trigger 上区分，或复用 M2 run_mode)。
> 验证：补单测覆盖 未发布不被 CRON/EVENT 触发 + DEV/PROD 两条触发路径；mvn -q -pl module-orchestration -am test。报告改动文件与测试结果。
> ```

> **用户模拟测试提示词**

### 步骤 7.3 — 新鲜度策略参与就绪判定（E2）

- **目标**：把 `edge.freshnessPolicy` / `subscription.freshnessPolicy` 从「编译提示」升级为「运行触发条件」：`LATEST` / `SAME_FRESHNESS_WINDOW` / `SAME_BATCH` 真正参与多输入就绪判定。
- **涉及文件**：触发处理器 barrier 逻辑；`AssetReadiness`（记录 `batch_id`/`freshness_window`）；`OrchestrationService`。
- **实现要点**：
  - `LATEST`：任一最新到达即视为就绪（现状行为）。
  - `SAME_BATCH`：多输入必须同一 `batch_id` 才触发（避免用到不同批次数据）。
  - `SAME_FRESHNESS_WINDOW`：多输入必须落在同一新鲜度窗口（如同一 `logical_date` 桶）。
  - 从事件 payload 取 `batchId`/新鲜度信息写入 `asset_readiness`，barrier 校验一致性。
- **验收标准**：`SAME_BATCH` 下不同批次不触发、同批次才触发；`SAME_FRESHNESS_WINDOW` 下跨窗口不触发。

> **Agentic 开发提示词**
> ```

> 2. barrier 判定按目标节点的多输入策略：LATEST=任一最新即就绪；SAME_BATCH=全部输入 batch_id 一致才就绪；SAME_FRESHNESS_WINDOW=全部输入 freshness_window 一致才就绪。策略来源=edge.freshness_policy 或 subscription.freshness_policy。

## 3. 风险提示（M3 专项）

- **R-M3-1 快照与实时定义的一致性**：F1 要求「触发跑快照、编辑改草稿」，必须把 `PipelineCompileService` 抽出「从 tasks/edges 集合编译」的纯函数，避免只能读实时库导致快照失真。
- **R-M3-2 版本膨胀**：每次发布产生快照，需 `checksum` 去重（无变更不生成新版本）+ 历史版本可归档（`status=ARCHIVED`），必要时保留窗口。
- **R-M3-3 跨模块边界（F3）**：编排发起审批应经 `module-security` 服务/事件，避免编排直接写 security 库，遵循模块化单体边界（`RTK.md`）。
- **R-M3-4 触发器持久化改造（E1）**：`PipelineSyncRefTriggerHandler` 从内存 readiness 改 DB `asset_readiness`，须保证多副本幂等与旧用例（`PipelineSyncRefTriggerHandlerTest`）不回归。
- **R-M3-5 新鲜度/质量时序竞态（E2/E3）**：资产更新与质量检查、多输入到达存在乱序，barrier 与质量态判定要幂等且可清理陈旧状态，明确「以更新+对应通过质检为准」口径。
- **R-M3-6 事件契约兼容（R5）**：`pipeline.published` 增补 `versionId`、新增 `pipeline.rolled_back` 须向后兼容既有消费者（catalog/quality/modeling），参考 `docs/全局事件契约（Outbox 总线 · 事件清单 · 发布订阅 · 载荷 Schema）.md`。
- **R-M3-7 与 M4 边界**：算子执行打通（G）、任务类型扩展（B）不在 M3；E3 的质量条件依赖现有 `quality.check.completed`，更丰富的告警联动在 M4 I3。
```
