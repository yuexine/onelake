# 编排模块 V2 · M4 详细任务拆解、迁移脚本与提示词

> 上游文档：`docs/数据开发与编排模块V2升级计划.md`（M4 = 任务类型扩展 B1–B7 + 算子执行打通 G1–G3 + 大盘告警 I1–I4）
>
> 前置：M1/M2/M3 已落地并完成验收，编排 schema 的 Flyway 迁移头为 **V30**。M4 迁移必须从 **V31+** 续接，不得复用已占用的 V19–V30。M4 大量能力构建在 M1 的节点级动态图执行（`run_pipeline_graph_op`）之上。
>
> 口径：基于当前代码事实（2026-07-13）。对齐现状：`TaskType` 已扩展为 `SYNC_REF/SPARK_SQL/PYSPARK/QUALITY_GATE/TRINO_SQL/PYTHON/SHELL/BRANCH/CONDITION/SENSOR/WAIT/SUB_PIPELINE/NOTIFY/ASSERTION`，并标注 `EXEC/CONTROL/OBSERVE` 分类；`pipeline_task.task_type` 为 varchar，**无 DB CHECK 约束**，由应用层枚举校验。`RuntimeContractService` 保持 Spark 为默认主线，已将图模式下的 TRINO 标为 `READY`；`PYTHON/SHELL` 已接入 Bubblewrap 默认禁网沙箱、M1 回调/MinIO 日志与 Java 编译期限制校验，但能力开关 `ONELAKE_SCRIPT_SANDBOX_ENABLED` 默认关闭，未显式启用时仍为 `RESTRICTED`。`TRINO_SQL`、`BRANCH/CONDITION`、`SENSOR/WAIT` 与 `SUB_PIPELINE/NOTIFY/ASSERTION` 后端执行链路均已落地；统一编辑器已按 `EXEC/CONTROL/OBSERVE` 提供 Palette 和新类型 Inspector，算子模板消费及后续观测能力仍按后续步骤实现。`BuiltInOperatorCatalog` 的算子 manifest 已含 `templateKind`=SELECT_EXPR/COLUMN_EXPR/FILTER/SPARK_SQL/SPARK_SINK/RAW_SQL + `{{ }}` SQL 模板，但 `PipelineCompileService` **未消费**算子模板；`module-common` 已提供 `NotificationService`，M2 已发 `pipeline.run.sla_missed/timeout`，M3 已落地不可变版本与资产自动化契约。
>
> 使用方式：每步含「目标 / 涉及文件 / 实现要点 / 验收标准 / Agentic 开发提示词 / 用户模拟测试提示词」。

## 0. M4 范围与执行顺序

```text
Epic 8 任务类型扩展（B1–B7）
  8.1 DB 迁移 V31 + TaskType 扩展 + category(EXEC/CONTROL/OBSERVE) + 图执行器类型分发 [基石·先做]
  8.2 TRINO_SQL（轻量 SQL 走 Trino，不起 Spark）
  8.3 PYTHON / SHELL（沙箱脚本节点）

Epic 9 算子执行打通（G1–G3）
  9.1 算子 → Spark SQL 代码生成器（消费 manifest template）
  9.2 画布拖拽算子生成可执行节点（operator_ref + config）
  9.3 算子版本锁定（run 复现）
  9.4 前端算子拖拽落地

Epic 10 观测与告警（I1–I4）
  10.1 运行大盘 API + 前端（成功率/时长趋势/失败 Top/资源）
  10.2 甘特 / 时间线（单 run 节点时间线）

```

依赖关系：`8.1 → {8.2..8.6} → 8.7`；`9.1 → 9.2 → 9.3 → 9.4`（9.x 依赖 8.1 的图执行器分发与 M3 版本快照做算子锁定）；`10.1/10.2` 依赖 M1 节点级数据；`10.3` 依赖 M2 的 SLA/timeout 事件；`10.4` 依赖 Catalog 血缘。建议 Epic 8 → 9 → 10。

## 1. 迁移脚本草案（合并预览）

> 落位：`onelake-app/bootstrap/src/main/resources/db/migration/orchestration/`，续接 M3 的 `V30`。全部 `ADD COLUMN ... DEFAULT` / `CREATE TABLE IF NOT EXISTS`，兼容存量数据。

### V31__task_type_extension_operator.sql

```sql
-- M4-B/G：任务类型扩展分类 + 算子引用 + 算子版本锁定
-- 关联 docs/数据开发与编排模块V2升级计划.md §4.2（B）、§4.7（G）

ALTER TABLE orchestration.pipeline_task
  ADD COLUMN IF NOT EXISTS category         VARCHAR(16) NOT NULL DEFAULT 'EXEC', -- EXEC|CONTROL|OBSERVE
  ADD COLUMN IF NOT EXISTS operator_ref     VARCHAR(128),                        -- G2 拖入的算子标识
  ADD COLUMN IF NOT EXISTS operator_version VARCHAR(32);                         -- G3 锁定版本

-- 说明：task_type 为 varchar 且无 DB CHECK 约束，新增枚举值仅需应用层 TaskType 扩展，无需改约束。
CREATE INDEX IF NOT EXISTS idx_task_operator ON orchestration.pipeline_task (operator_ref)
  WHERE operator_ref IS NOT NULL;

ALTER TABLE orchestration.task_run
  ADD COLUMN IF NOT EXISTS operator_version VARCHAR(32);  -- G3 run 复现锁定
```

### V32__alert_rule.sql（I3，可选）

```sql
-- M4-I3：告警路由规则（可选；不建规则时走默认站内通知）
CREATE TABLE IF NOT EXISTS orchestration.alert_rule (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id  UUID NOT NULL,
  dag_id     UUID REFERENCES orchestration.dag(id) ON DELETE CASCADE, -- NULL=租户级全局规则
  event_type VARCHAR(64) NOT NULL,        -- pipeline.run.failed|pipeline.run.sla_missed|pipeline.run.timeout
  channel    VARCHAR(24) NOT NULL DEFAULT 'INAPP', -- INAPP|EMAIL|WEBHOOK
  target     VARCHAR(256),                -- 接收人 id / webhook url
  enabled    BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_alert_rule_tenant_event ON orchestration.alert_rule (tenant_id, event_type);
```

## Epic 8 · 任务类型扩展（B1–B7）

### 步骤 8.1 — DB 迁移 V31 + TaskType 扩展 + 图执行器类型分发

- **目标**：`TaskType` 增补 10 个新类型并加 `category(EXEC/CONTROL/OBSERVE)`；M1 图执行器 `run_pipeline_graph_op` 按 `task_type` 分发到不同节点执行器；按 ADR 能力矩阵放开 `RuntimeContractService` 多引擎准入。
- **涉及文件**：`V31__task_type_extension_operator.sql`；`TaskType.java`（增 `TRINO_SQL/PYTHON/SHELL/BRANCH/CONDITION/SENSOR/WAIT/SUB_PIPELINE/NOTIFY/ASSERTION` + `category`）；`PipelineTask.java`（category/operatorRef/operatorVersion）；`onelake-app/dagster/definitions.py`（`run_node` 按类型分发）；`RuntimeContractService`（多引擎 spec）；`PipelineCompileService`（新类型校验）。
- **实现要点**：
  - `category` 用于前端分组与校验（CONTROL/OBSERVE 型不产资产、不 spark-submit）。
  - 图执行器 `run_node` 按 `task_type` dispatch：EXEC 型走对应引擎；CONTROL/OBSERVE 型在执行器内评估（见 8.4/8.5）。
  - `RuntimeContractService` 新增 TRINO/脚本引擎 spec，去掉「非 SPARK 一律 block」的硬编码，改为按引擎能力矩阵判定。
- **验收标准**：`make migrate` 成功；新增类型可创建/校验；老 4 类型行为不变；图执行器按类型分发不回归。

### 步骤 8.2 — TRINO_SQL（Trino 引擎节点）

- **当前状态**：本地数据面已为节点执行使用专用 `onelake_pipeline` 身份和 Trino file access control，服务端仅允许 `iceberg.default/ods/dwd`，显式访问 `system` catalog 或其他 schema 会被拒绝。
- **目标**：轻量 SQL 转换/校验走 Trino，不必起 Spark。
- **涉及文件**：`definitions.py`（`run_node` 的 TRINO 分支，用 Trino client 执行 SQL）；`SparkRunConfigBuilder`/新 `TrinoRunConfigBuilder`（节点 config 带 Trino catalog/schema）；`RuntimeContractService`（TRINO ready）。
- **实现要点**：Dagster 执行器对 TRINO_SQL 节点用 Trino 连接执行（复用数据面 Trino，`docker-compose` 已有 trino 服务）；产出可物化为 Iceberg（`CREATE TABLE AS`）或仅校验；沿用 M1 节点回调/日志。
- **验收标准**：TRINO_SQL 节点独立执行成功，产出/校验正确，与 Spark 节点可在同一流水线协作。

> **Agentic 开发提示词**

> 目标：实现 TRINO_SQL 任务类型（B）。约束：
> 1. onelake-app/dagster/definitions.py run_node 的 TRINO_SQL 分支：用 Trino 连接(数据面已有 trino 服务，host/port 取环境变量)执行 sql_or_script；支持 CTAS 物化到 Iceberg 或纯校验查询。执行输出/错误经 M1 回调与 MinIO 日志上报，语义与 Spark 节点一致(状态/短路/重试)。

### 步骤 8.3 — PYTHON / SHELL（沙箱脚本节点）

- **当前状态**：后端与 Dagster 执行链路已完成；部署需使用加固后的 `dagster-user-code` 镜像、定制 seccomp、可用的 Linux `/proc` 进程树计量、GRAPH 模式，并同时配置 `ONELAKE_SCRIPT_SANDBOX_ENABLED=true` 与 `ONELAKE_SCRIPT_SANDBOX_ALLOWED_TENANTS=<tenant UUIDs>`。运行时保守地将节点 CPU/内存总额分配为内核 hard rlimit，并以进程树聚合监控补充诊断；工作目录同时限制总字节与条目数，非 UTF-8 stdout/stderr 的替换/截断会显式标记。
- **目标**：通用脚本节点，受控沙箱执行。
- **涉及文件**：`definitions.py`（PYTHON/SHELL 分支，隔离子进程执行）；节点 config（脚本内容、超时、资源上限、环境白名单）。
- **实现要点**：在 Dagster 执行器内以子进程执行脚本，限制超时/输出大小/工作目录；禁网或按白名单；产物/日志经 M1 通道上报；**安全是重点**（不得访问控制面凭证、限制文件系统与网络）。
- **验收标准**：脚本节点可执行并回传状态/日志；超时被终止；越权访问被限制。

> **Agentic 开发提示词**
> ```
> 目标：实现 PYTHON/SHELL 沙箱脚本节点（B）。约束：
> 1. definitions.py run_node 的 PYTHON/SHELL 分支：将脚本写临时文件，用受限子进程执行(独立工作目录、超时 kill、stdout/stderr 大小上限、最小环境变量白名单、默认禁网或白名单)。输出/错误经 M1 回调与 MinIO 日志上报。
> 2. 安全硬约束：不注入控制面/内部 token 到脚本环境；限制文件系统访问范围；资源(CPU/内存/时间)上限来自节点 config，缺省保守。
> 3. Java 侧 config 校验：脚本大小上限、超时上限；PipelineCompileService 拒绝明显危险模式(可选静态检查)。
> 验证：mvn -q -pl module-orchestration -am test；补脚本超时/大输出截断用例。报告改动文件、沙箱限制清单与测试结果。
> ```

### 步骤 8.4 — BRANCH / CONDITION（条件分支、跳过下游）

- **目标**：按条件选择分支或跳过下游（控制型节点，不产资产）。
- **涉及文件**：`definitions.py`（执行器内评估条件，决定下游哪些边「激活/跳过」）；`TaskRunStatus` 复用 `SKIPPED`；图执行器短路逻辑扩展。
- **实现要点**：
  - CONDITION：评估布尔表达式（基于参数/上游 outputs），为假则把该分支下游置 `SKIPPED`（区别于 `UPSTREAM_FAILED`，是「有意跳过」）。
  - BRANCH：多路选择，仅激活命中分支，其余分支 `SKIPPED`。
  - 表达式求值受控（复用 M2 ParamRenderer + 白名单布尔求值），禁任意代码执行。
- **验收标准**：条件为真走激活分支，为假下游 SKIPPED；BRANCH 只激活命中分支；SKIPPED 不算失败。

> **Agentic 开发提示词**
> ```
> 目标：实现 BRANCH/CONDITION 控制型节点（B）。约束：
> 1. definitions.py 图执行器：CONDITION 节点评估布尔条件(基于 M2 参数与上游 outputs，受控白名单求值，禁任意代码)，为假把其下游分支置 SKIPPED；BRANCH 节点多路选择，仅命中分支下游继续，其余置 SKIPPED。
> 2. 复用 TaskRunStatus.SKIPPED 语义(有意跳过，非失败)，与 M1 UPSTREAM_FAILED 区分；SKIPPED 不使整体 run 失败。
> 3. Java 侧：这两类为 CONTROL category，不 spark-submit、不产资产；PipelineCompileService 校验分支/条件 config 完整(条件表达式、分支映射)。
> 验证：补单测覆盖 条件真/假分支、BRANCH 命中、SKIPPED 不致 run 失败；mvn -q -pl module-orchestration -am test。报告改动文件与测试结果。
> ```

> **用户模拟测试提示词**
> ```
> 作为数据工程师，验证条件分支：
> 1. 建含 CONDITION 的流水线(如「行数>0 才继续」)，用满足/不满足两种数据分别运行，确认满足时下游执行、不满足时下游为「已跳过」且整体不算失败。
> 2. 建 BRANCH 多路(如按参数 env 走不同分支)，确认只激活命中分支、其余跳过。
> 报告：条件真/假与多路分支的激活/跳过是否正确、SKIPPED 是否被误判为失败。
> ```

### 步骤 8.5 — SENSOR / WAIT（等待就绪/时间窗口）

- **目标**：等待外部资产/分区就绪，或等待到某时间窗口后再放行下游。
- **涉及文件**：`definitions.py`（SENSOR 轮询就绪、WAIT 等待时间）；节点 config（等待目标、超时、轮询间隔）；可复用 M3 `asset_readiness` / Catalog 资产状态。
- **实现要点**：
  - SENSOR：轮询目标资产/分区是否就绪（查 Catalog 或 `asset_readiness`），就绪则通过、超时则失败或跳过（可配）。
  - WAIT：等待到 `logical_date + offset` 或固定时长。
  - 轮询在执行器内带上限，避免无限阻塞占用资源。
- **验收标准**：SENSOR 在目标就绪后放行、超时按策略处置；WAIT 到点放行。
- **落地状态（2026-07-12）**：SENSOR 已通过内部令牌接口轮询 `asset_readiness`，并回退读取持久化资产更新状态；支持资产 FQN、可选分区、1～86400 秒超时、1～300 秒轮询间隔及 `FAILED/SKIPPED` 超时策略。WAIT 已支持 `logical_date + offsetSeconds` 与固定 `durationSeconds`，运行时统一受 86400 秒硬上限保护。两类节点在等待前先通过 M1 回调进入 `RUNNING`，均保持 `OBSERVE` 且不发布资产产出。

> **Agentic 开发提示词**
> ```
> 目标：实现 SENSOR/WAIT 观测型节点（B）。约束：
> 1. definitions.py：SENSOR 节点按 config(目标资产 FQN/分区、超时、轮询间隔)轮询就绪(查 Catalog 资产状态或 asset_readiness)，就绪→SUCCEEDED 放行下游；超时→按 config 选择 FAILED 或 SKIPPED。WAIT 节点等待到 logical_date+offset 或固定时长后放行。
> 2. 轮询/等待带硬上限，避免无限阻塞；等待期间节点状态为 RUNNING，经 M1 回调可见。

### 步骤 8.6 — SUB_PIPELINE + NOTIFY + ASSERTION（P2 三类）

- **目标**：子流水线引用触发、运行内通知、轻量断言。
- **涉及文件**：`definitions.py`（三类分支）；`SUB_PIPELINE` 回调后端触发另一条流水线；`NOTIFY` 调 `module-common` 通知；`ASSERTION` 执行轻量校验。
- **实现要点**：
  - SUB_PIPELINE：节点通过内部回调（复用 M1 内部接口鉴权）请求后端 `triggerPipelineRun(subDagId)`，可选等待其完成。
  - NOTIFY：调用 `NotificationService`/发通知事件，内容支持参数渲染。
  - ASSERTION：比 QUALITY_GATE 更轻的行内断言（表达式为假则失败），复用受控求值。
- **落地状态（2026-07-13）**：SUB_PIPELINE 通过 M1 内部 token 接口按父 run + taskKey + attempt 的 EVENT 幂等键触发目标流水线的 M3 已发布快照，触发前锁定并核对当前 task_run attempt/父运行状态，同时用 PostgreSQL 事务级 advisory lock 串行化同一 EVENT 键，保证多实例并发复用同一子运行；继承父运行业务时间，支持异步返回或最长 86400 秒的有界等待，并遍历租户内发布快照拒绝自引用、调用环和超过 32 层的递归。NOTIFY 在节点就绪后通过 `module-common.NotificationService` 原子持久化通知，标题、消息和链接支持 M2 参数及上游 outputs 渲染，并按父 run + taskKey 幂等。ASSERTION 复用受控表达式求值，为假时回调 `FAILED` 并沿用 M1 下游短路；GRAPH 模式允许仅由控制/观测节点组成的流水线直接运行，三类均不声明资产产出。
- **验收标准**：SUB_PIPELINE 能触发子流水线；NOTIFY 产生通知；ASSERTION 断言失败使节点失败。

> **Agentic 开发提示词**

### 步骤 10.3 — 失败 / SLA 告警接通知总线（I3）

## 3. 风险提示（M4 专项）

- **R-M4-1 多引擎破边界（R4）**：引入 Trino/脚本必须先出 ADR 明确「Spark 默认、Trino/脚本受限补充」，并改造 `RuntimeContractService` 为能力矩阵，否则与「Spark-only 主线」文档冲突、触发被误 block。
- **R-M4-2 脚本节点安全（B）**：PYTHON/SHELL 是最高安全风险面——必须沙箱化（超时/资源/文件系统/网络白名单、不注入内部凭证），并在 ADR/安全评审记录。
- **R-M4-3 算子模板注入（G1）**：模板 `{{ }}` 填充必须区分标识符/字面量转义 + 白名单，复用 M2 受控渲染，禁止拼接任意用户输入进可执行 SQL。
- **R-M4-4 SUB_PIPELINE 成环/递归**：子流水线触发需防自引用与循环依赖（复用 M3 依赖成环校验思路），并限制递归深度。
- **R-M4-5 控制型语义与短路**：BRANCH/CONDITION 的 `SKIPPED` 必须与 M1 `UPSTREAM_FAILED` 严格区分（有意跳过≠失败），否则大盘成功率/告警口径失真。
- **R-M4-6 告警风暴**：I3 必须去重+限频，尤其回填/catchup 批量失败时；规则路由避免全员轰炸。
- **R-M4-7 事件契约兼容（R5）**：新增算子/类型不改既有 `pipeline.run.*`/`pipeline.task.loaded` payload 结构，仅增量字段，保持 catalog/quality/modeling 消费者兼容。
- **R-M4-8 V3 边界**：Flink 实时编排、AI 辅助开发（对标 DataWorks Data Agent）明确不在 M4（V2），列入 V3（计划 §10）。
