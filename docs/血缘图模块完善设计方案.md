## 一、模块定位（结合整体系统设计）

血缘图是 **L3-3「数据血缘」** 的核心载体，归属 `module-catalog`，并被多条用户旅程复用，本质是平台的 **影响分析（Impact Analysis）统一底座**：

- 旅程一（DE 建数据资产）：加工产出自动登记 → 资产详情页展示血缘。
- 旅程四（Ops 故障排查）：run 日志 →「查看下游影响」跳血缘图高亮受影响下游。
- 旅程五（Schema 变更闭环）：破坏性变更前做影响分析（表/任务/API）。
- 旅程三（密级随动）：密级 / 脱敏策略变更前预览「影响 N 资产 / M 个 API」。

结论：完善方向是 **「成熟图引擎 + 影响分析服务」**，而非单纯把图画好看。

1. 用成熟图引擎（X6）+ 自动布局（dagre），不手写坐标。
2. 字段级 = 节点字段行 + hover 单字段链路高亮，而非全局开关。
3. 影响分析独立成服务，输出结构化清单 + 导出 + 通知。
4. 血缘来源运行时上报（OneLake 已有 Outbox 基础）。
5. 上下游按需逐层展开 + depth + 方向切换 + 路径高亮 + 大图虚拟化。

## 五、详细设计

### 5.1 影响分析合规设计：`catalog.asset_consumer` 投影表

「受影响任务 / API / 订阅方」分布在 `orchestration` / `dataservice` 等其它 schema。架构铁律是 **module-catalog 不得直读其它模块的表**。因此采用 **Outbox 事件投影**：下游引用关系投影到 catalog 自有表，catalog 只查自己。

#### 5.1.1 新增投影表（写新迁移 `catalog/V8__asset_consumer.sql`，不改历史迁移）

> ⚠️ 迁移序号必须为 `V8`。当前 `catalog/` 目录已用到 `V7__query_template.sql`，若误用 `V2__catalog.sql` 会触发 Flyway "migration already exists" 报错。

```sql
CREATE TABLE IF NOT EXISTS catalog.asset_consumer (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL,
  asset_fqn     VARCHAR(512) NOT NULL,        -- 被消费的资产
  consumer_type VARCHAR(16) NOT NULL,         -- API / JOB / SUBSCRIPTION
  consumer_ref  VARCHAR(256) NOT NULL,        -- apiId / dagId / subscriptionId
  consumer_name VARCHAR(256),                 -- 展示名（/api/order、dws_user_order 等）
  owner_name    VARCHAR(128),                 -- 受影响 owner（用于「通知负责人」）
  status        VARCHAR(16) NOT NULL DEFAULT 'ACTIVE',  -- ACTIVE / REMOVED
  synced_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, asset_fqn, consumer_type, consumer_ref)
);
-- 影响分析查询模式为 WHERE tenant_id=? AND asset_fqn IN (...) AND status='ACTIVE'，
-- 使用部分索引让 COUNT 直接命中、跳过 REMOVED 软删行
CREATE INDEX IF NOT EXISTS idx_asset_consumer_fqn
  ON catalog.asset_consumer (tenant_id, asset_fqn)
  WHERE status = 'ACTIVE';
```

#### 5.1.3 影响分析查询（catalog 内，零跨模块读）

```java
public ImpactReportDTO impact(UUID tenantId, String rootFqn) {
    List<String> direct   = directDownstream(tenantId, rootFqn);   // 一跳
    List<String> all      = downstream(tenantId, rootFqn);         // 已有 BFS
    List<String> indirect = all - direct;
    Set<String> scope     = all ∪ {rootFqn};

    int jobs = consumerRepo.countByTypeAndFqnIn(tenantId, "JOB", scope);
    int apis = consumerRepo.countByTypeAndFqnIn(tenantId, "API", scope);
    int subs = consumerRepo.countByTypeAndFqnIn(tenantId, "SUBSCRIPTION", scope);
    boolean touchesDwsAds = scope.stream().anyMatch(f -> f.startsWith("dws.") || f.startsWith("ads."));
    return new ImpactReportDTO(rootFqn, direct, indirect, jobs, apis, subs,
        severityOf(apis, subs, jobs, touchesDwsAds),
        severityReasonsOf(apis, subs, jobs, touchesDwsAds));
}

// 多维度加权判定（替代单一 apis>0 规则）
static String severityOf(int apis, int subs, int jobs, boolean touchesDwsAds) {
    if (apis > 0 || subs >= 10) return "HIGH";      // 对外 API 或订阅方 ≥10 即高危
    if (jobs >= 5 || touchesDwsAds) return "MEDIUM"; // 关键链路任务或触及 DWS/ADS
    return "LOW";
}

static List<String> severityReasonsOf(int apis, int subs, int jobs, boolean touchesDwsAds) {
    List<String> reasons = new ArrayList<>();
    if (apis > 0)  reasons.add("影响对外 API " + apis + " 个");
    if (subs >= 10) reasons.add("影响订阅方 " + subs + " 个");
    if (jobs >= 5) reasons.add("影响任务 " + jobs + " 个");
    if (touchesDwsAds) reasons.add("影响 DWS/ADS 聚合层");
    return reasons;
}
```

DTO 中暴露 `severity` + `severityReasons: List<String>`，前端在影响分析面板用 `<List>` 渲染理由，避免「高危」徽章背后无可解释依据。

「通知受影响负责人」聚合 `asset_consumer.owner_name` 即可，**通知渠道**见 §5.2.3。

### 5.2 血缘图查询 API（新增，不破坏现有 downstream）

#### 5.2.1 DTO

```java
public record LineageGraphDTO(String rootFqn, List<Node> nodes, List<Edge> edges) {
    public record Node(
        String fqn, String name, String layer,     // SOURCE/ODS/DWD/DWS/ADS/API
        String nodeType,                            // TABLE / API / JOB
        String classification, java.math.BigDecimal qualityScore,
        String ownerName, Long rowCount, java.time.Instant syncedAt,
        List<Column> columns) {}
    public record Column(String name, String type, String classification) {}
    public record Edge(String fromFqn, String toFqn, String jobRef, List<ColumnEdge> columnEdges) {}
    public record ColumnEdge(String fromColumn, String toColumn, String transform) {}
}

public record ImpactReportDTO(
    String rootFqn,
    List<String> directDownstream, List<String> indirectDownstream,
    int affectedJobs, int affectedApis, int affectedSubscribers,
    String severity,
    List<String> severityReasons) {}
```

**Node 精简模式**：大图（`depth >= 3` 或节点数 > 50）时，service 用 `Node.summary(...)`（省略 `columns`、`qualityScore`、`rowCount`）减少序列化开销。前端按需再发 `/lineage/node-detail?fqn=` 单点拉字段行。Controller 通过 `?summary=true` 参数切换。

#### 5.2.2 Service

```java
// CatalogLineageService
LineageGraphDTO graph(UUID tenantId, String rootFqn, String direction /*UP|DOWN|BOTH*/, int depth);
ImpactReportDTO impact(UUID tenantId, String rootFqn);
```

要点：
- 双向 BFS，限制层数 `depth`（默认 3，防大图）。
- **批量查询（防止 N+1）**：
  - 节点元数据：`AssetRepository.findByTenantIdAndOmFqnIn(tenantId, fqns)`（**新增方法**）
  - 上游边：`LineageEdgeRepository.findByTenantIdAndDownstreamFqnIn(tenantId, fqns)`（**新增**，配合 UP/BOTH 方向）
  - 下游边：`LineageEdgeRepository.findByTenantIdAndUpstreamFqnIn(tenantId, fqns)`（**新增**，配合 DOWN/BOTH 方向）
  - 每层 BFS 批量拉一次，不在节点级别单查。
- 边的 `column_level` JSONB 解析为 `columnEdges`。
- `direction=BOTH` 时先 DOWN 再 UP 各一次 BFS，避免重复访问根节点。

#### 5.2.3 Controller 端点

```java
@GetMapping("/lineage/graph")            // ?fqn=&direction=BOTH&depth=3&summary=false
@GetMapping("/lineage/impact")           // ?fqn=
@GetMapping("/lineage/impact/export")    // ?fqn=  produces text/csv，导出/通知按 @PreAuthorize(DE/ADMIN)
@PostMapping("/lineage/impact/notify")   // ?fqn=  @PreAuthorize(DE/ADMIN)，触发通知
```

读接口 `authenticated`；导出/通知加 DE/ADMIN 角色校验，敏感操作记 `AuditLogger`。

**CSV 导出字段（固定列，按消费者一行一条）**：

| 列 | 来源 |
|---|---|
| `root_fqn` | 影响分析根节点 |
| `asset_fqn` | 受影响资产 |
| `layer` | ODS/DWD/DWS/ADS/API |
| `classification` | L1–L4 |
| `owner_name` | asset.owner_name |
| `consumer_type` | API/JOB/SUBSCRIPTION |
| `consumer_name` | `/api/order` / `dws_user_order` 等 |
| `consumer_owner` | asset_consumer.owner_name |
| `severity` | root 的 severity |

**通知渠道（`POST /lineage/impact/notify`）**：

- **首选渠道**：站内消息，复用现有 `NotificationAPI`（`module-common.notification` 表 + 未读拉取），消息类型 `IMPACT_ANALYSIS`，payload 携带 `rootFqn` + `severity` + `consumerCount`，支持「跳转血缘图 ?fqn=」。
- **可扩展**：通知发送抽象为 `NotificationChannel` SPI，本次只实现 `InAppNotificationChannel`，邮件/IM 留作后续扩展点（不进入 P1 范围）。
- **去重**：同一 `rootFqn` 24h 内只通知一次（按 `notification.key = "impact:" + rootFqn` 去重）。

**`asset_consumer.owner_name` 数据来源**：阶段 A' 在事件 payload 里要求生产者携带（API 从 `api_definition.owner_id` → user service 反查；JOB 从 DAG owner 反查；SUBSCRIPTION 从申请人反查）。若 owner 反查不可用，置空并在前端显示「-」，不阻断 P0 上线。

### 5.3 前端：用已安装的 X6 重写血缘图

`package.json` 已含 `@antv/x6@^2.18` + `@antv/x6-react-shape@^2.1`，**需新增 dagre 依赖**（自动布局）：

```bash
pnpm add dagre
pnpm add -D @types/dagre
```

- **引擎**：X6 Graph 替换手写 SVG；节点用 `react-shape` 渲染 React 组件，复用 `ClassificationBadge` / `StatusBadge`（满足密级色跨模块强约定）。
- **布局**：dagre 分层有向布局，方向 `LR`（ODS→ADS）。
- **降级策略**：dagre 初始化失败或节点数 > 200 时，退化为固定网格布局（`x = 40 + layer*220, y = 60 + i*90`，沿用现有 mock 坐标算法），保证可用性。
- **数据**：`CatalogAPI.lineageGraph(fqn, direction, depth)`，mock 仅作 fallback。

交互（对齐 L3-3.1.4 + 原型 §8.6.4）：
1. 方向切换：上游 / 下游 / 双向（Radio）。
2. depth 控制：1/2/3/全部（Select），大图虚拟化（X6 视口裁剪）。
3. 节点点击：选中高亮 + 调 `impact` 刷新右侧面板。
4. 逐层展开：节点「+」按钮，懒加载下一层（depth+1 增量请求）。
5. 路径高亮：选中 → 下游链路高亮（橙），非相关节点淡化（opacity 0.25）。
6. 字段级开关（L3-3.1.2）：开 → 节点展开字段行；hover 某字段 → 仅高亮该字段 column edge 贯穿链路。

## 七、风险与注意点

1. **最终一致性**：投影表异步更新（`OutboxDispatcher` `fixedDelay=1000` 即 **1 秒**轮询），影响数字可能 1–2 秒延迟，可接受；上线时执行一次全量回填，把已存在 API / 订阅 / DAG 绑定写入投影表。回填脚本需要独立 PR（含 dry-run 和 tenant 过滤）。
2. **大图性能**：默认 `depth=3`；超出用 X6 视口虚拟化 + 懒加载「+」展开 + Node 精简模式（见 §5.2.1），避免一次渲染上千节点。
3. **多租户**：所有查询带 `tenant_id`；节点元数据批量查（`findByOmFqnIn`）、lineage 边批量查（`findByUpstreamFqnIn` / `findByDownstreamFqnIn`）避免 N+1。
4. **架构合规**：严禁 catalog 直读 dataservice/orchestration 表。现有 5 处跨 schema 直读（见 §5.5）作为独立技术债 issue 跟踪；本次**强制**迁移 dataservice 行，其余登记后跟进。
5. **OM 一致性**：OneLake 血缘与 OpenMetadata `/lineage` 双写，需明确以 `catalog.lineage_edge` 为平台内权威，OM 回写为最佳努力（失败不阻断业务，沿用现有 try-catch 容错）。
6. **JOB consumer 口径**：`lineage_edge.job_ref` 仅用于展示，不参与影响计数（job_ref 现状是 `runId` 而非稳定 `dagId`）。影响分析里的「受影响任务」一律以 `asset_consumer` 投影表为准。
7. **通知频率**：`/lineage/impact/notify` 按 `rootFqn` 24h 去重，避免 DE 误触发导致 owner 被站内消息轰炸。
8. **severity 阈值需上线后调参**：当前 `apis>0` 或 `subs>=10` 判 HIGH 是基于经验值，应在 P1 上线后看真实分布再调（在 `application.yml` 外置阈值，不写死）。
