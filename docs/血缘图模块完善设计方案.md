# 血缘图模块当前状态与完善计划

> 修订日期：2026-07-09
>
> 旧方案中多项“新增”能力当前已经部分落地。本文改为当前状态与后续完善项。

## 1. 当前实现

| 区域 | 当前代码 |
| --- | --- |
| 后端 | `CatalogController`、`CatalogLineageService`、`LineageEdge`、`AssetConsumer`、事件 handlers |
| API | `/api/v1/catalog/lineage/graph`、`/lineage/downstream`、`/lineage/impact`、`/lineage/impact/export`、`/lineage/impact/notify` |
| 数据 | `catalog.asset`、`catalog.lineage_edge`、`catalog.asset_consumer` |
| 前端 | `/catalog/lineage`，仍存在 fallback/mock 场景 |
| 事件 | `integration.table.loaded`、`pipeline.task.loaded`、`modeling.model.loaded`、`security.pii.detected` 等可刷新资产/血缘/字段标签 |

## 2. 当前能力

- 表级血缘。
- 字段级映射 JSON。
- 下游影响分析。
- CSV 导出。
- 通知影响对象。
- 资产消费方投影。
- Integration -> Catalog 成功事件建档和字段血缘。
- PII 扫描结果合并到 Catalog 字段。

## 3. 后续完善

| 优先级 | 项目 |
| --- | --- |
| P0 | 前端 LineageGraph 去 mock fallback，失败时展示真实错误 |
| P0 | 血缘 graph API 与前端节点/边 schema 固化 |
| P1 | 多层展开、路径高亮、字段级筛选 |
| P1 | 下游订阅/API/任务/质量门禁影响统一展示 |
| P1 | OpenMetadata writeback/readback 与本地 Catalog 合并策略 |
| P2 | 大图性能、布局缓存、差异对比 |

## 4. 文档规则

- “血缘图可见”必须注明数据来源：Catalog 本地、OpenMetadata 同步或前端 fallback。
- 字段级血缘应写成 `lineage_edge.column_level`，不要只说“已支持”而不说明载荷结构。
