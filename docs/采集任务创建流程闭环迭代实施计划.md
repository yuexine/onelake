# 采集任务创建流程闭环迭代实施计划

> 调研日期：2026-06-15
> 范围：采集任务向导、任务列表/详情、`module-integration` 任务创建/发布/触发、Airbyte 驱动、schema 快照、运行回写与 Outbox 事件。

## 1. 当前结论

采集任务创建已经具备“控制面可用骨架”：前端向导能调用真实创建接口，后端能保存 `DRAFT` 任务，列表/详情能读真实任务，发布接口能把任务置为 `ENABLED`，触发接口能创建 `sync_run` 并发布运行事件。

但端到端闭环还没有完成。当前发布不会自动创建 Airbyte connection；触发前必须已有 `airbyteConnectionId`；向导里的选表、字段映射、调度和试跑主要还是样例/提示；schema 快照服务仍是 stub；reconcile 只读 Airbyte job status，没有日志、行数、错误、checkpoint 和取消能力的完整回写。

### 2026-06-15 下一轮实施更新

已推进阶段 A 和阶段 B 的最小闭环：

1. `DataSourceController` 新增 `schemas`、`tables`、`columns` 探查接口。
2. `DatabaseDiscoveryClient` 支持 MySQL/Postgres 的 schema/table/column discovery。
3. `SourceSchemaSnapshotServiceImpl` 改为从真实 discovery 拉 columns，不再生成占位 schema。
4. `SyncTaskWizard` 接入真实 schemas/tables/columns，保存草稿走真实 `createSyncTask`。
5. `SyncTaskServiceImpl#enable` 在缺少 `airbyteConnectionId` 时调用 `AirbyteSyncDriver.ensureConnection`，支持从数据源 config 的 `airbyteSourceId`/`externalSourceId` 与 `airbyteDestinationId`/`externalDestinationId` 或配置项 `onelake.dataplane.airbyte.destination-id` 生成 connection。

仍未完成的是完整 Airbyte source/destination 动态创建、dry-run、run logs/cancel、Dagster 调度注册和运行指标回写。

因此当前状态是：

```text
已闭合：向导提交 -> 后端落任务 -> 发布状态 -> 列表/详情展示
未闭合：真实源端发现 -> Airbyte connection -> 真实运行入湖 -> 指标/日志回写 -> 诊断/血缘/下游事件
```

## 2. 前端现状

| 页面/接口 | 已有能力 | 主要缺口 |
| --- | --- | --- |
| `SyncTaskWizard.tsx` | 加载真实 schema/table/columns，保存草稿接 `createSyncTask`，发布接 `createSyncTask -> enableSyncTask` | 试跑未接后端；编辑模式、复制和批量操作尚未接入 |
| `SyncTaskList.tsx` | 列表、刷新、暂停、删除、触发接 `IntegrationAPI` | 搜索/筛选在前端本地过滤；批量操作、编辑、复制仍是提示 |
| `SyncTaskDetail.tsx` | 详情、运行历史、触发、暂停接真实接口 | 运行日志路由无后端支撑；血缘仍是推断小图；编辑按钮未接 |
| `IntegrationAPI` | 覆盖任务 create/list/get/update/delete/enable/disable/trigger/runs 与数据源 schema/table/column discovery | 缺 dry run、run cancel、run logs、编辑模式向导接口 |

## 3. 后端现状

| 层 | 已有能力 | 主要缺口 |
| --- | --- | --- |
| API | `SyncTaskController` 已有 create/list/get/update/delete/enable/disable/trigger/run alias/reconcile/runs | list 未分页；run logs/cancel/detail/dry-run 缺失 |
| Service | `create` 校验租户/数据源归属/重名，保存 `DRAFT` 并发任务创建事件；`enable` 可补齐 Airbyte connection | 发布仍不注册调度；Airbyte source/destination 动态创建尚未完成 |
| Service | `enable/disable` 改状态并发状态事件 | 状态机较松：重复 enable、PAUSED -> ENABLED、DRAFT -> PAUSED 等边界未显式约束 |
| Service | `trigger` 要求 `ENABLED` 且已有 `airbyteConnectionId`，调用 Airbyte 同步并落 `sync_run` | 状态映射、日志、取消和指标回写仍不足 |
| Airbyte | `enable` 已调用 `ensureConnection`；`AirbyteSyncDriver` 已有 `ensureConnection`、`triggerSync`、`getJobStatus`、`cancel` | 缺 source/destination 动态创建和日志读取 |
| Schema | `SourceSchemaSnapshotService` 可从 MySQL/Postgres 真实 columns 写快照、算 drift、发 drift 事件 | CDC/file schema 解析尚未接入 |
| Event | 已有 `integration.sync_task.created`、`integration.sync_run.started`、`integration.table.loaded`、`integration.sync.failed`、`integration.schema.drift` | 事件链依赖真实 run 终态，且 payload 指标不足 |

## 4. 目标闭环

目标流程：

```mermaid
flowchart LR
  A["选择数据源"] --> B["发现 schema/table/columns"]
  B --> C["配置字段映射、目标表、调度"]
  C --> D["保存草稿"]
  D --> E["发布任务"]
  E --> F["ensure Airbyte source/destination/connection"]
  F --> G["手动或调度触发 sync_run"]
  G --> H["Airbyte job 入湖"]
  H --> I["reconcile 状态、行数、错误、checkpoint、日志"]
  I --> J["发布 table.loaded 或 sync.failed"]
  J --> K["目录、质量、告警、编排联动"]
```

## 5. 迭代实施计划

### 阶段 A：创建表单真实化

目标：让向导不再靠样例表和样例字段映射创建任务。

实施内容：

1. 后端扩展 `DatabaseDiscoveryClient`：
   - `listSchemas(sourceId)`
   - `listTables(sourceId, schema)`
   - `describeTable(sourceId, table)`
2. 新增 API：
   - `GET /api/v1/integration/datasources/{id}/schemas`
   - `GET /api/v1/integration/datasources/{id}/tables?schema=`
   - `GET /api/v1/integration/datasources/{id}/tables/{table}/columns`
3. `SourceSchemaSnapshotServiceImpl#captureColumns` 改为调用真实 discovery；首次快照不再生成占位 schema。
4. 前端向导第 1 步加载真实 schema/table；第 2 步根据 columns 自动生成默认 field mapping。
5. 保存草稿按钮调用 `createSyncTask`，不再只是提示。

验收：

1. 从一个 MySQL/Postgres 数据源进入向导，可以看到真实库表和字段。
2. 保存草稿后任务为 `DRAFT`，刷新列表仍存在。
3. `mvn -q -pl module-integration -am test` 与 `pnpm build` 通过。

### 阶段 B：发布即生成数据面管道

目标：用户点击发布后，OneLake 自动获得可触发的 Airbyte connection。

实施内容：

1. 在 `SyncTaskServiceImpl#enable` 中加载任务和数据源，若无 `airbyteConnectionId`，调用新的管道准备服务。
2. 引入 `SyncDriver` / `SyncDriverRegistry`，避免 service 直接绑定 Airbyte。
3. Airbyte driver 补齐：
   - ensure source
   - ensure destination
   - ensure connection
   - 连接状态检查
4. 增加任务字段或扩展配置：
   - `driverType`
   - `externalSourceId`
   - `externalDestinationId`
   - `lastPublishedAt`
   - `publishError`
5. 发布失败时保持 `DRAFT` 或进入 `PUBLISH_FAILED`，并把错误返回给前端。

验收：

1. 没有 `airbyteConnectionId` 的任务发布成功后会回写 connection id。
2. Airbyte 不可达时前端能看到明确错误，任务不会假装启用。
3. 单元测试覆盖发布成功、Airbyte 失败、重复发布复用 connection。

### 阶段 C：试跑与触发运行闭环

目标：发布后可试跑或正式触发，并能在任务详情看到运行状态。

实施内容：

1. 新增 `POST /api/v1/integration/sync-tasks/{id}/dry-run`，执行连接、schema、映射兼容性、目标表命名和调度表达式校验。
2. `trigger` 创建 run 时先落 `QUEUED/RUNNING`，Airbyte 返回 job id 后更新。
3. Airbyte 状态映射改为显式 map：
   - `pending/queued` -> `QUEUED`
   - `running` -> `RUNNING`
   - `succeeded` -> `SUCCEEDED`
   - `failed/error/incomplete` -> `FAILED`
   - `cancelled/canceled` -> 新增 `CANCELLED` 或映射为 `FAILED` 并记录 error code
4. 新增 run 详情、日志、取消：
   - `GET /api/v1/integration/sync-runs/{id}`
   - `GET /api/v1/integration/sync-runs/{id}/logs`
   - `POST /api/v1/integration/sync-runs/{id}/cancel`
5. 前端试跑按钮接 dry-run；任务详情运行历史的日志按钮接日志页或抽屉。

验收：

1. 点击触发后立即生成 `sync_run`，详情页刷新可看到状态变化。
2. reconcile 能写入 `errorMsg`、`rowsRead`、`rowsWritten`、`checkpoint`。
3. 取消运行能调用 Airbyte cancel 并更新本地状态。

### 阶段 D：调度与事件下游闭环

目标：`scheduleCron` 变成真实调度，不只是一列配置。

实施内容：

1. 明确边界：integration 负责任务定义，orchestration/Dagster 负责调度执行。
2. 发布任务时注册/更新调度；暂停任务时禁用调度。
3. 手动触发和调度触发都带 `sync_run_id` tag，便于 Dagster sensor 回写。
4. `reconcile` 终态时继续发布：
   - `integration.table.loaded`
   - `integration.sync.failed`
5. catalog/quality/alert/orchestration 消费事件时补足幂等和失败重试观测。

验收：

1. Cron 到点能产生 run 并回写终态。
2. 成功 run 触发下游 DAG 或目录更新。
3. 失败 run 进入告警中心，且重复 reconcile 不重复创建下游副作用。

### 阶段 E：编辑、复制、批量和可观测性补齐

目标：把前端任务管理从主链路推进到生产可用。

实施内容：

1. 编辑按钮打开向导编辑模式，暂停态任务允许修改。
2. 复制任务以原配置预填新建向导。
3. 批量触发、批量暂停、批量删除接后端批量 API 或前端串行调用加聚合结果。
4. 监控接口从 `findAll()` 改为按 tenant、时间窗、状态分页/聚合查询。
5. 前端补 loading、partial failure 和 permission disabled 状态。

验收：

1. 主列表所有可见操作都有真实后端效果。
2. 监控大盘不跨租户、不全表扫描。
3. 前端构建和关键路由浏览器冒烟通过。

## 6. 推荐下一轮最小切片

下一轮建议只做两个闭环点：

1. `真实 schema discovery -> 向导选表/字段映射`
2. `enable -> ensure Airbyte connection -> 回写 airbyteConnectionId`

这两个点完成后，“创建采集任务”才从 UI 配置闭合到数据面管道准备。触发运行、日志、取消和 Dagster 调度可以作为再下一轮，避免一次性跨太多组件。
